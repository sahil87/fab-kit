### F01 — Add cross-process locking to .fab-runtime.yaml read-modify-write cycles

**Impact**: high · **Effort**: medium · **Dimension**: correctness · **Verifier confidence**: high

**Files**:
- `src/go/fab/internal/runtime/runtime.go:183-199`
- `src/go/fab/internal/runtime/runtime.go:304-345`
- `src/go/fab/cmd/fab/hook.go:145-149`
- `src/go/fab/cmd/fab/hook.go:122-123`

**Problem**: Every mutator of the shared per-repo .fab-runtime.yaml (WriteAgent runtime.go:183-199, ClearAgent :203-231, ClearAgentIdle :239-273, GCIfDue :304-345) does an unlocked LoadFile -> mutate map -> SaveFile cycle. The temp+rename in SaveFile (runtime.go:99-139) prevents torn files but not lost updates. These functions are invoked from Claude Code hooks (hook stop/user-prompt/session-start, hook.go:122, 147-148, 170-171) which by design fire concurrently from multiple agent panes in the same repo (the operator workflow's whole premise). Two stop hooks racing: A loads {}, B loads {}, A saves {A}, B saves {B} — agent A's entry is silently lost, so the operator's pane map (which reads _agents) sees a stale/incomplete picture. Worse, each handler calls GCIfDue immediately after its own write, so a GC sweep snapshot from one process can clobber an entry another process just wrote. No flock/O_EXCL exists anywhere in either module (verified by grep).

**Fix**: Wrap the load-mutate-save cycle in an advisory lock: open a sibling lock file (e.g. .fab-runtime.yaml.lock) with os.OpenFile(O_CREATE), syscall.Flock(fd, LOCK_EX), then load/mutate/save/unlock. Both linux and darwin (the only supported GOOS per the proc build tags) support flock. A small withRuntimeLock(fabRoot, func(m map[string]interface{}) error) helper would cover all four mutators.

**Adversarial-verifier notes**: VERIFIED: (1) All four mutators do unlocked load-mutate-save exactly as cited — WriteAgent src/go/fab/internal/runtime/runtime.go:183-199, ClearAgent :203-231, ClearAgentIdle :239-273, GCIfDue :304-345; SaveFile :99-139 is temp+rename only (torn-write protection, not lost-update protection). (2) Hook call sites confirmed at src/go/fab/cmd/fab/hook.go:122-123, 147-148, 170-171, each followed by GCIfDue. (3) grep over all of src/go/ for flock/LOCK_EX/O_EXCL/lockfile: zero matches — no locking exists anywhere in either module. (4) No constraint conflict: constitution Portability (V) governs kit content, not the Go binary; the runtime package already uses Unix-only syscall.Kill (runtime.go:356), proc is linux/darwin build-tagged, and docs/memory/pipeline/change-lifecycle.md:252 records 'Windows compatibility is not a concern (explicitly confirmed)' — flock works on both supported GOOS. (5) Not already implemented, not trivial. Strongest supporting evidence the reviewer missed: docs/memory/runtime/runtime-agents.md:75 claims atomicity 'even under concurrent hook invocations' — the design explicitly contemplates concurrent hooks on one file but only addresses partial writes, conflating rename-atomicity with race safety; no design decision accepts last-writer-wins.

CORRECTIONS: (a) The parenthetical '(the operator workflow's whole premise)' is wrong. The operator's documented premise is one agent per worktree — src/kit/skills/fab-operator.md:23 ('All pipeline work MUST run in a freshly spawned agent tab in its own worktree... Even a one-liner change gets its own worktree') and runtime-agents.md:50 ('Each worktree has its own file; there is no cross-worktree state or sharing'). Distinct operator-spawned agents therefore never contend on the same .fab-runtime.yaml. The realistic race surface is multiple Claude sessions opened in the SAME worktree (a scenario the session_id-keyed _agents map and the pane-map scan at internal/pane/pane.go:228,287 explicitly support), plus the cross-process GC clobber when two such sessions coexist. Single-session hooks (stop -> user-prompt) are sequential in the turn lifecycle. (b) Impact should be downgraded high -> medium: race windows are milliseconds, GC writes at most once per 180s (runtime.go:316-320, and in-process GC reloads after its own write at :310 so no self-clobber), and the docs note runtime state 'self-heals within one hook cycle' (runtime-agents.md:165). The non-trivial consequence when it does hit: a lost ClearAgentIdle leaves a stale idle_since, so fab pane send's idle validation passes against a busy agent and the operator can inject keystrokes mid-turn; a lost WriteAgent leaves an idle agent unmatched, blocking pane send until the next hook fire. (c) Implementation notes: the suggested .fab-runtime.yaml.lock sibling needs a .gitignore entry; runtime-agents.md:75 must be updated per docs-are-source-of-truth; no _cli-fab.md change needed (no command-signature change).

**Impact**: high · **Effort**: medium · **Dimension**: correctness · **Verifier confidence**: high

- `src/go/fab/cmd/fab/hook.go:209-216`
- `src/go/fab/cmd/fab/hook.go:276-298`
- `src/go/fab/internal/status/status.go:297-326`
- `src/go/fab/internal/statusfile/statusfile.go:286-319`

**Problem**: Each .status.yaml mutation is an unlocked load-then-save of the whole document: cmd loadStatus (cmd/fab/status.go:48-62) loads, then status.Start/Finish/SetAcceptance call statusFile.Save which rewrites the entire file from the in-memory snapshot. Add tmp.Sync() before Close in statusfile.Save to match runtime.SaveFile's durability posture.

### F04 — Fold runtime-file GC into the same load/save as the agent-entry mutation in session hooks

**Impact**: medium · **Effort**: small · **Dimension**: performance · **Verifier confidence**: high

**Files**:
- `src/go/fab/cmd/fab/hook.go:146-148`
- `src/go/fab/cmd/fab/hook.go:122-123`
- `src/go/fab/cmd/fab/hook.go:170-171`
- `src/go/fab/internal/runtime/runtime.go:183-198`
- `src/go/fab/internal/runtime/runtime.go:304-320`
- `src/go/fab/internal/runtime/runtime.go:124`

**Problem**: Every Stop, UserPromptSubmit, and SessionStart event from every agent runs a hook handler that performs two independent load-modify-save cycles on .fab-runtime.yaml: WriteAgent/ClearAgent/ClearAgentIdle does ReadFile+yaml.Unmarshal+Marshal+temp+fsync+rename (runtime.go:183-198), then GCIfDue immediately re-stats and re-parses the entire file (runtime.go:306-310) before checking the last_run_gc throttle (runtime.go:316-320). In the common throttled case the second full parse is pure waste; when GC is due, the file is marshaled and atomically written a second time, with a second fsync (tmpFile.Sync at runtime.go:124 runs on every SaveFile). Net cost per hook event: 2 YAML parses and 1-2 fsync'd atomic writes of the same file within one process. The non-atomic read-modify-write pair also widens the lost-update window between concurrent agents' hooks.

**Fix**: Add a runtime.UpdateAgent(fabRoot, sessionID, mutate func(map[string]interface{}), gcInterval) that loads the file once, applies the entry mutation, runs the GC sweep inline when `now - last_run_gc >= interval` (the map is already in hand — pidAlive is just a kill(pid,0)), and saves once. Consider dropping the fsync for this ephemeral, fully re-derivable state file (statusfile.Save already omits it).

(3) runtime.go:304-320 — GCIfDue os.Stat (306) and fully re-parses via LoadFile (310) BEFORE the last_run_gc throttle check (315-320), so the throttled common case wastes a complete second YAML parse. (4) runtime.go:121-127 — SaveFile fsyncs (tmpFile.Sync at :124) on every write, so a due GC produces a second fsync'd atomic write. Stronger: runtime-agents.md:87 describes the intended design as 'the GC sweep piggybacks on the same file read + write round-trip when it's actually due' — the implementation does NOT do this; the suggestion would make code match the doc. ### F05 — Make the .fab-status.yaml active-pointer swap atomic

**Impact**: medium · **Effort**: small · **Dimension**: correctness · **Verifier confidence**: high

**Files**:
- `src/go/fab/internal/change/change.go:176-182`
- `src/go/fab/internal/change/change.go:152-162`

**Problem**: change.Switch updates the active-change pointer with os.Remove followed by os.Symlink (change.go:179-181); Rename does the same (change.go:159-160). This is non-atomic: any concurrent fab process (hooks resolving the active change via resolveFromCurrent, log.Command, pane map) that reads in the gap sees no pointer and either errors with "No active change" or silently falls back to the single-change guess and resolves a different change. Two concurrent Switch invocations can also interleave so the second os.Symlink fails with EEXIST after the other process re-created the link, returning an error for an operation that conceptually succeeded. Given the multi-agent operator design where panes share a worktree, this window is hit in practice.

### R7 — Validate the fab_version string before using it in the download URL and filesystem cache path

**Why refuted**: Cited line-level facts are accurate: readFabVersion (config.go:55-72) only rejects empty fab_version (line 68), and the value flows unvalidated into both the download URL (download.go:28, fmt.Sprintf with v%s) and the cache path (cache.go:29-30, filepath.Join(home, ".fab-kit/versions", version)). No semver/format validation exists anywhere (only doctor.go regexes parse external tool output; sync.go parseSemver is lenient and unrelated). So the lack of validation is factually true.

BUT the headline impact — "drive an out-of-tree filesystem write" by running a routine command in an untrusted repo — is materially overstated to the point of being unrealizable (refutation test a):

1. Write is gated behind HTTP 200. In Download (download.go:30-49), http.Get and the StatusCode != 200 check run BEFORE os.MkdirAll(cacheDir) and extractArchive. Grep confirms CacheDir(version) is written only at download.go:40 (post-200); all other RemoteCacheDir/CacheDir/LocalCacheDir uses are read-only os.Stat. No write to the escaped dir happens without a successful download.

2. Filesystem-escape and valid-URL are mutually exclusive payloads. I ran a Go repro: version "../../../tmp/x" yields fsPath=/home/tmp/x (escapes) but URL path normalizes to /sahil87/fab-kit/releases/tmp/x/... (404). version "2.1.6/../../../../../../tmp/evil" yields fsPath=/tmp/evil but URL -> /tmp/evil/... (404). The "v%s" prefix glues "v" onto the first segment, so the "..\" that escapes the filesystem destroys the /download/vX/asset structure GitHub needs to return 200. Any string with ".." (required to escape) breaks the URL; no string both escapes and downloads.

3. URL injection is bounded and benign. Host github.com and repo sahil87/fab-kit are hardcoded constants (download.go:17,28); HTTPS; version only mutates the path within that fixed trusted repo. It cannot redirect to an attacker host or serve attacker-controlled content — no supply-chain compromise.

Net: the realizable security impact is essentially nil, not "medium." What remains is minor defensive input-validation hygiene (and clearer error messages), which does not match the finding's security framing (path-traversal write / supply-chain). A senior engineer would not endorse this at the stated severity, so confirmed=false.
