# docs-reorg-memory

## Summary

Analyzes memory files for themes and suggests reorganization. Read-only unless user approves changes. Identifies up to 10 themes, produces a **Shape Report** — folders over the ~12 width bound / over depth — folder depth >2, i.e. topic files past 3 path segments / under the ~5 floor (reserved domains `_shared`/`_unsorted` exempt), **plus over-size topic-file rows (split candidates, ~400 lines / ~15KB)** — detects **duplicate coverage** (the same topic in 2+ files), triages `_unsorted/` staging, and proposes a reorg plan whose Migration Map `Kind` is one of `move-section` / `split-domain` / `merge-domain` / `flatten` / `move` / `split-file` / `merge-file`. On approval it is the **memory rebalancer**: it performs the approved migrations, rewrites the **bundle-relative** links the moves break, preserves each moved file's FKF frontmatter, authors `description:` frontmatter (+ `type: memory`) on new files and creates each new sub-domain's `description:`-only index stub before `fab memory-index` runs, and regenerates indexes (and per-folder `log.md`) via `fab memory-index`. Index/log files (root, domain, and sub-domain tiers) are regenerated by `fab memory-index`, never hand-edited. It also **orchestrates a one-time compatibility migration** for pre-fab-kit memory trees (see below).

**File-splitting, duplicate-coverage, and `_unsorted/` triage** (260718-dsrx): reorg owns structure at **file** granularity as well as folder. (1) The Shape Report gains over-size **file rows** (topic files past ~400 lines / ~15KB, sourced from the Step 1 `fab memory-index --check --json` `warnings[]` `file-size` findings; older-binary ⇒ read-all-files line counts) as `split-file` candidates — proposed only for a ≥2-topic file (heading clusters); a long-but-cohesive file is reported, not split (the same reactive soft-SHOULD stance as folder bounds). The Migration Map gains `split-file` (fan one multi-topic file into ≥2 topic files in the same domain/sub-domain — bodies move **verbatim**, restyling stays `/docs-distill-memory`'s job; new files get `type: memory` + a change-id-free `description:` and target ~300 lines; original path kept for the dominant topic, else removed; Link Impact retargets an **anchored** inbound link to its heading's new file and an **un-anchored** link to the dominant-topic file, ambiguity → abort escape). (2) A **duplicate-coverage** pass flags the same topic in 2+ files (near-identical filenames/descriptions, same filename in two domains, heavy heading overlap) in a `## Duplicate Coverage` table; remediation is a `merge-file` (fold B's unique sections into canonical A, rewrite inbound links to A, delete emptied B — file-granularity parallel of `merge-domain`, with Link Impact + the no-dangling-link guard) or `move-section` for partial overlap, with a cross-reference to the open single-sourcing seam audit (not scope). (3) An **`_unsorted/` triage** lists every staged file with a per-file `move`-to-domain (default) or `delete` (stale ephemera; explicit per-file confirmation) proposal, while `_unsorted/` keeps its bounds exemption (never split/merged/flattened). No behavioral change for a well-shaped tree (idempotency) — the new passes propose nothing on a tree without over-size files, duplicate coverage, or `_unsorted/` staging.

**Prose optimization** (260620-skop): skill content trimmed to remove re-explanation of partial-owned concepts (the §7 bundle-relative payoff, the "reorg authors ONE mechanical file / synthesis lives in backfill" delegation, the older-binary fallback, and the "indexes are generated, not hand-authored" rule now stated once and referenced elsewhere) and a `## Contents` TOC added; no behavioral change — the 5-step apply, strict-order compatibility orchestration, no-dangling-link hard block, and decline=no-mutation logic are unchanged (Flow / Tools / Sub-agents unchanged).

**No change-ids in authored descriptions** (260717-3plm): reorg authors `description:` frontmatter in its description-authoring paths — the new-topic-file / new-sub-domain-stub `description:` (apply Step 3) and the `removed-domains.md` `description:` one-liner (compatibility orchestration Step 1). Per the FKF §3.2 clarification amended by this change, each such `description:` is **free of change-ids** (a routing signal, not a provenance record). The `removed-domains.md` **body tombstone rows keep their change IDs verbatim** — a citation-carrying removal record, not transition narration; the ban applies to the frontmatter description only. No body-style rewrite rule is added: reorg moves/merges whole files and authors only descriptions plus the mechanical tombstone rows, so it authors no narratable body prose. No Flow/Tools/Sub-agents change.

## FKF-Aware Moves (`$(fab kit-path)/reference/fkf.md` §7)

Memory↔memory links are **bundle-relative** (`](/{domain}[/{sub}]/{file}.md)`, resolved from `docs/memory/`), so a move's link blast radius is small: only links whose **bundle path changes** (links *to* a file that changes domain/sub-domain) need rewriting; a same-domain reshuffle that keeps files in the domain rewrites nothing — the §7 payoff ("reorg rewrites *far* fewer links"). Two invariants the move path enforces:

- **Frontmatter-preserving moves.** A moved topic file keeps its FKF frontmatter (`type: memory` + `description:`) **byte-for-byte** — moving never strips, regenerates, or re-stamps it. A genuinely *new* topic file a split creates is stamped with `type: memory` alongside its synthesized `description:` so it is FKF-conforming.
- **Bundle-relative link rewrites.** Each broken link's path-after-`/` is updated to the moved file's new bundle path, `#anchor` preserved. The no-dangling-link guard keys on residual `](/…)` memory links (a hard block, with the same abort-escape rollback as before).

## Compatibility Migration (pre-fab-kit trees)

Detection is a **mechanical Go primitive**, not prose the agent re-derives: the skill runs `fab memory-index --check --json` and consumes its severity tier + JSON loss report (single source of truth — the classifier lives in `internal/memoryindex` where the generator's own logic does). Exit 0 (clean) / exit 1 (benign drift) ⇒ **no compatibility findings**, the `## Compatibility` section is omitted (born-compatible case). Exit 2 (destructive loss) ⇒ the JSON `losses[]` enumerate the divergences by `category`, mapped into the approve-before-mutate findings report:

- **`description`** — a curated description that would regenerate to `—` (the file lacks `description:` frontmatter; the generator reads descriptions exclusively from that field).
- **`tombstone`** — a row whose **`docs/memory/`-relative link target is absent on disk** (the generator drops removal-history rows it walks no folder for; external/absolute links are excluded by the primitive — no false positives; strikethrough is a non-required corroborating hint). Candidates are **user-confirmed** before relocation.
- **`grouping`** — a custom structural heading in the root `index.md` beyond the domains-only table (e.g. `### Apps` / `### Packages`) — flattened by the domains-only regen.

The findings-report (Step 3) and on-approval orchestration (Step 5) behavior is **unchanged** — they now consume the primitive's structured output instead of re-deriving `frontmatter.Field` semantics / tombstone heuristics / the flatten rule in prose. **One call feeds three consumers** (260718-dsrx): the same `fab memory-index --check --json` invocation also records the report's **`warnings[]`** array — `file-size` → the Shape Report over-size file rows (split candidates), `unsorted-nonempty` → the `_unsorted/` triage pass — alongside the `losses[]` compatibility findings. **Older-binary fallback**: when `fab memory-index --check`'s loss tiers / `--json` / the `warnings` key are unavailable, the skill falls back to the legacy prose detection for `losses[]` (same three signals, judged by eye during the read-all-files pass), measures file-size split candidates from the read-all-files line counts, and reads `_unsorted/` by direct folder listing — then warns the user to upgrade `fab`.

On **explicit approval** (decline = report findings and stop, no mutation), the remediation runs in **strict order**:

1. **Relocate tombstones → `docs/memory/_shared/removed-domains.md`** — the skill authors this **ONE** mechanical file (a `description:` frontmatter one-liner + H1 + the confirmed tombstone rows lifted verbatim, change IDs preserved). **Merge-not-duplicate** if it already exists (idempotent). This is the only per-file content reorg authors — bounded to mechanical row relocation, NOT per-file description synthesis.
2. **Dispatch `/docs-hydrate-memory` backfill mode** as a **general-purpose sub-agent** (standard subagent context). The dispatch **names the operation** ("backfill this tree"), passes **no file manifest** (backfill re-scans `docs/memory/` itself — the loose seam), and **signals the defer-regen caller form** so the sub-agent does not run `fab memory-index`.
3. **Rebalance + regenerate** — the existing split/merge/flatten apply, then `fab memory-index` **once** (reorg owns the single regen; the sub-agent ran zero). With compatibility findings only (no structural migrations), reorg runs `fab memory-index` once directly.

The competence seam: reorg's job stays structural (detect, relocate the mechanical row file, rebalance); **per-file description synthesis lives in hydrate's backfill mode**, which reorg *calls* rather than absorbs. Idempotency (a fab-kit design principle): re-running on an already-converted tree is a no-op — no duplicate tombstone rows, backfill skips frontmatter-present files, byte-stable index.

## Apply Path

For any move-bearing migration the skill, on approval:

1. **Moves files / sections** to their new paths (`git mv` semantics where possible; plain move otherwise), **preserving the moved file's FKF frontmatter** (`type: memory` + `description:`) byte-for-byte.
2. **Rewrites bundle-relative links** broken by the move — the dominant case being links *to* a moved file (their path-after-`/` changes), `#anchor` preserved — per the proposal's **Link Impact** list. A link to a sibling whose bundle path did not change needs no edit (§7).
3. **Authors `description:` frontmatter** (+ `type: memory`) — the curated index fields — on any new topic file a split creates, and creates each new sub-domain's stub `index.md` (only the `description:` frontmatter one-liner) **before** step 4's `fab memory-index` runs; the generator fills the body and round-trips the description.
4. **Regenerates indexes (and logs)** via `fab memory-index` (root + every domain + every sub-domain index, plus each folder's `log.md` — seed-merging any `log.seed.md` beneath the git projection, FKF §6). Generated rows are never hand-edited — the frontmatter stub is the only hand-curated part. The index carries no dates (a pure function of content); recency lives in `log.md`.
5. **Enforces a no-dangling-link guard**: a residual broken `](/…)` bundle-relative link is a hard block — that migration is not finalized until every broken link is rewritten. **Abort escape**: if a dangling link cannot be rewritten (target genuinely gone, or ambiguous), that migration is rolled back instead — moves and link rewrites restored, `fab memory-index` re-run — and the remaining approved migrations continue.

### Link Impact (mandatory before approval)

For any `split-domain` / `merge-domain` / `flatten` / `move` migration (any move-bearing `Kind`), the proposal MUST list every bundle-relative link that would break, each paired with its rewrite (e.g. `](/pipeline/runtime-agents.md)` → `](/pipeline/runtime/runtime-agents.md)`), so the blast radius is visible before the user approves. Under FKF §7 this radius is small — only links whose bundle path changes are listed; same-domain reshuffles that keep files in the domain break nothing. A migration that breaks no links states "Link Impact: none" explicitly.

Link rewriting is **skill-driven** (the agent edits each link per the Link Impact list) — it is NOT a `fab` Go subcommand (Pure Prompt Play — a fab-kit design principle).

### External Sub-Domain Addressing

A split creates `docs/memory/{domain}/{sub-domain}/{topic}.md`. Sub-domains are first-class, externally addressed: a sub-domained file is referenced as `{domain}/{sub-domain}/{file-name}` (the flat `{domain}/{file-name}` form remains valid for un-split domains). `fab memory-index` generates a `{domain}/{sub-domain}/index.md` for every sub-domain and references it from the parent domain index, so the selective-load walk is domain index → sub-domain index → file.

## Flow

```
User invokes /docs-reorg-memory
│
├─ Pre-flight: docs/memory/index.md and domain files must exist
├─ Read: all memory files across all domains (recursing into sub-domains)
│        + ONE fab memory-index --check --json call feeding THREE consumers:
│            losses[] (description/tombstone/grouping) → compatibility detection (exit 2)
│            warnings[] file-size → Shape Report over-size file rows (split candidates)
│            warnings[] unsorted-nonempty → _unsorted/ triage pass
│          older-binary ⇒ legacy prose fallback (losses), read-pass line counts (file-size), folder listing (_unsorted/)
├─ Diagnose: Shape Report (per-folder counts + folder-level depth vs ~5/~12 width + folder-depth ≤2 bounds
│            + over-size FILE rows: split candidates when ≥2 topic clusters, long-but-cohesive reported not split)
│            + Compatibility report (only when a pre-fab-kit divergence is found)
│            + Duplicate Coverage table (same topic in 2+ files; only when found)
│            + _unsorted/ Triage (per-file move/delete; only when _unsorted/ non-empty)
├─ (identify themes, propose reorganization: move-section/split-domain/merge-domain/flatten/move/split-file/merge-file)
├─ (present plan + Link Impact + Compatibility findings + Duplicate Coverage + _unsorted/ Triage, ask for approval)
│
├─ [if compatibility findings approved, strict order]
│  ├─ Write: docs/memory/_shared/removed-domains.md (description: + H1 + verbatim tombstone rows; merge-not-dup)
│  ├─ Dispatch: /docs-hydrate-memory backfill mode (general-purpose sub-agent, names op, no manifest, defer-regen)
│  └─ (then the structural apply below; fab memory-index runs once at the end — reorg owns the regen)
│  └─ [if declined] report findings, stop — no mutation
│
└─ [if approved, per migration]
   ├─ Write/Edit: move files/sections (FKF frontmatter preserved); split-file (verbatim bodies → new topic files,
   │              new type: memory + change-id-free description:, dominant topic kept/removed) / merge-file (B's unique
   │              sections → canonical A, delete emptied B); rewrite broken bundle-relative links (split-file: anchored
   │              → heading's new file, un-anchored → dominant-topic file); author description: + sub-domain index stubs
   │  _unsorted/ triage: move (rides move kind) / delete (explicit per-file confirmation)
   ├─ Bash: fab memory-index   (regenerates root + domain + sub-domain indexes deterministically)
   └─ Verify: no headings lost, no dangling `](/…)` bundle-relative link (hard guard) → finalize
              (unrewritable link / ambiguous split-file retarget → abort escape: roll back migration, re-run memory-index, continue)
```

### Tools used

| Tool | Purpose |
|------|---------|
| Read | All memory files and indexes (themes + shape diagnosis, incl. over-size file rows + duplicate-coverage heading overlap) |
| Write/Edit | Moved files (FKF frontmatter preserved), `split-file`/`merge-file` file fans/folds, rewritten bundle-relative links, new `description:` + `type: memory` frontmatter (only with approval); the one mechanical `_shared/removed-domains.md` on compatibility approval; `_unsorted/` triage moves + confirmed deletes |
| Bash | `fab memory-index --check --json` — ONE call feeding three consumers: mechanical compatibility detection (exit 2 + `losses[]`), Shape Report over-size file rows (`warnings[]` `file-size`), and `_unsorted/` triage (`warnings[]` `unsorted-nonempty`); older-binary ⇒ legacy prose + read-pass line counts + folder listing. `fab memory-index` to regenerate indexes (domain + sub-domain tiers) after approved moves; once at end of compatibility orchestration |
| Agent | Dispatch `/docs-hydrate-memory` backfill mode as a general-purpose sub-agent during compatibility orchestration |

### Sub-agents

`/docs-hydrate-memory` (backfill mode) — dispatched as a general-purpose sub-agent during compatibility orchestration to synthesize per-file `description:` frontmatter (standard subagent context; defer-regen signal so reorg owns the single `fab memory-index` regen).
