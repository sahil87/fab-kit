# docs-reorg-memory

## Summary

Analyzes memory files for themes and suggests reorganization. Read-only unless user approves changes. Identifies up to 10 themes, produces a **Shape Report** (folders over the ~12 width bound / over depth 3 / under the ~5 floor; reserved domains `_shared`/`_unsorted` exempt), and proposes a reorg plan. On approval it is the **memory rebalancer**: it performs the approved moves (split / merge / flatten / move), rewrites the relative links the moves break, adds `description:` frontmatter to any new file or sub-domain index, and regenerates indexes via `fab memory-index`. Index files (domain and sub-domain tiers) are regenerated by `fab memory-index`, never hand-edited.

## Apply Path

For any move-bearing migration the skill, on approval:

1. **Moves files / sections** to their new paths (`git mv` semantics where possible; plain move otherwise).
2. **Rewrites relative links** broken by the move — both directions (links *from* moved files to siblings, and links *to* moved files from elsewhere in the domain) — per the proposal's **Link Impact** list.
3. **Adds `description:` frontmatter** to any new file or new sub-domain `index.md` a split creates (the generated index reads it).
4. **Regenerates indexes** via `fab memory-index` (root + every domain + every sub-domain index).
5. **Enforces a no-dangling-link guard**: a residual broken relative link is a hard block — that migration is not finalized until every broken link is rewritten.

### Link Impact (mandatory before approval)

For any `split` / `merge` / `flatten` / `move` migration, the proposal MUST list every relative link that would break, each paired with its rewrite (e.g. `](runtime-agents.md)` in `execution-skills.md` → `](runtime/runtime-agents.md)`), so the blast radius is visible before the user approves. A migration that breaks no links states "Link Impact: none" explicitly.

Link rewriting is **skill-driven** (the agent edits each link per the Link Impact list) — it is NOT a `fab` Go subcommand (Constitution I, Pure Prompt Play).

### External Sub-Domain Addressing

A split creates `docs/memory/{domain}/{sub-domain}/{topic}.md`. Sub-domains are first-class, externally addressed: a sub-domained file is referenced as `{domain}/{sub-domain}/{file-name}` (the flat `{domain}/{file-name}` form remains valid for un-split domains). `fab memory-index` generates a `{domain}/{sub-domain}/index.md` for every sub-domain and references it from the parent domain index, so the selective-load walk is domain index → sub-domain index → file.

## Flow

```
User invokes /docs-reorg-memory
│
├─ Pre-flight: docs/memory/index.md and domain files must exist
├─ Read: all memory files across all domains (recursing into sub-domains)
├─ Diagnose: Shape Report (per-folder counts + depth vs ~5/~12/depth-3 bounds)
├─ (identify themes, propose reorganization: split/merge/flatten/move)
├─ (present plan + Link Impact for any move-bearing migration, ask for approval)
│
└─ [if approved, per migration]
   ├─ Write/Edit: move files/sections; rewrite broken relative links; add description: frontmatter
   ├─ Bash: fab memory-index   (regenerates root + domain + sub-domain indexes deterministically)
   └─ Verify: no headings lost, no dangling relative link (hard guard) → finalize
```

### Tools used

| Tool | Purpose |
|------|---------|
| Read | All memory files and indexes |
| Write/Edit | Moved files, rewritten relative links, new `description:` frontmatter (only with approval) |
| Bash | `fab memory-index` to regenerate indexes (domain + sub-domain tiers) after approved moves |

### Sub-agents

None.
