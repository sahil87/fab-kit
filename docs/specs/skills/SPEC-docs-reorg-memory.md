# docs-reorg-memory

## Summary

Analyzes memory files for themes and suggests reorganization. Read-only unless user approves changes. Identifies up to 10 themes, produces a **Shape Report** (folders over the ~12 width bound / over depth — folder depth >2, i.e. topic files past 3 path segments / under the ~5 floor; reserved domains `_shared`/`_unsorted` exempt), and proposes a reorg plan whose Migration Map `Kind` is one of `move-section` / `split-domain` / `merge-domain` / `flatten` / `move`. On approval it is the **memory rebalancer**: it performs the approved migrations, rewrites the relative links the moves break, authors `description:` frontmatter on new files and creates each new sub-domain's `description:`-only index stub before `fab memory-index` runs, and regenerates indexes via `fab memory-index`. Index files (domain and sub-domain tiers) are regenerated by `fab memory-index`, never hand-edited. It also **orchestrates a one-time compatibility migration** for pre-fab-kit memory trees (see below).

## Compatibility Migration (pre-fab-kit trees)

During its Step 1 read-all-files pass, the skill also detects three ways a hand-curated, pre-fab-kit tree diverges from the convention `fab memory-index` depends on, surfaced in the approve-before-mutate findings report (a `## Compatibility` section, omitted entirely when no divergence is found):

- **Missing `description:` frontmatter** on topic files — `frontmatter.Field` semantics (no frontmatter, or no `description:` key, counts as missing). The generator reads descriptions exclusively from this field, so a missing one renders `—` on regen.
- **Tombstone rows** — a row in an existing hand-curated index whose **`docs/memory/`-relative link target is absent on disk** (the **primary** signal); strikethrough `~~...~~` is a **corroborating hint**, not required (un-struck tombstones are caught); the signal is scoped to relative paths so external links never false-positive. Candidates are **user-confirmed** before relocation. These are the removal-history rows the generator silently drops.
- **Custom structural groupings** in the root `index.md` beyond the domains-only table (e.g. `### Apps` / `### Packages` / `### Cross-cutting`) — flattened by the domains-only regen.

On **explicit approval** (decline = report findings and stop, no mutation), the remediation runs in **strict order**:

1. **Relocate tombstones → `docs/memory/_shared/removed-domains.md`** — the skill authors this **ONE** mechanical file (a `description:` frontmatter one-liner + H1 + the confirmed tombstone rows lifted verbatim, change IDs preserved). **Merge-not-duplicate** if it already exists (idempotent). This is the only per-file content reorg authors — bounded to mechanical row relocation, NOT per-file description synthesis.
2. **Dispatch `/docs-hydrate-memory` backfill mode** as a **general-purpose sub-agent** (standard subagent context). The dispatch **names the operation** ("backfill this tree"), passes **no file manifest** (backfill re-scans `docs/memory/` itself — the loose seam), and **signals the defer-regen caller form** so the sub-agent does not run `fab memory-index`.
3. **Rebalance + regenerate** — the existing split/merge/flatten apply, then `fab memory-index` **once** (reorg owns the single regen; the sub-agent ran zero). With compatibility findings only (no structural migrations), reorg runs `fab memory-index` once directly.

The competence seam: reorg's job stays structural (detect, relocate the mechanical row file, rebalance); **per-file description synthesis lives in hydrate's backfill mode**, which reorg *calls* rather than absorbs. Idempotency (Constitution III): re-running on an already-converted tree is a no-op — no duplicate tombstone rows, backfill skips frontmatter-present files, byte-stable index.

## Apply Path

For any move-bearing migration the skill, on approval:

1. **Moves files / sections** to their new paths (`git mv` semantics where possible; plain move otherwise).
2. **Rewrites relative links** broken by the move — both directions (links *from* moved files to siblings, and links *to* moved files from elsewhere in the domain) — per the proposal's **Link Impact** list.
3. **Authors `description:` frontmatter** — the single hand-curated index field — on any new topic file a split creates, and creates each new sub-domain's stub `index.md` (only the `description:` frontmatter one-liner) **before** step 4's `fab memory-index` runs; the generator fills the body and round-trips the description.
4. **Regenerates indexes** via `fab memory-index` (root + every domain + every sub-domain index). Generated rows and "Last Updated" cells are never hand-edited — the frontmatter stub is the only hand-curated part.
5. **Enforces a no-dangling-link guard**: a residual broken relative link is a hard block — that migration is not finalized until every broken link is rewritten. **Abort escape**: if a dangling link cannot be rewritten (target genuinely gone, or ambiguous), that migration is rolled back instead — moves and link rewrites restored, `fab memory-index` re-run — and the remaining approved migrations continue.

### Link Impact (mandatory before approval)

For any `split-domain` / `merge-domain` / `flatten` / `move` migration (any move-bearing `Kind`), the proposal MUST list every relative link that would break, each paired with its rewrite (e.g. `](runtime-agents.md)` in `execution-skills.md` → `](runtime/runtime-agents.md)`), so the blast radius is visible before the user approves. A migration that breaks no links states "Link Impact: none" explicitly.

Link rewriting is **skill-driven** (the agent edits each link per the Link Impact list) — it is NOT a `fab` Go subcommand (Constitution I, Pure Prompt Play).

### External Sub-Domain Addressing

A split creates `docs/memory/{domain}/{sub-domain}/{topic}.md`. Sub-domains are first-class, externally addressed: a sub-domained file is referenced as `{domain}/{sub-domain}/{file-name}` (the flat `{domain}/{file-name}` form remains valid for un-split domains). `fab memory-index` generates a `{domain}/{sub-domain}/index.md` for every sub-domain and references it from the parent domain index, so the selective-load walk is domain index → sub-domain index → file.

## Flow

```
User invokes /docs-reorg-memory
│
├─ Pre-flight: docs/memory/index.md and domain files must exist
├─ Read: all memory files across all domains (recursing into sub-domains)
│        + compatibility detection: missing description: frontmatter, tombstone rows, custom groupings
├─ Diagnose: Shape Report (per-folder counts + folder-level depth vs ~5/~12 width + folder-depth ≤2 bounds)
│            + Compatibility report (only when a pre-fab-kit divergence is found)
├─ (identify themes, propose reorganization: move-section/split-domain/merge-domain/flatten/move)
├─ (present plan + Link Impact + Compatibility findings, ask for approval)
│
├─ [if compatibility findings approved, strict order]
│  ├─ Write: docs/memory/_shared/removed-domains.md (description: + H1 + verbatim tombstone rows; merge-not-dup)
│  ├─ Dispatch: /docs-hydrate-memory backfill mode (general-purpose sub-agent, names op, no manifest, defer-regen)
│  └─ (then the structural apply below; fab memory-index runs once at the end — reorg owns the regen)
│  └─ [if declined] report findings, stop — no mutation
│
└─ [if approved, per migration]
   ├─ Write/Edit: move files/sections; rewrite broken relative links; author description:
   │              frontmatter + sub-domain index stubs (description: only, before memory-index)
   ├─ Bash: fab memory-index   (regenerates root + domain + sub-domain indexes deterministically)
   └─ Verify: no headings lost, no dangling relative link (hard guard) → finalize
              (unrewritable link → abort escape: roll back migration, re-run memory-index, continue)
```

### Tools used

| Tool | Purpose |
|------|---------|
| Read | All memory files and indexes; compatibility detection (frontmatter, unresolved relative links, root groupings) |
| Write/Edit | Moved files, rewritten relative links, new `description:` frontmatter (only with approval); the one mechanical `_shared/removed-domains.md` on compatibility approval |
| Bash | `fab memory-index` to regenerate indexes (domain + sub-domain tiers) after approved moves; once at end of compatibility orchestration |
| Agent | Dispatch `/docs-hydrate-memory` backfill mode as a general-purpose sub-agent during compatibility orchestration |

### Sub-agents

`/docs-hydrate-memory` (backfill mode) — dispatched as a general-purpose sub-agent during compatibility orchestration to synthesize per-file `description:` frontmatter (standard subagent context; defer-regen signal so reorg owns the single `fab memory-index` regen).
