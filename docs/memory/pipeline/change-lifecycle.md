# Change Lifecycle

**Domain**: pipeline

## Overview

A "change" is the unit of work in Fab. Each change lives in its own folder under `fab/changes/`, tracks its progress via `.status.yaml`, and is identified by a unique name. This doc covers the naming convention, folder structure, status tracking, active change pointer, git integration, and the utility skills (`/fab-status`, `/fab-switch`) that navigate between changes.

## Requirements

### Folder Naming Convention

Change folders SHALL use the format `{YYMMDD}-{XXXX}-{slug}`:

| Component | Generated by | Purpose | Example |
|-----------|-------------|---------|---------|
| `YYMMDD` | Agent (today's date) | Chronological sort, temporal context | `260115` |
| `XXXX` | Agent (4 random lowercase alphanumeric) | Uniqueness guarantee | `a7k2` |
| `slug` | Agent (2-6 words from description) | Human readability | `add-oauth` |

All components MUST be lowercase. The name is unique by construction (date + random token), requires no collision scanning, and sorts chronologically in `ls`. The `{YYMMDD}-{XXXX}` prefix is immutable ‚Äî only the slug can be changed via `changeman.sh rename`. When the change originates from a Linear ticket, issue IDs are stored in the `issues` array in `.status.yaml` (via `statusman.sh add-issue`) ‚Äî not embedded in the folder name.

**Example**: `260115-a7k2-add-oauth`

### Active Change Tracking (`.fab-status.yaml` symlink)

`.fab-status.yaml` is a symlink at the repository root that points to the active change's `.status.yaml`. It replaces the former `fab/current` pointer file. The symlink target is always a relative path from the repo root (e.g., `fab/changes/260302-ye59-example-change/.status.yaml`). This collapses change identity and status into a single filesystem entry ‚Äî `readlink` gives the change name (from the target path), `cat` gives the full status.

```
.fab-status.yaml ‚Üí fab/changes/260302-ye59-example-change/.status.yaml
```

**Observation model**: All ephemeral per-worktree state lives as two sibling files at the repo root: `.fab-status.yaml` (symlink ‚Üí change identity + pipeline state) and `.fab-runtime.yaml` (agent idle state). Both gitignored, both at the same level, scannable with a single glob. This reduces cross-worktree observation from three reads to two (no separate pointer indirection).

**Access pattern**: `resolve.go` reads `.fab-status.yaml` (via `readlink` for default-mode resolution), and since mz4q `archive.go` also reads the link directly (`os.Readlink` + `ExtractFolderFromSymlink`) to capture whether the change being archived is the active one **before** renaming its folder into `archive/` ‚Äî resolve now validates the pointer target, so a post-rename resolution would see a dangling pointer and never report the archived change as active, silently leaving the pointer behind. Only `change.go` writes `.fab-status.yaml`, through the shared `setActivePointer` helper (used by both `switch` and `rename`): the new symlink is created at a temp name in the repo root and `os.Rename`d over `.fab-status.yaml` ‚Äî rename atomically replaces the old link on POSIX, eliminating both the empty-pointer window and the concurrent-switch EEXIST race of the prior `os.Remove` + `os.Symlink` sequence. No skill or other script reads or writes the symlink directly ‚Äî the mechanism is an implementation detail encapsulated by these modules.

**Lifecycle**:
- **Created** by `fab change switch` (called by `/fab-switch`) ‚Äî creates symlink pointing to `fab/changes/{name}/.status.yaml`
- **Updated** by `fab change switch` ‚Äî the old symlink is atomically replaced via `setActivePointer` (temp symlink + `os.Rename`); a concurrent reader sees either the old or the new pointer, never a missing one
- **Updated** by `fab change rename` ‚Äî symlink re-pointed to the renamed folder's `.status.yaml` via the same atomic helper; the update stays best-effort (error discarded), and with the atomic swap a failure now leaves the **old** pointer in place rather than no pointer at all (the dangling old pointer is then handled by resolve's fall-through below)
- **Created** by `/fab-new` ‚Äî `/fab-new` calls `fab change switch "{name}"` after advancing intake to ready (Step 10), creating the symlink as a side effect. The newly created change is immediately active; no separate `/fab-switch` step is required
- **Not created** by `/fab-draft` ‚Äî `/fab-draft` performs identical Steps 1‚Äì9 to `/fab-new` (create folder, initialize `.status.yaml`, generate intake, advance to ready) but does NOT call `fab change switch` and does NOT create or modify `.fab-status.yaml`. The user must run `/fab-switch {name}` to activate a draft change
- **Read** by `resolve.go` ‚Äî `readlink` extracts the folder name from the target path. All other scripts and skills resolve the active change via `fab resolve` (which delegates to `resolve.go`)
- **Removed** by `fab change switch --none` (called by `/fab-switch --none` or `/fab-archive`) ‚Äî symlink is deleted to deactivate the current change
- **Optionally created** by `fab change switch` (called by `/fab-archive restore --switch`) ‚Äî created for the restored change when `--switch` flag is used

**Resolution pattern** (used by all skills via `fab resolve` ‚Üí `resolve.go`):
1. If the `.fab-status.yaml` symlink exists, extract the folder name from the target path, then `os.Stat` the target `fab/changes/<folder>/.status.yaml`; when it exists, return the folder (primary path). A **dangling pointer** (target archived/deleted underneath the gitignored symlink ‚Äî folder archived from another worktree, branch switch, manual delete) falls through to step 2 instead of returning the stale folder; the link is NOT removed, since `fab resolve` is documented as a pure query with no side effects. (Before mz4q the pointer was trusted without validating the target: `fab resolve` silently printed the stale folder while `fab preflight`/`fab score` failed with misleading messages)
2. If `.fab-status.yaml` is absent or broken, attempt single-change guessing:
   - Enumerate non-archive folders in `fab/changes/` that contain a valid `.status.yaml`
   - Exactly 1 candidate ‚Üí return it, emit `(resolved from single active change)` to stderr
   - 0 candidates ‚Üí exit 1 with `No active change. Run /fab-new <description> to start one, or /fab-switch to activate an existing one.` (the missing-`fab/changes/`-directory path emits the same actionable message ‚Äî honoring `_preamble.md`'s promise that preflight stderr contains the specific error and suggested fix)
   - 2+ candidates ‚Üí exit 1 with `No active change (multiple changes exist ‚Äî use /fab-switch).`
3. Guessing is transparent to callers ‚Äî `preflight.sh`, `/git-branch`, and `/git-pr` all call `fab resolve` and benefit automatically

`.fab-status.yaml` SHOULD be added to `.gitignore` since it is local working state (each developer has their own active change).

### Status Tracking (`.status.yaml`)

Every change folder SHALL contain a `.status.yaml` manifest with these fields:

- `id` ‚Äî the 4-character change ID (the `XXXX` component of the folder name). Derived from `name` at creation time, immutable. Makes the change ID directly available without parsing the folder name
- `name` ‚Äî the full change folder name
- `created` ‚Äî ISO 8601 datetime
- `progress` ‚Äî map of 6 stages to their state (`intake`, `apply`, `review`, `hydrate`, `ship`, `review-pr`). The stage marked `active` is the current stage (single source of truth for where the change is). There is no separate `stage:` field ‚Äî current stage is derived from the `active` entry in the progress map. The `tasks` key was removed in qszh (plan generation became an apply-internal sub-step); the `spec` key was removed in j6cs ‚Äî requirement capture was absorbed into apply entry as `plan.md`'s `## Requirements` section, so there is no separate `spec` stage. The binary tolerates an orphan `progress.spec` key on an un-migrated `.status.yaml` (`Validate()` skips it; `GetProgressMap()` omits it); only the `1.9.7-to-1.10.0` migration removes it.
- `plan` ‚Äî generation status and counts: `generated: bool`, `task_count: int`, `acceptance_count: int`, `acceptance_completed: int`. Replaces the prior `checklist:` block (renamed in qszh, location is fixed at change root, the prior `path` field was dropped). Counts are maintained by the PostToolUse `artifactBookkeeping` hook on every `plan.md` write (heading-bounded section parse). The legacy `set-checklist` CLI command was renamed to `set-acceptance`.
- `confidence` ‚Äî SRAD confidence scoring: `certain`, `confident`, `tentative`, `unresolved` counts, and derived `score` (0.0-5.0). Initialized to 0.0 (no assessed confidence). Computed by `fab score`, which reads `intake.md` only (the sole scoring source as of j6cs) ‚Äî invoked at the intake stage by `/fab-new` and `/fab-clarify`. The `confidence.indicative` flag was **retired in j6cs**: it is no longer written by any code path. The binary keeps a decode-tolerant `Indicative *bool` field so a legacy `indicative: true` key on an un-migrated/archived file round-trips harmlessly, but `encodeConfidence` never emits it. Used as the **single intake gate** by `/fab-ff` and `/fab-fff` (flat threshold 3.0 for all change types via `fab score --check-gate --stage intake`, obtained through `getGateThreshold`); the separate per-type spec gate was removed. Displayed as `{score} of 5.0`. Consumers (`/fab-status`, `/fab-switch`, `fab change list`) read uniformly from `.status.yaml`
- `stage_metrics` ‚Äî per-stage operational data map. Each stage that has been activated gets an entry with flow-style YAML: `{started_at, driver, iterations, completed_at}`. Automatically managed by event commands (`start`, `advance`, `finish`, `reset`, `fail`) via `_apply_metrics_side_effect`. On `start` (`active`): creates entry with `started_at`, `driver`, `iterations` (incremented, not reset ‚Äî tracks rework cycles); also emits a `stage-transition` event via `logman.sh transition` (best-effort). On `finish` (`done`): sets `completed_at`. On `advance` (`ready`): no-op (preserves existing metrics from active phase). On `reset`: transitions the target stage back to `active` and cascades all downstream stages to `pending`; when stages are set to `pending` or `skipped` during this cascade, an entry with `iterations > 0` keeps only its `iterations` counter ‚Äî the timing fields (`started_at`, `driver`, `completed_at`) are cleared and rewritten on the next activation ‚Äî while zero-iteration entries are removed entirely (k4ge; previously every cascaded entry was deleted, which zeroed the review cycle counter that `fab pr-meta` reports, so PR meta always showed "1 cycle" regardless of rework). On `fail` (`failed`): no-op (preserves timing data). Initialized as empty `stage_metrics: {}` in the template. **`iterations` semantics**: counts per-stage activations ‚Äî each time a stage transitions to `active`, `iterations` is incremented by 1. First activation = 1, each rework re-entry increments by 1. It is NOT the count of apply‚Üíreview pairs. The `iterations` value determines the `stage-transition` event's `action` field: `iterations == 1` ‚Üí `"enter"`, `iterations > 1` ‚Üí `"re-entry"`
- `summary` ‚Äî optional per-change one-line log summary (added in 5943). It is the FKF C-lite source line `fab memory-index` joins with git history to generate `log.md` (see [fkf.md](../../specs/fkf.md) ¬ß6.3 ‚Äî `log.md` generation is a later FKF change). Modeled exactly on `change_type_source` (`yaml:"summary,omitempty"`, drop-when-empty in `syncToRaw`, insert-when-absent before `last_updated`): an empty/absent summary serializes to nothing and a change with no summary degrades gracefully (the generator falls back to the change slug). Written/read via `fab status set-summary <change> <text>` / `get-summary <change>` ‚Äî the conflict-free write path (each change touches only its own `.status.yaml`). It is authored **once during the change** ‚Äî at hydrate, or carried from the intake (¬ß6.3) ‚Äî but 5943 only *creates the field + verbs*; **no stage auto-populates it** (authoring wiring is deferred to a later FKF change). The template seeds `summary: ""`; migration `2.4.2-to-2.5.0` adds it to in-flight changes (idempotent, skips `archive/**`)
- `last_updated` ‚Äî refreshed on every status change. All `.status.yaml` mutations SHOULD go through `lib/statusman.sh` write functions (or CLI commands), which handle validation and `last_updated` refresh automatically

**Write concurrency & durability** (mz4q, binary-review B1 F02/F03/F06/F07):

- **Flock-serialized writes**: every `.status.yaml` load-mutate-save cycle runs under an exclusive advisory flock on the sibling `fab/changes/<folder>/.status.yaml.lock` (shared `internal/lockfile` helper ‚Äî `O_CREATE` + `LOCK_EX|LOCK_NB` retry, ~10s bounded acquisition so a pathological holder, e.g. a stage hook re-invoking `fab status` on the same change, errors instead of deadlocking; lock files are never deleted). Serialized writers: the 12 mutating `fab status` subcommands (via `withStatusLock` in `cmd/fab/status.go`), the artifact-write hook's bookkeeping, and `score.Compute`'s confidence persist. Read-only commands stay lock-free ‚Äî rename atomicity protects readers. This fixes the last-writer-wins race where hook bookkeeping holding a stale snapshot could silently revert a `fab status start/finish` from another pane. Both lock-sibling patterns are gitignored (repo `.gitignore`; the scaffold fragment shipped to user projects carries `.status.yaml.lock` ‚Äî its `.fab-*` line already covers the runtime lock)
- **Single-Save hook bookkeeping**: `fab hook artifact-write` mutates the loaded `StatusFile` in memory via non-saving `Apply*` variants (`ApplyChangeType`/`ApplyAcceptance`/`ApplyConfidence`/`ApplyConfidenceFuzzy` ‚Äî validate-before-mutate preserved; the `Set*` functions remain as Apply+Save wrappers, CLI contracts unchanged) plus `score.ComputeWithStatus(fabRoot, changeDir, intakeContent, statusFile)` (in-memory confidence mutation + confidence log; caller owns persistence and locking), then calls `Save` **exactly once** when dirty ‚Äî one serialize+rename and one `last_updated` bump per hook event, replacing up to 4 full-document Saves, 3 folder resolutions, 2 status loads, and 2 intake reads per event. `log.ConfidenceLog` now takes the already-resolved absolute change dir
- **Durability follows criticality**: `statusfile.Save` fsyncs the temp file before rename ‚Äî `.status.yaml` is the pipeline state machine's source of truth, so a crash never leaves an empty/torn file visible (the ephemeral `.fab-runtime.yaml` dropped its fsync in the same change ‚Äî see [runtime-agents.md](../runtime/runtime-agents.md))
- **Classified read errors**: `statusfile.Load` reports `status file not found: <path>` only for `os.IsNotExist`; permission-denied/is-a-directory/I/O failures wrap the real cause (`read status file <path>: <err>`) instead of masquerading as absence and routing agents down wrong recovery paths
- **No silent dropped writes**: `syncToRaw` inserts absent keys on write (generalizing the former `insertTrueImpact`-only logic) ‚Äî `issues`/`prs`/`plan`/`confidence`/`stage_metrics`/`last_updated` always, `name`/`change_type` when non-empty ‚Äî so writes to well-formed-but-sparse documents (restored pre-0.24.0 archives, hand-edited files) persist instead of exiting 0 having written nothing. `SetProgress` creates a missing stage key in an existing `progress:` mapping and returns an error on a malformed shape (`progress:` absent or not a mapping), propagated through the Start/Advance/Finish/Reset/Skip/Fail transitions ‚Äî previously a missing-stage `fab status start` persisted `stage_metrics` and `.history.jsonl` while silently dropping the progress transition. This is the write-time-insertion posture, NOT validate-at-Load refusal, which would conflict with the documented legacy tolerance for un-migrated/archived files

**State vocabulary** (all progress fields draw from this fixed set):

| State | Meaning | Used by |
|-------|---------|---------|
| `pending` | Not yet started | All stages |
| `active` | Work in progress (generating artifact, executing tasks) | Exactly one stage at a time |
| `ready` | Stage work product exists, eligible for advancement or clarification | Any number of stages (not counted toward active limit) |
| `done` | Completed successfully, advanced past | All stages |
| `failed` | Completed with failures requiring rework | review, review-pr |

**State transition table (event-driven)**:

| From State | Event   | To State        | Review Only? |
|------------|---------|-----------------|--------------|
| pending    | start   | active          | no           |
| failed     | start   | active          | YES          |
| active     | advance | ready           | no           |
| active     | finish  | done (+next)    | no           |
| ready      | finish  | done (+next)    | no           |
| ready      | reset   | active (+cascade) | no         |
| done       | reset   | active (+cascade) | no         |
| active     | fail    | failed          | YES          |

**Deriving current stage (routing)**: Use a three-tier fallback: (1) find the first stage with `active` or `ready` state, (2) if no active/ready entry, find the first `pending` stage after the last `done`/`skipped` stage, (3) if all stages are `done`/`skipped`, return `review-pr` (`CurrentStage`'s all-done fallback ‚Äî this doc previously mis-stated it as `hydrate`; corrected in k4ge). The second tier handles the post-reset state where the target stage was marked `done` but the next stage was left `pending`. Zero or one stage SHALL be `active` at any time (`ready` is not counted). Used by `/fab-continue` for dispatch and preflight's `stage` field. Skills distinguish `active` (generate) from `ready` (advance) via `display_state`.

**Deriving display stage**: A separate derivation for user-facing display (`DisplayStage`, `src/go/fab/internal/status/status.go`), a five-tier walk: (1) first `active` stage ‚Äî in-progress work supersedes a parked failure, (2) first `failed` stage ‚Äî a parked failure outranks ready/done so it surfaces instead of being masked by the last-done fallback (tier added in dkn3; only `review` and `review-pr` can hold `failed` per `AllowedStates`, so at most two candidates exist and the first in pipeline order wins), (3) first `ready` stage, (4) last `done`/`skipped` stage, (5) first `pending` stage if nothing started. Returns both the stage name and its state. Used by `/fab-status` and `/fab-switch` for the "Stage:" line, by preflight's `display_stage`/`display_state` fields, by `fab change list` rows, and by `fab pane map --json`'s `display_state` field (see [pane-commands.md](../runtime/pane-commands.md)). The distinction ensures users see "where you are" (display stage) separately from "what's next" (routing stage). `CurrentStage()` (routing) deliberately has no `failed` tier ‚Äî failed-state routing is owned by skill dispatch (`/fab-continue`'s review-failed row), not display derivation.

**Two-write transitions**: Moving from one stage to the next (in normal forward flow) requires two writes: (1) set the current stage to `done`, (2) set the next stage to `active`. Both writes MUST happen atomically in a single `.status.yaml` update. Use `fab status finish <change> <stage> [driver]` for forward transitions ‚Äî `finish` atomically marks the current stage `done` and activates the next pending stage. **Exception**: Reset flow cascades downstream stages to `pending`. Use `fab status reset <change> <stage> [driver]` to reset a stage to `active` and cascade all downstream stages to `pending`. Other event commands: `start` (pending/failed ‚Üí active), `advance` (active ‚Üí ready), `fail` (active ‚Üí failed, review only). The `driver` parameter is optional but skills always pass it.

**Stage hooks (`stage_hooks` in `fab/project/config.yaml`)**: `fab status start` and `fab status finish` honor an optional `stage_hooks.{stage}.pre/post` map ‚Äî live Go behavior (`config.go`/`status.go`) first documented in c5tr; `_cli-fab.md` ¬ß stage_hooks is the canonical reference (the scaffold does not seed the key ‚Äî add it by hand). A `pre` hook runs before `start`'s transition is applied and **blocks the stage from starting** on non-zero exit (the transition is not applied); a `post` hook runs after `finish`'s transition is saved ‚Äî the stage is already `done` and the next stage already auto-activated, so a failing post hook errors without unwinding the transition. Two sharp edges: `finish`'s auto-activation does NOT fire the next stage's `pre` hook (pre hooks fire only on an explicit `start`), and the **failing-post-hook re-run trap** ‚Äî re-running `finish` after fixing a failed post hook errors on done‚Üídone (`done` is not a valid `finish` source state); run the hook command by hand instead, or `reset` the stage first if the transition genuinely needs replaying. Hooks execute as `sh -c "<command>"` from the repo root with stdout/stderr passthrough; an absent/empty hook (or missing config file) is a silent no-op.

**Review failure backward movement**: When `/fab-continue` review behavior identifies issues requiring rework, it sets `review: failed` and moves `apply` back to `active` (the fail+reset pair). Apply is the only earlier execution stage (the `spec` stage was removed in j6cs) ‚Äî requirement-level rework edits `plan.md`'s `## Requirements` section and re-runs apply rather than resetting to a spec stage. In the `/fab-ff`/`/fab-fff` auto-rework loop the same fail+reset pair fires on **every** failed review verdict that starts a new cycle (choreography stated once in `_pipeline.md`, szxd) ‚Äî except at exhaustion: after the `{max_cycles}`-th failed cycle (the code-review.md ¬ß Rework Budget `Max cycles:` knob, default 3 ‚Äî c5tr) the orchestrator fires `fail` **only** (no reset), deliberately leaving `review: failed` (apply `done`) as the terminal resting state. From that state, a user-invoked `/fab-continue` does not re-run review ‚Äî its review-failed dispatch row resets apply and presents the rework menu directly; orchestrator re-runs instead recover via `start review` (`_pipeline.md` Resumability). Since dkn3 this resting state also *displays* honestly: `DisplayStage`'s `failed` tier returns (`review`, `failed`), so `/fab-status`, `fab change list`, and preflight's `display_stage`/`display_state` show `review ‚Äî failed` instead of falling through to the last done stage (`apply ‚Äî done`). Follow-up cleanup candidate (flagged by dkn3's review, not auto-applied): `fab-continue.md`'s review-failed dispatch row still keys on a direct `progress.review` read ‚Äî a compensating guard written when preflight's derived state could never be `failed`, now redundant since preflight surfaces `review`/`failed` directly. See [execution-skills.md](execution-skills.md).

### Event History (`.history.jsonl`)

Each change folder MAY contain a `.history.jsonl` file ‚Äî an append-only event log recording significant workflow events. Created automatically on first event. Four event types:

- **`command`** ‚Äî skill invocation: `{"ts":"...", "event":"command", "cmd":"fab-ff", "args":"..."}`
- **`confidence`** ‚Äî score change: `{"ts":"...", "event":"confidence", "score":4.2, "delta":"+0.3", "trigger":"calc-score"}`
- **`review`** ‚Äî review outcome: `{"ts":"...", "event":"review", "result":"passed"}` (or `"failed"` with optional `"rework":"..."`). Canonical `result` values are `"passed"` and `"failed"`. Validation is permissive ‚Äî non-canonical values (e.g., `"smoke-test"`, `"test-pass"`) are accepted for ad-hoc debugging in archived changes.
- **`stage-transition`** ‚Äî stage activation: `{"ts":"...", "event":"stage-transition", "stage":"apply", "action":"enter", "driver":"fab-ff"}` (first entry) or `{"ts":"...", "event":"stage-transition", "stage":"apply", "action":"re-entry", "from":"review", "reason":"fix-code", "driver":"fab-ff"}` (rework re-entry). The `action` field is `"enter"` when `iterations == 1` (first activation) and `"re-entry"` when `iterations > 1` (rework). `from` and `reason` are present only on re-entries.

Events are auto-logged as side effects: `preflight.sh --driver` logs command invocations, `statusman.sh finish/fail review` logs review outcomes, `statusman.sh` event functions log stage transitions (via `_apply_metrics_side_effect`), and `calc-score.sh` logs confidence changes via `logman.sh`. Skills never call `logman.sh` directly. The file is informational ‚Äî no skill reads it for logic. Useful for debugging workflow progression, understanding change history, and tracing rework loops.

### The 6 Stages

Changes progress through 6 stages in a defined graph:

```
intake ‚Üí apply ‚Üí review ‚Üí hydrate ‚Üí ship ‚Üí review-pr
```

The intake (`intake.md`) is created by `/fab-new`, which leaves the intake stage as `ready` (artifact exists, open for refinement). Intake is the sole planning stage and the sole confidence gate: generation leaves it at `ready`, users can run `/fab-clarify` to refine it, and `/fab-continue` finishes it while entering apply. After intake, the next stage is `apply` ‚Äî at apply entry the skill generates a unified `plan.md` carrying `## Requirements` (RFC-2119 + GIVEN/WHEN/THEN, absorbed from the former `spec.md`) plus `## Tasks` + `## Acceptance` in one pass, then executes tasks. The legacy `tasks` stage was removed in qszh; the `spec` stage was removed in j6cs (its requirement-capture work moved into apply entry, and `spec.md` no longer exists as an artifact). The artifact flow is `intake.md ‚Üí plan.md ‚Üí code`.

The stages split into four phases:
- **Planning** (1): intake (the only manual gate ‚Äî all human judgment is frontloaded here)
- **Execution** (2-3): apply (generates the unified `plan.md` at entry, then executes), review
- **Hydration** (4): hydrate (hydrates into memory files)
- **Integration** (5-6): ship (PR creation via `/git-pr`), review-pr (PR review feedback via `/git-pr-review`)

After hydrate completes, the change folder remains in `fab/changes/`. To move it to the archive, run `/fab-archive` ‚Äî a standalone housekeeping command (not a pipeline stage). `/fab-archive` moves the folder to `fab/changes/archive/yyyy/mm/` (date-bucketed by the folder's `YYMMDD` prefix), updates the archive index, marks backlog items done (exact-ID check always; keyword scan with interactive confirmation), and conditionally removes `.fab-status.yaml` if the archived change was active. Whether it was active is captured by reading the symlink directly (exact folder match) **before** the folder rename (mz4q ‚Äî a prerequisite of resolve's dangling-target validation, which would otherwise never see the post-rename pointer as the archived change); `pointer: cleared` is reported only when the symlink actually pointed at the archived change, `pointer: skipped` otherwise ‚Äî including when no symlink exists, where the old post-rename resolution path could have single-change-guessed the answer. To restore an archived change, run `/fab-archive restore <change-name>` ‚Äî this moves the folder back to `fab/changes/`, removes the index entry, and optionally activates via `--switch`. Existing flat archive entries are migrated to the date-bucketed structure via `/fab-setup migrations` (see `$(fab kit-path)/migrations/0.24.0-to-0.32.0.md`).

**Archive CLI exit semantics (k4ge)**: `fab change archive` requires exactly one argument (`cobra.ExactArgs(1)` ‚Äî zero args is a non-zero usage error, no longer exit-0 help text). Re-archiving an already-archived change is an idempotent soft skip ‚Äî `already archived: {change}` on stdout, exit 0 ‚Äî including the genuinely-archived case where the source folder is gone from `fab/changes/` (an archive-scan fallback in `Archive()`; see [execution-skills.md](execution-skills.md)). An argument matching no change anywhere (active or archived) still propagates the resolution error non-zero, and ambiguous archive matches do not soft-skip. Partial failure (the move succeeded but the backlog mark failed) prints the YAML report and exits non-zero ‚Äî re-running soft-skips. `fab batch archive` with no arguments defaults to `--list` ‚Äî it prints archivable changes and archives nothing; the bulk action requires explicit `--all` (260612-ye8r, aligned with `batch new`/`batch switch`). An empty `--all` set is a benign no-op (`No archivable changes found.` + `Archived 0, skipped 0, failed 0.` footer, exit 0); explicitly named targets that are genuinely archived route to the loop's `already archived, skipping` path (counted skipped, exit 0); the explicit-args path where nothing resolves anywhere keeps exit 1 (`No valid changes to archive.`).

**Backlog scan honesty (hv7t)**: `fab batch new --list`/`--all` no longer silently shrinks when `fab/backlog.md` cannot be read ‚Äî `backlog.ParsePending` returns `([]Item, error)` (it previously swallowed open errors and returned nil, indistinguishable from an empty backlog) and both paths propagate the error non-zero. Parsing is whole-file via `internal/lines`, so a >64KB line can no longer abort the scan mid-file: items below the long line appear in `--list`/`--all`, and a real ID after it no longer triggers a misleading `not found` error. `ExtractContent` keeps `(string, error)` with errors made accurate ‚Äî a genuinely missing ID is `not found in backlog`; read failures return the real cause. The per-ID warn-and-skip behavior of `fab batch new <id>...` is preserved (`Warning: [{id}] {err}, skipping`), now printing the actual error.

**Pipeline shortcuts**: `/fab-ff` fast-forwards from intake through hydrate, gated on a single intake confidence gate (flat 3.0 for all change types via `fab score --check-gate --stage intake`), with autonomous rework on review failure (`{max_cycles}`-cycle cap ‚Äî code-review.md ¬ß Rework Budget `Max cycles:`, default 3; exhaustion leaves `review: failed` as the terminal state ‚Äî see "Review failure backward movement" above). `/fab-fff` extends the same pipeline further through ship and review-pr ‚Äî same single intake gate, same rework behavior, wider scope. Since szxd both skills are thin wrappers over the shared `_pipeline.md` bracket, parameterized by driver name and terminal stage. The former second (spec) gate was removed in j6cs along with the spec stage. Both accept `--force` to bypass the gate. After either pipeline completes, run `/fab-archive` to move the change to archive.

**Non-terminal review-pr outcome (Copilot timeout)**: One review-pr outcome is deliberately non-terminal: when `/git-pr-review` requests a Copilot review and it does not arrive within 10 minutes, the review-pr stage is **left `active`** ‚Äî no finish, no fail, NOT a clean finish ‚Äî so a later `/git-pr-review` re-run can process the pending review (`start` cannot reactivate a `done` stage). `/fab-fff` surfaces this as `Review-PR pending (Copilot review requested, timed out waiting) ‚Äî re-run /git-pr-review when ready` instead of `Pipeline complete.`

### Git Integration (Optional)

Fab works without git. Change folders are the unit of identity, not branches ‚Äî the same change can be worked on across multiple branches, worktrees, or even repos. Fab's state **bookkeeping** never couples to git state ‚Äî no `branch:` field is stored in `.status.yaml`. The deliberate exception is the **ship path** (w7dp): `/git-pr` and `/git-pr-review` enforce branch‚Üîchange correspondence with a hard guard that STOPs before any status mutation, commit, or push when the current branch doesn't match the resolved change (see below) ‚Äî shipping a change requires its branch to be checked out, and `/git-branch` aligns it. (`docs/specs/architecture.md` ¬ß Git Integration states the same state-bookkeeping-vs-ship-path split.)

**Why decoupled (during development)**: A developer might work on the same change across multiple worktrees, a change might span multiple branches, or a change might move between branches after a rebase. Storing a static `branch:` field would go stale; instead, `/fab-status` uses `git branch --show-current` for live display.

**Branch management** is handled by the standalone `/git-branch` command ‚Äî separate from change activation. `/fab-switch` only updates `.fab-status.yaml` and never executes git commands.

| Condition | Action | Report |
|-----------|--------|--------|
| **Already on target** | No git operation | `(already active)` |
| **Target branch exists locally** | `git checkout` to it | `(checked out)` |
| **Target exists only on the remote** | `git checkout --track "origin/{branch}"` (never recreate a divergent local with `checkout -b`) | `(checked out, tracking origin/{branch})` |
| **On `main`/`master`** | `git checkout -b` (auto-create) | `(created)` |
| **On other branch, no upstream, resolves to no change or to the SAME change being branched** | `git branch -m` (rename) | `(renamed from {old})` |
| **On other branch, no upstream, matches a different change** | `git checkout -b` (create new ‚Äî never rename another change's branch away) | `(created, leaving {old} intact)` |
| **On other branch, has upstream** | `git checkout -b` (create new) | `(created, leaving {old} intact)` |

The local-only rename is **guarded**: before renaming, the skill checks `fab change resolve "$(git branch --show-current)"`. If the current branch resolves to a *different* change's folder (e.g., sitting on change B's unpushed branch after `/fab-switch` to change A), that branch is left intact and a new branch is created instead ‚Äî known caveat: the `checkout -b` fallback inherits the old change's HEAD, so unpushed commits carry over onto the new branch. Disposable branches that resolve to no change (e.g., `wt create` random names) take the rename path, and so does a branch resolving to the **same** change being branched (g8st ‚Äî e.g., a worktree placeholder named with the change's own 4-char ID; previously this fell through to `checkout -b` and left a stray placeholder branch). `/fab-new` Step 11's twin table carries the same rows and guard ‚Äî the two twins are kept in sync via in-file comments.

**Dirty-tree carried-over note (g8st)**: both twins read `git status --porcelain` before the branch operation; when the tree is dirty and the matched case creates or renames a branch (`git checkout -b` / `git branch -m`), the uncommitted work rides onto the new branch and the report line gets a non-blocking trailing note: ` ‚Äî note: {dirty_count} uncommitted change(s) carried over from {old_branch}`. Warn, never stash-prompt ‚Äî these paths run inside no-questions/orchestrated flows (ff/fff/operator) where a blocking prompt would violate the autonomy posture. The twins' **single deliberate divergence** lives outside the shared cases: fab-new derives `{dirty_count}` excluding `fab/changes/{name}/` (the change's own just-created artifacts always exist uncommitted by Step 11), while git-branch counts the full porcelain output.

**Default-branch resolution convention (g8st)**: autonomous git paths never assume the default branch is literally `main`/`master`. The shared chain: `git symbolic-ref --short refs/remotes/origin/HEAD` (strip `origin/`) ‚Üí `gh repo view --json defaultBranchRef -q .defaultBranchRef.name` ‚Üí literal fallback. `/git-pr` uses the resolved `{default_branch}` in its Step 2 guard, with literal `main`/`master` as the documented last fallback (the former Step 1b nudge that also consumed it was removed in w7dp ‚Äî superseded by the branch-matches-change hard guard; see [execution-skills.md](execution-skills.md)); `/fab-operator` resolves it **after `git fetch origin`** as the base for dependency cherry-picks and `--merge-on-complete` rebases, probing the just-fetched `origin/main`/`master` refs as its literal fallback (see [runtime/operator.md](../runtime/operator.md)). The chain is inlined per consuming skill ‚Äî deliberately not a shared `_` helper (two consumers, three lines; a helper would add loading cost for every other skill).

`/fab-switch` suggests `/git-branch` after activation. `/fab-status` displays the current branch via `git branch --show-current`. `/git-pr` and `/git-pr-review` **hard-STOP** when the current branch doesn't match the resolved change (w7dp ‚Äî exact folder-name equality or the folder name as a branch substring, the same two-form match the removed non-blocking Step 1b nudge used; the STOP precedes any status mutation, commit, or push, with `/git-branch`//`/fab-switch`/pass-the-intended-change recovery guidance and no autonomous checkout; a detached HEAD routes to each skill's detached-HEAD STOP instead). Both skills also accept an optional explicit `<change>` argument, resolved transiently ‚Äî see [execution-skills.md](execution-skills.md). Fab never commits, pushes, merges, or deletes branches ‚Äî that remains the user's responsibility (or `/git-pr`'s for shipping).

### Restoring an Archived Change

To bring an archived change back to active:
1. Run `/fab-archive restore <change-name>` ‚Äî moves `fab/changes/archive/yyyy/mm/{name}/` (or flat `archive/{name}/` for pre-migration entries) back to `fab/changes/{name}/`
2. The archive index entry is removed from `fab/changes/archive/index.md`
3. All artifacts (`.status.yaml`, `intake.md`, `plan.md`, etc.) are preserved without modification
4. Optionally use `--switch` to activate the restored change (creates `.fab-status.yaml` symlink)
5. Without `--switch`, run `/fab-switch {name}` to activate manually

The restore operation is idempotent ‚Äî if the folder already exists in `fab/changes/`, the move is skipped and remaining steps (index cleanup, optional pointer update) complete normally.

### Renaming a Change

To rename a change's slug (e.g., if the scope evolved or a typo was made):
```
changeman.sh rename --folder <current-folder> --slug <new-slug>
```

This atomically: renames the folder, updates `.status.yaml` `name` field, updates `.fab-status.yaml` symlink if it points to the renamed change, and logs the operation. The `{YYMMDD}-{XXXX}` prefix is preserved ‚Äî only the slug portion changes. Git branch rename is not included (manual if needed).

### Abandoning a Change

To discard a change that won't be completed:
1. Delete the change folder: `rm -rf fab/changes/{name}/`
2. Clear the pointer (if active): `rm .fab-status.yaml`
3. Optionally delete the git branch: `git branch -d {branch}`

There is no `/fab-abandon` skill ‚Äî this is a manual operation. To preserve context about why a change was dropped, move the folder to `archive/` instead.

### Listing Changes

`fab change list [--archive]` enumerates changes with their stage, state, and confidence, outputting one structured line per change to stdout in the format `name:display_stage:display_state:score`. The trailing `:indicative` field was **dropped in j6cs** (the indicative flag was retired); `fab-switch.md`'s parser and `change_test.go` were updated in lockstep. By default it scans `fab/changes/` (excluding `archive/`). With `--archive`, it scans `fab/changes/archive/` instead.

Stage and state are derived via `DisplayStage` for each change's `.status.yaml`. Confidence score is read from the `confidence` block. If a change's `.status.yaml` fails to load, the output is `name:unknown:unknown` with a stderr warning that echoes the **actual load error** (`Warning: failed to load .status.yaml for <name>: <err>`) ‚Äî corruption such as git merge-conflict markers (`.status.yaml` is git-tracked) is distinguishable from absence (mz4q F06; previously every failure printed "not found"). If no changes are found, the output is empty (exit 0). If `fab/changes/` does not exist, exit 1 with error to stderr. Missing archive directory with `--archive` is treated as empty (exit 0).

Used by `/fab-switch` (no-argument flow) and `/fab-status` to enumerate changes without manual directory scanning.

### `/fab-status`

`/fab-status` shows the current change state at a glance: name, live git branch, display stage with state qualifier, next action, progress through all stages, checklist status, confidence score, and version drift warning.

The skill uses `lib/preflight.sh` for data retrieval (including `display_stage` and `display_state` fields), then formats the output. The "Stage:" line shows the display stage (where you are) with a state qualifier (e.g., `Stage: intake (1/6) ‚Äî done`). The "Next:" line shows the routing stage with the default command (e.g., `Next: apply (via /fab-continue)`). It uses `git branch --show-current` for live branch display. The progress table's legend covers all six progress symbols ‚Äî `‚úì` done, `‚óè` active, `‚ó∑` ready, `‚óã` pending, `‚úó` failed, `‚è≠` skipped ‚Äî the `‚è≠` glyph matching the Go renderer's `ProgressLine` (the legend lacked a skipped glyph until w7dp, so skipped stages rendered outside the documented set).

**Impact line over-threshold channel**: When `.status.yaml` carries a `true_impact` block, an Impact line renders under the change summary (omitted entirely when the block is absent ‚Äî no placeholder). The over-threshold highlight is a **warning-emoji (‚öÝÔ∏è) prefix plus bold**, applied when `true_impact.net > 100` OR `true_impact.excluding.net > 50` (when `excluding` is present) ‚Äî both thresholds hard-coded, not project-configurable ‚Äî mirroring fab-operator's health-emoji convention. ANSI SGR escapes (e.g., `\e[33m`) MUST NOT be used: they do not survive the assistant-message render path (verified empirically in fab-operator ¬ß4); emoji + bold are the surviving channels.

### `/fab-switch [change-name] [--none]`

`/fab-switch` changes the active change pointer. It does not perform git operations ‚Äî branch management is handled by the separate `/git-branch` command.

**Switching to a change:**
1. Match `change-name` against `fab/changes/` (supports partial/slug match)
2. **Ambiguous match** ‚Äî if multiple changes match, list them and ask the user to pick. Never guess
3. **No match** ‚Äî list available changes and ask
4. Create `.fab-status.yaml` symlink via `fab change switch` (symlink points to `fab/changes/{name}/.status.yaml`)
5. Display the switched change's status summary (three-line format: `Stage: {display_stage} ({N}/6) ‚Äî {state}` + `Confidence: {score} of 5.0` + `Next: {routing_stage} (via {default_command})`). The `{state}` qualifier spans all six states the `display_state` derivation can emit ‚Äî `active`, `failed`, `ready`, `done`, `skipped`, `pending` (enumeration completed in w7dp; `ready` is the standard state of a freshly switched draft, `failed` surfaces a parked review/review-pr failure, `skipped` a skipped trailing stage)
6. Append a hint: `Tip: run /git-branch to create or switch to the matching branch`

**Next-line derivation (k4ge)**: switch's `Next:` line mirrors `/fab-status` ‚Äî `{routing_stage}` is the routing stage from the three-tier derivation above, and `{default_command}` is the command that drives *that* stage (`intake`/`apply`/`review`/`hydrate` ‚Üí `/fab-continue`, `ship` ‚Üí `/git-pr`, `review-pr` ‚Üí `/git-pr-review`). Only when every stage is `done` or `skipped` does the line collapse to the bare post-pipeline suggestion `Next: /fab-archive`. This fixed an off-by-one where switch printed `NextStage(routing)` with a shifted command map (`hydrate ‚Üí /git-pr`, `ship ‚Üí /git-pr-review`, `review-pr ‚Üí /fab-archive`), so post-review states showed `/fab-archive` while review-pr work remained ‚Äî diverging from `fab status` for the same state.

**Deactivating (`--none`):**
1. Remove `.fab-status.yaml` symlink ‚Äî deactivates the current change. Idempotent (no error if already deactivated)

### `/git-branch [change-name]`

`/git-branch` creates or checks out a git branch matching the active (or specified) change. When an explicit argument matches no change at all, falls back to creating a standalone branch with the literal name; an *ambiguous* argument STOPs instead (g8st). Standalone command ‚Äî does not modify fab state (`.fab-status.yaml`, `.status.yaml`).

**Behavior:**
1. Resolve change name (from argument or `.fab-status.yaml`). If resolution fails with an explicit argument, distinguish the two failure modes by `fab change resolve` stderr ‚Äî both exit 1 (g8st):
   - **Ambiguous multi-match** (stderr contains `Multiple changes match`): STOP with the candidate list ‚Äî no branch is created. (Previously the skill unconditionally entered standalone fallback, printing a false `No matching change found` and creating a junk branch.)
   - **True no-match** (stderr contains `No change matches`, or any other resolution failure): enter **standalone fallback** ‚Äî use the raw argument as a literal branch name, print `No matching change found ‚Äî using standalone branch '{name}'`.
   - If no argument was provided, display the resolver's error and stop.
2. Derive branch name: `{change-name}` for change-resolved branches; raw argument as-is for standalone
3. Context-dependent action:
   - **Already on target** ‚Üí no-op ("already active")
   - **Target branch exists locally** ‚Üí switch to it (`git checkout`)
   - **Target exists only on the remote** (`git rev-parse --verify "origin/{branch}"` succeeds, local verify fails) ‚Üí `git checkout --track "origin/{branch}"` ‚Äî never recreate a divergent local (g8st)
   - **On `main`/`master`** ‚Üí auto-create branch
   - **On other branch, no upstream** ‚Üí rename guard: if the current branch resolves to no change (`fab change resolve` fails) or to the **same** change being branched (g8st), rename it (`git branch -m`); if it matches a *different* change's folder, create a new branch instead (`git checkout -b`), leaving it intact
   - **On other branch, has upstream** ‚Üí create new branch, leaving current intact
4. Report result ‚Äî on create/rename actions with a dirty tree, the report line carries the non-blocking carried-over note (see ¬ß Git Integration above)

## Migration Note (Old-Format `.status.yaml`)

Existing `.status.yaml` files created before v5p2 may contain a `stage:` field. To migrate:

1. **Remove the `stage:` line** ‚Äî current stage is now derived from the `active` entry in the progress map
2. **Ensure `intake:` is present in the progress map** ‚Äî intake is a formal pipeline stage. If missing, add `intake: done` (the preflight script also infers this automatically)
3. **Set the appropriate stage to `active`** ‚Äî e.g., if the old `stage:` was `spec`, set `spec: active` in the progress map

Skills will tolerate old-format files ‚Äî the preflight script infers `intake: done` when the entry is missing.

### qszh Migration (1.8.0 ‚Üí 1.9.0)

`.status.yaml` files created before qszh contain a `progress.tasks` key and a `checklist:` block. The `1.8.0-to-1.9.0.md` migration handles this automatically (idempotent on `plan.md` presence): drops `progress.tasks` (preserving `progress.apply` state, advancing if it was `tasks: active|ready`), replaces `checklist:` with `plan:` (`generated: true`, `task_count = count(- [ ]/- [x]) under ## Tasks`, `acceptance_count = old checklist.total`, `acceptance_completed = old checklist.completed`), and concatenates legacy `tasks.md` + `checklist.md` into a single `plan.md` with `## Tasks` and `## Acceptance` headings (acceptance IDs preserved verbatim ‚Äî `CHK-NNN` survives the merge for in-flight changes; only newly-generated plans use `A-NNN`). The legacy files are left on disk with a `<!-- Migrated to plan.md on {DATE} ‚Äî safe to delete. -->` note. `Load()` is also tolerant of legacy files when reading directly: it upgrades a `checklist:` block to a `plan:` raw mapping with field migration (`completed ‚Üí acceptance_completed`, `total ‚Üí acceptance_count`) and drops `checklist:` when both blocks coexist. Archived changes (`fab/changes/archive/**`) are untouched.

## Design Decisions

### Symlink Replaces Pointer File
**Decision**: `.fab-status.yaml` is a symlink at the repo root pointing to the active change's `.status.yaml`, replacing the former `fab/current` two-line text file.
**Why**: The conductor orchestrator needs to observe N worktrees efficiently. The old pointer file required three reads per worktree (`fab/current` ‚Üí parse folder name ‚Üí `.status.yaml` ‚Üí `.fab-runtime.yaml`). The symlink collapses two reads into one: `readlink .fab-status.yaml` gives change identity (from target path) and `cat .fab-status.yaml` gives full status. Grouping `.fab-status.yaml` alongside `.fab-runtime.yaml` at repo root consolidates all ephemeral per-worktree state into two sibling files scannable with a single glob. Windows compatibility is not a concern (explicitly confirmed).
**Rejected**: Keeping `fab/current` as a text file ‚Äî adds unnecessary indirection for every cross-worktree observation cycle. The original reasons for a text file (Windows, cleaner errors) are no longer relevant: Windows is not a target, and broken symlinks are handled gracefully (`os.Lstat` succeeds, `os.Stat` fails ‚Üí treat as no active change).
*Introduced by*: 260307-x2tx-status-symlink-pointer; *Supersedes*: "Pointer File, Not Symlink" (doc/fab-spec/ARCHITECTURE.md)

### Name Stable Across Lifecycle
**Decision**: Change folders keep the same `{YYMMDD}-{XXXX}` prefix from creation through archive. No automatic rename on archive. The slug portion can be explicitly renamed via `changeman.sh rename`, which atomically updates the folder, `.status.yaml`, and `.fab-status.yaml`.
**Why**: The immutable prefix simplifies references ‚Äî specs, changelogs, and design decisions can reference the change by date-ID without tracking renames. Explicit slug rename is supported for scope evolution and typo correction, but the date-ID prefix ensures continuity.
**Rejected**: Renaming with archive prefix ‚Äî adds complexity, breaks back-references. Auto-rename ‚Äî would silently break references.
*Source*: doc/fab-spec/ARCHITECTURE.md; *Updated by*: 260216-u6d5-DEV-1039-add-changeman-rename

### Fixed State Vocabulary
**Decision**: All `.status.yaml` progress fields draw from a fixed set of states (`pending`, `active`, `ready`, `done`, `failed`).
**Why**: Prevents ad-hoc state names from creeping in across skills. Makes status parsing predictable. The `ready` state disambiguates "work in progress" (`active`) from "artifact exists, awaiting advancement" (`ready`).
**Rejected**: Free-form state strings ‚Äî inconsistent across skills, harder to parse programmatically. The previously unused `skipped` state was removed as dead code.
*Source*: doc/fab-spec/TEMPLATES.md; *Updated by*: 260226-i9av-add-ready-state-to-stages (added `ready`, removed `skipped`)

### Branch Management in `/git-branch`, Not `/fab-switch`
**Decision**: Git branch operations are handled by the standalone `/git-branch` command, not `/fab-switch`. `/fab-switch` only updates `.fab-status.yaml` ‚Äî it performs zero git operations. `/fab-new` never activates changes. The `branch:` field was removed from `.status.yaml`; `/fab-status` uses `git branch --show-current` for live display.
**Why**: Change activation (updating `.fab-status.yaml`) and git branch management are independent concerns. Coupling them in `/fab-switch` required `--no-branch-change` as an escape hatch (used by `dispatch.sh` for worktree-based pipelines), added interactive friction for users who just wanted to switch the active pointer, and forced every feature touching change activation to consider git side effects. Separating them lets each command do one thing well.
**Rejected**: Keeping branch in `/fab-switch` ‚Äî couples two concerns, requires negative flags. Shell script instead of command ‚Äî loses conversational invocability in Claude sessions. Silent ignore of old flags ‚Äî defers cleanup, adds dead-flag debt.
*Introduced by*: 260208-q8v3-branch-to-switch; *Updated*: 260216-7ltw-DEV-1038 (removed `--switch` flag); *Updated*: 260224-vx4k-decouple-git-from-fab-switch (extracted to standalone `/git-branch`)

### Single Source of Truth: Progress Map with `active`/`ready` Markers
**Decision**: Remove the `stage:` field from `.status.yaml`. The current stage is determined by a three-tier fallback: (1) first `active` or `ready` entry, (2) first `pending` after last `done`, (3) the post-pipeline fallback if all done (`review-pr` in the current 6-stage code ‚Äî see "Deriving current stage" above; this decision originally named `hydrate`, then the terminal stage). State vocabulary: `pending`, `active`, `ready`, `done`, `failed`. The `active`/`ready` distinction disambiguates "work in progress" from "artifact exists, awaiting advancement."
**Why**: The old model had two sources of truth ‚Äî `stage: spec` AND `progress.spec: pending` ‚Äî forcing skills to cross-reference both. The `active` marker serves as an explicit "you are here" pointer; `ready` signals the artifact exists and the stage is eligible for advancement or clarification. The pending-after-done fallback handles the post-reset state where no stage is `active` or `ready`.
**Rejected**: Keeping `stage:` with simplified progress (still two sources of truth). Adding a `next_stage:` field (second source of truth, stale risk). ~~Adding a `ready` state~~ ‚Äî originally rejected as "expands vocabulary for narrow edge case," but revisited after a concrete dispatch bug where `spec: active` was ambiguous (spec not yet generated vs. spec exists and awaits advancement). The `ready` state was adopted.
*Introduced by*: 260212-v5p2-simplify-stages-entry-paths; *Updated by*: 260213-wo9v-fix-reset-auto-advance (three-tier fallback); *Updated by*: 260226-i9av-add-ready-state-to-stages (added `ready` state, reversed prior rejection)

### Reset Flow Stops at Target Stage
**Decision**: When `/fab-continue` resets to a planning stage, the target stage is advanced to `ready` after regeneration. The next stage is NOT set to `active` ‚Äî it remains `pending`. The user can run `/fab-clarify` to refine or `/fab-continue` to proceed.
**Why**: Auto-advancing past the regenerated stage leaves the next stage `active` with a stale or invalidated upstream input. Example: `fab-continue intake` regenerates intake.md, but if it also sets `spec: active`, the user is stranded at spec with a spec generated against the prior intake. The `ready` state signals the artifact exists and is open for refinement.
**Rejected**: Leaving target as `active` (confusing ‚Äî artifact is fresh but stage says "in progress"). Marking as `done` (skips the `/fab-clarify` checkpoint). Auto-advancing to next stage (the bug this decision fixes).
*Introduced by*: 260213-wo9v-fix-reset-auto-advance; *Updated by*: 260227-ijql-streamline-planning-dispatch (changed from `done` to `ready`); 260423-qszh-merge-tasks-checklist (example updated for 7-stage pipeline ‚Äî `spec ‚Üí tasks` example replaced with `intake ‚Üí spec`)

### Keyword Scan Interactive-Only (`/fab-archive`)
**Decision**: The keyword-based backlog scan in `/fab-archive` always runs interactively with user confirmation. `/fab-archive` is always user-invoked (never called by auto-mode pipelines), so the interactive confirmation is natural.
**Why**: Keyword matches are heuristic ‚Äî false positives are possible. Requiring user confirmation for keyword-matched candidates prevents accidental backlog closures. The exact-ID check is deterministic and runs automatically.
**Rejected**: Running keyword scan in auto mode with auto-close ‚Äî risks false-positive backlog closures.
*Introduced by*: 260213-wloh-archive-backlog-scan; *Updated*: 260213-jc0u-split-archive-hydrate (backlog marking moved from pipeline archive stage to standalone `/fab-archive`)

### No Dedicated Abandon Skill
**Decision**: Abandoning a change is a manual operation (delete folder, clear pointer).
**Why**: Abandonment is rare and destructive enough to warrant deliberate manual action. A skill would make it too easy to accidentally discard work.
**Rejected**: `/fab-abandon` skill ‚Äî low-value automation for a rare, deliberate action.
*Source*: doc/fab-spec/ARCHITECTURE.md

### Centralized Pointer Access
**Decision**: `.fab-status.yaml` (a symlink at repo root) encodes the active change pointer. `resolve.go` reads it (via `readlink`, validating the target `.status.yaml` exists); `archive.go` reads the link directly to capture the active change before the folder rename; only `change.go` writes it (via the atomic `setActivePointer` temp-symlink + `os.Rename` helper). Skills and other scripts delegate through `fab resolve` and `fab change switch`.
**Why**: Encapsulates the pointer mechanism as an implementation detail. The symlink encodes both change identity (from the target path) and status data (the target file itself) in one filesystem entry. The `id` field in `.status.yaml` provides the 4-char change ID without path parsing. Eliminates format knowledge leakage ‚Äî no script or skill manipulates the symlink directly.
**Rejected**: YAML pointer file (`yq` dependency on hot path). Keeping the old two-line `fab/current` text file (unnecessary indirection for cross-worktree observation).
*Introduced by*: 260302-a8ay-centralize-current-pointer; *Updated by*: 260307-x2tx-status-symlink-pointer (text file ‚Üí symlink); *Updated by*: 260612-mz4q-shared-state-concurrency-hook-hot-path (atomic swap, target validation, archive's direct pre-rename read)

### Atomic Pointer Swap, Stale Pointer Never Removed by Resolve
**Decision**: `setActivePointer` creates the symlink at a unique temp name and `os.Rename`s it over `.fab-status.yaml` ‚Äî POSIX rename atomically replaces the old link. `resolveFromCurrent` stats the pointer target and, on a dangling pointer, falls through to the no-active-change/single-change-fallback logic **without removing the link**.
**Why**: The prior `os.Remove` + `os.Symlink` left an empty-pointer window (a concurrent reader saw no active change) and a concurrent-Switch EEXIST race; `Rename`'s discarded symlink error could permanently leave no pointer. Temp+rename matches the project's established atomic-write convention (`.status.yaml`, `.fab-runtime.yaml`). Fall-through (rather than trusting the target-less pointer) replaces silent wrong answers (`fab resolve` printing a stale folder) with the actionable `/fab-switch` guidance; not removing the link preserves `fab resolve`'s documented pure-query/no-side-effects contract.
**Rejected**: Removing the stale link during resolve (side effect in a documented pure query). Validating only at preflight (leaves `fab resolve`'s silent wrong answer in place for agent callers).
*Source*: 260612-mz4q-shared-state-concurrency-hook-hot-path

### Bounded Sibling Flock for `.status.yaml` Writes
**Decision**: Mutating `.status.yaml` cycles serialize under the shared `internal/lockfile` flock helper (sibling `.lock` file, `LOCK_EX|LOCK_NB` retry, ~10s timeout) ‚Äî the same helper guarding `.fab-runtime.yaml`. Low-frequency writers outside the F01/F03 surfaces (`change.Rename`'s name update, prmeta, operator state) are deliberately not lock-wrapped.
**Why**: Temp+rename protects readers from torn files but not writers from lost updates; the realistic exposure is parallel review sub-agent dispatch (both editing `plan.md`, firing the bookkeeping hook) and cross-pane `fab status` writers. Bounded acquisition prevents a stage hook re-invoking `fab status` on the same change from deadlocking forever. Scope follows exposure ‚Äî locking every writer would add surface without a motivating race.
**Rejected**: Unbounded blocking flock (silent deadlock in unattended pipelines). Per-call-site fixes (the lost-update class would recur).
*Source*: 260612-mz4q-shared-state-concurrency-hook-hot-path

### Write-Time Key Insertion, Not Validate-at-Load Refusal
**Decision**: `syncToRaw` creates absent command-mutated keys at write time; `SetProgress` creates missing stage keys; only a malformed document shape (`progress:` absent or not a mapping) errors ‚Äî and that error propagates through the transition functions so callers cannot exit 0 on a dropped write.
**Why**: Sparse files are legitimate (restored pre-0.24.0 archives, hand-edits; migrations never touch `fab/changes/archive/**`), and skills rely on exit-0 semantics (`git-pr-review` commits bookkeeping assuming `finish` persisted) ‚Äî silent drops propagate into autonomous pipelines, violating constitution Principle III. Insertion matches the established posture (`TestLegacyChecklistFileSavesPlanBlock` fixed this exact bug class for `plan:`), while Load-refusal would conflict with the documented tolerance for un-migrated/archived files (1.9.7-to-1.10.0 migration notes).
**Rejected**: Validate-at-Load schema refusal (breaks legacy tolerance). Warn-but-continue (still reports success on a dropped write).
*Source*: 260612-mz4q-shared-state-concurrency-hook-hot-path
