# Configuration

**Domain**: _shared

## Overview

Fab uses a set of complementary configuration files ‚Äî the **5 Cs of Quality**: `fab/project/config.yaml` for project settings, `fab/project/constitution.md` for principles and constraints, `fab/project/context.md` for free-form project context, `fab/project/code-quality.md` for coding standards, and `fab/project/code-review.md` for review policy. Config says *what you use*; constitution says *how you use it*; context describes *what you're working with*; code-quality defines *how code should look when writing*; code-review defines *what to look for when validating*. All are generated by `/fab-setup` and loaded as context by all skills that generate or validate artifacts.

## Requirements

### Schema Discovery ‚Äî `fab config reference`

`fab config reference` (6nke) is the **canonical schema-discovery surface** ‚Äî the single place a user or agent consults to learn what may go in `config.yaml`. It is a pure query (`config` command group + `reference` subcommand, `cmd/fab/config.go`; same no-side-effect family as `fab resolve` / `fab resolve-agent`) that prints a fully-commented reference `config.yaml` to stdout, exit 0 on success (a usage error from `cobra.NoArgs` exits non-zero), no flags, no file writes, byte-stable for a given binary version. It reads no project config ‚Äî the output is a *reference*, not the invoker's settings. The scaffold's header pointer line (`# Full reference of all available options: fab config reference`) and the `2.9.2-to-2.10.0` migration (which backfills that line into existing configs ‚Äî see [migrations.md](/distribution/migrations.md)) both point here; `docs/specs/architecture.md` and `README.md` point at the command rather than embedding schema copies.

**Covers both key sets ‚Äî binary-consumed AND skill-consumed.** The Go `Config` struct (`internal/config`) models only *binary-consumed* keys (`test_paths`, `true_impact_exclude`, the top-level `providers:` map ‚Äî each `ProviderConfig` carrying `session_command`/`dispatch_command` since tykw, `agent.tiers` ‚Äî five role tiers as `{provider, model, effort}` since tykw, `stage_hooks`, `branch_prefix`, `fab_version`, `project.linear_workspace`); several keys are *skill-consumed* ‚Äî read only by markdown skills and therefore invisible to Go reflection (`project.name`/`description`, `source_paths`, `checklist.extra_categories`). The reference documents both. Two coverage tests guard this split (`cmd/fab/config_test.go`): recursive yaml-tag reflection over `Config` (a new binary key forces a reference update) plus a scaffold key-superset check (guards the skill-consumed keys reflection cannot see), alongside a `config.LoadPath` round-trip and a byte-stability assertion. A retired-keys guard (`TestConfigReferenceRetiresLegacyKeys`) additionally asserts the reference no longer renders `review_tools` or `agent.spawn_command` (both retired in tykw). The tykw rewrite dropped `TierProfile.SpawnCommand` (the per-tier dispatch command moved to `providers.<name>.dispatch_command`) and added `TierProfile.Provider` plus the `ProviderConfig` struct; the reference documents the `providers:` block and the five-tier `agent.tiers` with explicit providers.

**Layout**: baseline keys every project sets appear live with example values; the `agent.tiers` block also ships **live** with all five role tiers populated from fab-kit's built-in defaults (so `GetAgentTier` resolves out of the box ‚Äî the Go tests assert `GetAgentTier("doing")` succeeds against the reference); the remaining opt-in override blocks (`stage_hooks`, `branch_prefix`) appear commented-out with fab-kit's built-in defaults shown, mirroring the `2.2.0-to-2.3.0` migration block style (uncommenting = opting in). `fab_version` is documented as machine-managed. The `providers:` block is mixed-presentation: it ships as a **three-provider (claude/codex/gemini) starter template** with only `claude.session_command` live and everything else (claude's `dispatch_command`, the whole codex/gemini blocks) commented ‚Äî see ¬ß `providers` ‚Üí "Three-provider starter template" for the shipped strings and guarantees.

**Generated from constants ‚Äî no second copy to drift (6nke).** The reference body lives in Go as a `text/template` in `internal/configref` (`Render()` entrypoint; thin cobra wrapper in `cmd/fab/config.go`, mirroring the `internal/prmeta` render/command split). Every default/example value that has a canonical Go constant is *injected* from that constant ‚Äî `spawn.DefaultSpawnCommand`, the per-tier default profiles via `agent.DefaultTier` over `agent.TierNames`, and the pipeline stage names via `agent.StageNames`. This is strictly stronger than a drift-guard test on a hand-written copy: there is no second copy of the defaults, so they *cannot* drift, and no drift-guard test is needed for them. This supersedes the drift-prone hand-written `agent.tiers` comment block shipped by the `2.2.0-to-2.3.0` migration (whose defaults *were* hand-written prose and whose idempotency sentinel prevented ever refreshing them) ‚Äî though per that change's non-goals the old block stays as shipped history and is not retro-edited. (`fab config validate` ‚Äî unknown-key/typo linting ‚Äî is a deliberately-deferred non-goal; the `config` group name is claimed to leave room for it.)

### `config.yaml` Schema

`fab/project/config.yaml` SHALL contain the following sections:

#### `fab_version`
A top-level field (string) that declares the fab-kit version the repo expects. The system `fab` shim reads this field to resolve which cached `fab-go` binary to dispatch to. Format: bare semver string (e.g., `"0.43.0"`).

When present, the shim resolves and dispatches to the matching cached binary. When absent but `config.yaml` exists, the shim exits with an actionable error: `"No fab_version in config.yaml. Run 'fab init' to set one."`.

Set by `fab init` (on bootstrap) and `fab upgrade-repo` (on version update). Not user-editable under normal circumstances ‚Äî managed by the shim subcommands.

#### `project`
- `name` ‚Äî Project name (string)
- `description` ‚Äî Project description (string)
- `linear_workspace` ‚Äî Linear workspace slug (string, optional). When present, `/git-pr` constructs issue hyperlinks using `https://linear.app/{linear_workspace}/issue/{ISSUE_ID}`. When absent, issue IDs are rendered as bare text. Set once per project. Used exclusively by `/git-pr` for URL construction in the PR body's Change section.

#### `model_tiers`
Provider-specific model identifiers for the two-tier system. Replaces the former `src/kit/model-tiers.yaml` file. Structure:
```yaml
model_tiers:
  fast:
    claude: haiku   # or sonnet, etc.
```
When absent, `fab-sync.sh` falls back to `haiku` for the fast tier. (Note: the model-tier system was later eliminated ‚Äî all skills now run on the session's default model and no `model_tiers:` section is read. This subsection documents historical behavior; the former `model-tiers.md` memory file was removed in the 260608 domain restructure. As of c5tr the always-load descriptor no longer advertises "model tiers", and migration `2.1.6-to-2.2.0.md` defensively drops any re-appeared `model_tiers:` key.)

#### `checklist`
- `extra_categories` ‚Äî Project-specific quality categories added to the default checklist categories (functional_completeness, behavioral_correctness, scenario_coverage, edge_cases, code_quality, security)

#### `providers`

Top-level map (tykw) holding the **invocation grammar** for each agent, keyed by **opaque, user-chosen provider names** (`claude`, `codex`, ‚Ä¶). Extracted out of `agent:` so a tier is pure budget/role policy (`{provider, model, effort}`) while *how* to invoke that provider lives here, in a named table shared across all tiers that reference it. Modeled on `Config.Providers map[string]ProviderConfig` (`yaml:"providers"`); each `ProviderConfig` is `{SessionCommand, DispatchCommand string}` (`yaml:"session_command"`/`"dispatch_command"`). Read via nil-safe `GetProvider(name) (ProviderConfig, bool)` and resolved by `agent.ResolveProvider(name)` (project `providers:` per-field-merged over fab-kit's built-in `defaultProviders`).

Each provider MAY carry two command fields:

- **`session_command`** ‚Äî opens an interactive agent **session** (the relocated `agent.spawn_command` semantics). Consumed by `fab operator`, `fab batch new`/`batch switch`, and the new `fab agent`. Default (built-in `claude` provider) ‚Äî a `{model}`/`{effort}` template (`260703-gvxd`): `'claude --dangerously-skip-permissions -n "$(basename "$(pwd)")" --model {model} --effort {effort}'`, with the placeholders substituted from the resolved tier profile by `spawn.WithProfile`'s template mode (placed last so the resolved command is byte-identical to the former plain form + append mode). May contain shell expansions (`$(basename "$(pwd)")`) evaluated at invocation time (stored as a single-quoted YAML string).
- **`dispatch_command`** ‚Äî runs one headless **stage task** via `fab dispatch` (the relocated per-tier `spawn_command` semantics). **ABSENT `dispatch_command` = native Agent-tool dispatch** (the existing absence signal, unchanged ‚Äî in practice native dispatch only works for claude-family models, the Agent tool's model enum, but fab does not validate this).

```yaml
providers:
  claude:
    session_command: 'claude --dangerously-skip-permissions -n "$(basename "$(pwd)")" --model {model} --effort {effort}'
    # no dispatch_command ‚Üí claude's stages dispatch natively via the Agent tool
  codex:
    session_command: 'codex -m {model} -c model_reasoning_effort={effort}'
    dispatch_command: 'codex exec -m {model} -c model_reasoning_effort={effort}'
```

**The two fields are NOT merged into one `command`** ‚Äî session and dispatch are different invocations of the same binary (claude interactive `-n <name>` vs headless `-p`; codex TUI vs `codex exec`), and no single template expresses both.

**The `claude` provider is the built-in default** (not "nothing needed"): fab-kit's `defaultProviders` table contains `claude` with the default `session_command` and no `dispatch_command` (native). A project overrides or extends via its own `providers:` block, per-field-merged over the built-in.

**Provider names are opaque ‚Äî fab NEVER infers a provider from a model string** (`claude-*` ‚Üí claude would need a provider registry, which the no-validation/provider-neutrality contract refuses). The one documented footgun: **override a tier's `model` cross-provider ‚áí override its `provider` too** ‚Äî fab documents this, it does not validate it.

**No fallback between the two command fields** ‚Äî absence of `dispatch_command` signals native dispatch, never "use `session_command`" (the 3a‚Äì3d no-cross-fallback decision, carried over).

**`{model}`/`{effort}` placeholder substitution (template mode ‚Äî reuses 6tmi's `spawn.WithProfile`).** Both command fields MAY contain the literal placeholders `{model}` and/or `{effort}`, which relocate provider grammar into the config (consistent with the resolver's verbatim/no-validation philosophy). `spawn.WithProfile(cmd, model, effort)` ‚Äî the one place fab composes a command line ‚Äî operates in one of two modes selected by placeholder presence:
- **Template mode** (the command contains `{model}` or `{effort}`): substitute *every* occurrence of each placeholder with the resolved value. Template mode is **all-or-nothing** ‚Äî the presence of *any* placeholder disables the append below entirely, so a value whose placeholder is absent from the template is simply *not injected* (this prevents e.g. a Claude `--effort` flag being appended to a codex command that only templated `{model}`). Example: `'codex exec -m {model} -c model_reasoning_effort={effort}'`.
- **Append mode** (no placeholder): the Claude-shaped behavior, byte-for-byte ‚Äî append ` --model <id>` then ` --effort <level>` to the end (each omitted when its value is empty). This mode stays load-bearing for a user's plain-form `session_command` carried forward by the 2.13.0 migration. **The built-in `claude` `session_command` / `DefaultSessionCommand` fallback is itself a `{model}`/`{effort}` template as of `260703-gvxd`** ‚Äî it resolves via template mode, with the placeholders placed last so the resolved command is byte-identical to what this append mode produced for the former plain form (zero behavior change).

  **Empty-value token-drop rule (template mode).** An empty model/effort is the documented "inherit/omit" signal (see [context-loading.md](/_shared/context-loading.md) ¬ß Per-Stage Model Resolution). On substitution of an *empty* value the command is tokenized on whitespace and the token containing the placeholder is dropped, along with the immediately preceding token when it begins with `-`. This deterministically handles the four common flag shapes without a templating engine: `-m {model}` ‚Üí both tokens dropped; `--model {model}` ‚Üí both dropped; `--model={model}` ‚Üí the single token dropped (no preceding `-`-flag); `-c model_reasoning_effort={effort}` ‚Üí the `...={effort}` token *and* the preceding `-c` dropped. When *every* substituted value is non-empty, resolution is a plain `strings.ReplaceAll` on the raw string, so the author's whitespace is preserved exactly ‚Äî tokenization applies only on the empty-value drop path (6tmi rework: whitespace preservation). The token-drop grammar is quote-blind and limited to those four value-carrying flag shapes; a placeholder inside quotes or preceded by a valueless flag (e.g. `--verbose {model}`) is outside the supported grammar.

**Three-provider starter template in the reference and scaffold (ho9y).** Both `fab config reference` (`internal/configref`) and the new-project scaffold (`src/kit/scaffold/fab/project/config.yaml`) ship the `providers:` block as a **three-provider (claude/codex/gemini) starter template** ‚Äî every one of the six command strings appears *as text* so a user configuring a non-claude provider copies-and-adapts rather than composing CLI grammar from scratch. This is **template TEXT only** ‚Äî no new Go built-in providers (the `defaultProviders` table stays claude-only per ¬ß `agent` / ¬ß `providers` above), and no change to resolution or dispatch behavior; ho9y is a documentation/scaffold-surface change on top of tykw's schema.

Presentation follows the existing commented-opt-in convention ‚Äî **anything whose uncommenting changes default behavior ships commented**:

- **`claude.session_command`** is the only **live** line (it restates the built-in default `{{ .SessionCommand }}` ‚Äî a harmless restatement, consistent with how baseline keys show live example values).
- **`claude.dispatch_command`** ships commented (`claude -p --dangerously-skip-permissions --model {model} --effort {effort}`) ‚Äî uncommenting flips claude's stages from native Agent-tool dispatch to headless CLI dispatch.
- The **entire `codex` and `gemini` blocks** ship fully commented (opt-in providers): codex `codex -m {model} -c model_reasoning_effort={effort}` (session) / `codex exec -m {model} -c model_reasoning_effort={effort}` (dispatch); gemini `gemini -m {model}` (session) / `gemini -m {model}` (dispatch).

Two reviewer-verified facts on the gemini strings:

- **Gemini carries NO `{effort}` placeholder** ‚Äî the gemini CLI has no reasoning-effort flag, so the placeholder is simply omitted (the empty-effort token-drop rule above is not relied upon).
- **Gemini's `dispatch_command` is the bare `gemini -m {model}` ‚Äî NO `-p`.** `fab dispatch` pipes the stage prompt to the command's **stdin**, and gemini reads piped stdin as the prompt in non-TTY mode; gemini's `-p/--prompt` takes prompt TEXT as its argument (appended *after* stdin), so it MUST NOT be used for stdin-delivered dispatch. This corrects the intake's tentative `gemini -m {model} -p`.

**Presentation guarantees (parse-side + whole-block-uncomment).** The commented blocks parse as **absent from `Config`** ‚Äî round-trip guards assert `claude.DispatchCommand == ""` and `GetProvider("codex")`/`GetProvider("gemini")` return `!ok`, so the template documents without accidentally registering a provider. **Whole-block uncommenting yields valid YAML**: all per-provider explanatory prose is kept *out* of the commented blocks (gathered into a leading "Per-provider notes" comment above the `providers:` key, and one short trailing `# no {effort} flag; no -p (‚Ä¶)` on the gemini dispatch line), so stripping the leading `# ` from every line of a block leaves parseable YAML ‚Äî the same commented-block discipline as `stage_hooks`/`branch_prefix`. A config-test guard asserts the three-provider template is present (codex AND gemini documented; gemini carries no `{effort}`), alongside the existing round-trip / byte-stability / coverage / placeholder guards. The scaffold carries the identical template.

#### `agent`

Holds only the tier system now ‚Äî `agent.spawn_command` was **relocated to `providers.claude.session_command`** in tykw (see ¬ß `providers` and the migration `2.12.1-to-2.13.0`).

- `tiers` ‚Äî Optional map (l3ja) and the **sole per-stage-model override surface**. A **tier** is a named `{provider, model, effort}` agent profile (the per-tier command moved to `providers:` in tykw ‚Äî a tier no longer carries a command). Keys under `tiers:` are the **five role-tier names** (tykw): `default`, `operator`, `doing`, `review`, `fast` ‚Äî roles with concrete referents, replacing the former `thinking`/`doing`/`fast` cognitive modes. Each value is a `{provider, model, effort}` object (yaml keys `provider`, `model`, `effort`). Modeled on `AgentConfig.Tiers map[string]TierProfile` (`yaml:"tiers"`), and `TierProfile` is now `{Provider, Model, Effort string}` (`yaml:"provider"`/`"model"`/`"effort"` ‚Äî the tykw rewrite added `Provider`, removed `SpawnCommand`); yaml.v3 ignores unknown keys, so widening the struct is free for existing configs. Read via the nil-safe accessor `GetAgentTier(tier) (TierProfile, bool)` ‚Äî the bool distinguishes "no override" from "override present with empty fields", which the resolver needs for per-field merge.

  ```yaml
  agent:
    tiers:
      default:  { provider: claude, model: claude-fable-5,  effort: xhigh }
      operator: { provider: claude, model: claude-sonnet-5, effort: medium }
      doing:    { provider: claude, model: claude-opus-4-8, effort: xhigh }
      review:   { provider: claude, model: claude-fable-5,  effort: xhigh }
      fast:     { provider: claude, model: claude-sonnet-5, effort: low }
  ```

  **`thinking` was removed, not split (tykw).** With `review` split out into its own tier, `thinking`'s only remaining stage would be intake ‚Äî which never dispatches (it is pre-boundary, foreground). Intake rides `default`, honestly ‚Äî it runs wherever the interactive session runs.

  **What it overrides** ‚Äî only *what each tier means* (provider + model + effort), never *which stages belong to a tier*. The stage‚Üítier mapping is fab-owned and **NOT user-overridable**: there is no `stage_tiers` map and no per-stage escape hatch. The five tiers, their fixed stage groupings, and fab-kit's built-in default profiles are owned by the Go `internal/agent` package (mirrored in [`docs/specs/stage-models.md`](../../specs/stage-models.md), drift-guarded):

  | Tier | Fixed stages / role (NOT overridable) | fab-kit default profile |
  |------|---------------------------------------|-------------------------|
  | `default` | intake (advisory only ‚Äî foreground); spawned worker sessions (`fab batch`), `fab agent` with no tier; **per-field fallback for every other tier** | `claude` / `claude-fable-5` / `xhigh` |
  | `operator` | the operator coordinator session (`fab operator`) | `claude` / `claude-sonnet-5` / `medium` |
  | `doing` | `apply`, `review-pr`, `hydrate` ‚Äî execution that must not err | `claude` / `claude-opus-4-8` / `xhigh` |
  | `review` | `review` ‚Äî the critic (split from `doing` for author/critic separation) | `claude` / `claude-fable-5` / `xhigh` |
  | `fast` | `ship` ‚Äî near-mechanical work | `claude` / `claude-sonnet-5` / `low` |

  **`review` vs `review-pr` are deliberately different tiers** despite the shared word: `review` is the critic (discovers what's wrong from a diff ‚Üí `review` tier); `review-pr` is responsive (fixes already-articulated feedback ‚Üí `doing`). Do not group them.

  **Per-field merge with default-tier inheritance (tykw)**: any tier field left unset (provider, model, effort) inherits from the project's `default` tier, then from fab-kit's built-in for that tier. E.g. a project `agent.tiers.doing: { effort: high }` (no provider/model) resolves provider+model from the project's `default` tier (or its built-in) with effort=high winning. An empty model signals "inherit the session/orchestrator model". Inheriting `{provider, model, effort}` is safe *precisely because commands moved to `providers:`* ‚Äî the dangerous cross-semantics command inheritance can no longer happen (the reason l3ja/24ec's `{model, effort, spawn_command}` merge deliberately excluded inheriting a `spawn_command`).

  **Documented style: write `provider:` explicitly on every tier line** even though inheritance makes it optional ‚Äî per-line readability; inheritance is the safety net, not the style.

  **Resolution is provider-neutral with NO validation** ‚Äî `fab resolve-agent <stage|tier>` (see below; `_cli-fab.md` is the canonical CLI reference) maps stage ‚Üí tier ‚Üí `{provider, model, effort}` and echoes the strings verbatim. fab does not validate the model or effort against any provider's accepted set (Constitution Principle I); a misconfigured pair (e.g. Sonnet + `xhigh`) is NOT corrected by fab and surfaces as a dispatch-time error in the harness. This is what lets a project switch the underlying agent by adding a `providers:` entry and overriding `agent.tiers` with another provider's model IDs and effort vocabulary (`gpt-5` / `reasoning_effort:high`, etc.) with nothing in fab rejecting it. **Haiku is excluded from the defaults** (no effort param ‚Äî passing effort 400s ‚Äî and `ship` needs faithful PR-description comprehension), but a user MAY still override a tier to Haiku via pass-through. Model IDs are written **versioned** (`claude-sonnet-5`, `claude-opus-4-8`): bare family IDs (`claude-sonnet`) are invalid at the API and miss `ModelAlias`'s trailing-hyphen prefix match, so they fail both dispatch seams. The dispatch wiring (the orchestrators / `/fab-continue` calling `fab resolve-agent` before each stage's sub-agent) is documented in [context-loading.md](/_shared/context-loading.md) ¬ß Per-Stage Model Resolution. See [pipeline/change-lifecycle.md](/pipeline/change-lifecycle.md) for where this sits in the lifecycle, and [runtime/providers-and-tiers.md](/runtime/providers-and-tiers.md) for the full providers/tiers model.

  **Non-pipeline consumers.** The stage pipeline is not the only consumer of the agent-tier system. `fab operator` (the standalone coordination command ‚Äî NOT a pipeline stage) launches its coordinating agent on the **operator** tier (as of tykw ‚Äî previously it borrowed the doing tier via `fab resolve-agent apply`, 2sdj), and the new `fab agent [tier]` command (tykw) resolves any tier for interactive launch. This confirms the tier system is consumable by any command that spawns an agent, not just the stage pipeline. See [runtime/operator.md](/runtime/operator.md) and [runtime/providers-and-tiers.md](/runtime/providers-and-tiers.md).

#### `true_impact_exclude`
Optional top-level field. A YAML sequence of pathspec exclusion patterns ‚Äî typically directory prefixes ending in `/` (e.g., `fab/`, `docs/`, `vendor/`), but any pattern accepted by `git diff` `:(exclude)<pattern>` syntax is valid. Scaffold default: `[fab/, docs/]` so new fab-kit projects emit the impact block out of the box.

Semantics:
- **Present and non-empty** ‚Äî the Impact table of the PR `## Meta` block (rendered by `fab pr-meta` as of rj31; previously inlined in `/git-pr`) names the excluded scope in its `<sub>` provenance caption (`<sub>excludes `fab/`, `docs/` ¬∑ generated by fab-kit vX.Y.Z</sub>`, since the pnao normalization + 260625 layout revision ‚Äî no separate `**Impact**:` lead-in line), the caption reflecting the config values verbatim (each per-element backtick-wrapped, never hardcoded). Computed via `git diff --shortstat "$BASE...HEAD"` against the merge-base with and without the `:(exclude)<pattern>` pathspec args (canonical math in `internal/impact`).
- **Absent, `null`, or `[]`** ‚Äî the impact block is omitted entirely; the rest of the PR body matches the no-block output byte-for-byte.
- **No fab context** (`fab/project/config.yaml` does not exist) ‚Äî the block is omitted silently, preserving `/git-pr`'s fab-optional behavior.
- **True-impact pass returns zero** (every modified file falls inside an excluded path) ‚Äî the entire block is omitted to avoid a misleading `+0 / ‚àí0` line.

Consumed by the impact engine (`internal/impact`) and its consumers: the `fab pr-meta` subcommand (which renders the PR `## Meta` block Impact line for `/git-pr` as of rj31 ‚Äî `/git-pr` no longer assembles it inline), the `fab impact` CLI, and the apply/hydrate/ship `true_impact` write path. The exclude values are wrapped per-element in backticks in the rendered Impact line (never hardcoded).

#### `test_paths`
Optional top-level field (7t5a). A YAML sequence of glob/pathspec patterns identifying test files, mirroring the `source_paths` / `true_impact_exclude` style (top-level list of strings). Read into `config.Config` as `TestPaths []string` (`yaml:"test_paths"`).

**Purpose ‚Äî attribution, NOT exclusion.** `test_paths` does NOT strip lines from the impact universe ‚Äî it *attributes* the already-scaffolding-excluded lines to tests vs. implementation, so reviewers see the split (e.g. "140 impl / 400 test") instead of one conflated number. This is orthogonal to `true_impact_exclude`, which removes scaffolding/doc *noise* from the universe entirely. Tests are first-class deliverables (constitution principle VII, Test Integrity), not scaffolding noise ‚Äî adding test patterns to `true_impact_exclude` was explicitly rejected because it would conflate the two axes and destroy the test-coverage signal.

**No kit default (Portability, constitution principle V).** fab-kit is language-agnostic; there is no universal test-file pattern (`*_test.go`, `test_*.py`, `*.spec.ts` all differ by language), so the kit MUST NOT ship a hardcoded default `test_paths` list. Instead, the value is **detected per project at setup**: the scaffold (`$(fab kit-path)/scaffold/fab/project/config.yaml`) ships a standing annotated example comment block (Go/Python/JS-TS/Java-Kotlin patterns + the `:(glob)`/anti-substring note) above a `{TEST_PATHS}` placeholder, and `/fab-setup config` create-mode auto-detects the project's ecosystem from on-disk marker files (`go.mod`‚ÜíGo, `pyproject.toml`‚ÜíPython, etc.) and fills the anchored pattern **non-interactively**. Unrecognized or inline-test stacks (e.g. Rust) are left empty ‚Äî the breakdown then collapses to a single total. Existing projects are backfilled by the `2.7.1-to-2.8.0` migration (same detection; skips projects that already set `test_paths`). This repo dogfoods it with `test_paths: ["**/*_test.go"]`.

**Glob/pathspec format.** Include patterns are applied as `:(glob)<pattern>` magic pathspecs in the test-only `git diff --shortstat` pass, so wildcards behave like `.gitignore`-style globs ‚Äî notably `**` matches across directory boundaries, so `**/*_test.go` matches both root-level `foo_test.go` and nested `pkg/foo_test.go`. (Without `:(glob)`, plain git pathspec rules treat `*` as not crossing `/`, silently missing root-level test files and under-counting tests.) The pass combines these includes with the same `:(exclude)` args derived from `true_impact_exclude`, so tests are counted within the scaffolding-excluded universe; when `true_impact_exclude` is empty, the pass runs with the includes alone (tests attributed within the raw universe).

**Graceful collapse.** When `test_paths` is absent, `null`, or `[]`, the impact engine skips the test pass entirely (`Result.Tests` stays nil), no `tests` sub-block is written to `.status.yaml`, and impl/tests rendering collapses to today's single-number display ‚Äî matching the existing lazy-omit posture of the `excluding` sub-block.

Consumed by the impact engine (`internal/impact/`, via `ComputeForRepo`) and rendered by the `/git-pr` PR body (the `fab pr-meta` Meta-block Impact table ‚Äî nested `‚îî impl`/`‚îî tests` rows under the bold `true` row, normalized in pnao) and `fab change list --show-stats` (compact `{impl}i+{tests}t={total}` column). See the `true_impact` block in [schemas.md](/pipeline/schemas.md) for the `tests` sub-block schema and the render-time `impl = max(0, total ‚àí tests)` residual.

#### `stage_directives` (removed in c5tr)
Removed from the config surface in 260612-c5tr. The key promised per-stage artifact-generation directives but had **zero readers, ever** ‚Äî the scaffold seeded it, `/fab-setup` edited it, three migrations preserved/relocated it (j6cs's `1.9.7-to-1.10.0` relocated `spec` directives into `apply`), yet no skill or binary consumed it: directives placed there were silently ignored. Worse, `fab init` stamps `fab_version` past all migrations, so freshly scaffolded projects kept the zombie block permanently ‚Äî including a `[NEEDS CLARIFICATION]` directive that contradicted the intake-only marker rule (post-1.10.0) and a GIVEN/WHEN/THEN directive redundant with `_generation.md`'s mandated scenarios. Disposition: remove-dead (the wire alternative ‚Äî having `_generation`/`_review` consume `stage_directives.{stage}` ‚Äî would have added prompt surface to every generation for a feature nobody had ever successfully used). The scaffold no longer seeds it, `/fab-setup config` no longer offers it, the always-load descriptor no longer mentions it, and migration `2.1.6-to-2.2.0.md` drops it (plus a defensive `model_tiers`) from existing user configs ‚Äî see [migrations.md](/distribution/migrations.md). No directive was relocated; nothing replaces the key.

#### `stage_hooks`
Optional map of per-stage pre/post shell commands honored by `fab status start`/`finish` ‚Äî live Go behavior (`config.go`, `status.go`) first documented in c5tr; `_cli-fab.md` ¬ß stage_hooks is the canonical reference. Not seeded by the scaffold ‚Äî add the key by hand. Shape: `stage_hooks.{stage}.pre/post`, each value a command line executed as `sh -c` from the repo root with stdout/stderr passthrough; absent/empty hooks (or a missing config file) are silent no-ops. A failing `pre` hook **blocks `fab status start`** (the transition is not applied); a `post` hook runs after `finish`'s transition is saved (stage already `done`, next stage already auto-activated). Caveats: `finish`'s auto-activation does NOT fire the next stage's pre hook, and a failing post hook cannot be re-fired by re-running `finish` (done‚Üídone is rejected ‚Äî run the hook command by hand, or `reset` the stage first). Migration `2.1.6-to-2.2.0.md` explicitly preserves the key. See [pipeline/change-lifecycle.md](/pipeline/change-lifecycle.md) for the lifecycle-side summary.

A retired-keys guard (`TestConfigReferenceRetiresLegacyKeys`) asserts `review_tools` no longer appears in `fab config reference`.

### `context.md`

`fab/project/context.md` is a free-form markdown file describing the project's tech stack, conventions, architecture, and domain context. Skills load this to understand the project landscape. For monorepos, use labeled sections per package. Optional ‚Äî skills proceed without error if missing. Generated by `/fab-setup` during bootstrap from a scaffold template.

### `code-quality.md`

`fab/project/code-quality.md` is a markdown file defining coding standards consumed during apply and review. Contains three sections:
- `## Principles` ‚Äî Positive coding standards the agent follows as soft constraints during apply
- `## Anti-Patterns` ‚Äî Patterns to avoid, checked during review with file:line references on violation
- `## Test Strategy` ‚Äî Controls test timing. Valid values: `test-alongside` (default), `test-after`, `tdd`

Optional ‚Äî skills proceed without error if missing. Generated by `/fab-setup` during bootstrap from a scaffold template.

### `code-review.md`

`fab/project/code-review.md` is a markdown file defining review policy consumed by the validation sub-agent during review. Contains seven sections:
- `## Severity Definitions` ‚Äî Three-tier priority scheme: must-fix, should-fix, nice-to-have. Overrides the hardcoded defaults in the skill prompt when present
- `## Review Scope` ‚Äî What the sub-agent inspects (changed files only, exclusions for generated code/vendors)
- `## False Positive Policy` ‚Äî How to suppress findings (inline `<!-- review-ignore: {reason} -->` comments)
- `## Rework Budget` ‚Äî rework-cycle budget for the `/fab-ff`/`/fab-fff` auto-rework loop. The `Max cycles: {N}` line is **consumed** since c5tr: `_pipeline.md` reads it at bracket entry as the `{max_cycles}` cycle cap, defaulting to 3 when the file, section, or line is absent (pre-c5tr the knob existed but nothing read it ‚Äî `_pipeline` hard-coded 3). Only the cap is configurable; the escalation threshold (2 consecutive fix-code attempts) stays fixed
- `## Project-Specific Review Rules` ‚Äî Project-specific validation rules
- `## Parsimony Pass` *(optional)* ‚Äî Single field `Enabled: true|false` (default `true` when section absent or field unset). When `true`, the inward sub-agent's parsimony validation step runs (per [execution-skills](/pipeline/execution-skills.md) Validation Step 7). When `false`, the parsimony step is silently skipped (treated identically to a skip-list match ‚Äî other validation checks still run). This is the **only** parsimony-related project-level knob ‚Äî the 100-line advisory threshold and the `[docs, chore, ci]` skip list are intentionally hard-coded in the kit (per intake rationale: "until real-world usage shows the defaults are wrong, a single set of numbers across projects keeps the surface area small")
- `## Review Tools` *(optional ‚Äî the retired `review_tools` config block's home, tykw)* ‚Äî Toggles for the automated reviewers, listed as `- <tool>: false` bullets. **Absent section / absent tool = enabled** (list a tool `false` only to DISABLE it). Two tool sets: `codex`/`claude` control the review-stage outward-reviewer Codex‚ÜíClaude cascade (read by `/fab-continue` review's outward sub-agent in `_review.md` ‚Äî Codex tried first, Claude the fallback); `copilot` controls the `/git-pr-review` Phase 2 Copilot request (the `--tool copilot` flag force-overrides it). The scaffold `code-review.md` ships this section as a commented block; `fab/project/code-review.md` writers seed it only when a tool is disabled

Note the author-vs-critic split: `code-quality.md` guides the *writing* agent during apply; `code-review.md` guides the *reviewing* sub-agent during review. Different cognitive modes, different concerns.

Optional ‚Äî skills proceed without error if missing. Generated by `/fab-setup` during bootstrap from a scaffold template.

### Hardcoded Kit Knobs (Not Configurable)

A few quality/shape thresholds are deliberately **hardcoded in the kit**, not surfaced in `config.yaml` ‚Äî keeping a single set of numbers across projects keeps the config surface small until real-world usage shows the defaults are wrong:

- **Parsimony advisory threshold (100 lines) + the `[docs, chore, ci]` skip list** ‚Äî the only project-level parsimony knob is `code-review.md`'s `## Parsimony Pass` `Enabled: true|false` toggle (see above).
- **Memory tree shape bounds** (tciy) ‚Äî the ideal-shape guidance is hardcoded SHOULD guidance, NOT a `config.yaml` field: **~12 topic files per folder** (soft upper bound; `fab memory-index` warns above it), **~5** lower floor, **depth ‚â§ 3** (`docs/memory/{domain}/{sub-domain}/{topic}.md`), and a sub-domain earns its own index only when a cohesive cluster of **‚â• 8 files** exists. Reserved domains `_shared/` (cross-cutting) and `_unsorted/` (staging) are exempt from the width warning. The thresholds live in `internal/memoryindex` (`WidthWarnThreshold = 12`, `MaxDepth = 3`) and are documented as guidance in the hydrate skills, `docs-reorg-memory`'s Shape Report, and `docs/specs/templates.md`. A `config.yaml` surface was explicitly deferred (YAGNI; it can be added with the follow-up rebalance change if needed).

### `constitution.md` Structure

The constitution is the architectural DNA of a Fab project. It defines immutable principles that govern how specifications become code.

#### Structure

```markdown
# {Project Name} Constitution

## Core Principles

### I. {Principle Name}
{Description using MUST/SHALL/SHOULD keywords. Include rationale.}

## Additional Constraints
<!-- Project-specific: security, performance, testing, etc. -->

## Governance
**Version**: {MAJOR.MINOR.PATCH} | **Ratified**: {DATE} | **Last Amended**: {DATE}
```

#### Purpose

1. **Enforce discipline** ‚Äî prevent over-engineering and architectural drift
2. **Ensure consistency** ‚Äî all code follows the same patterns
3. **Guide AI agents** ‚Äî principles constrain agent behavior during planning and implementation

#### How Skills Use It

- `/fab-setup` generates it from project context (README, existing memory, conversation)
- `/fab-continue` and `/fab-ff` load it when generating the intake and the unified `plan.md` (`## Requirements` + `## Tasks` + `## Acceptance`) at apply entry
- `/fab-continue` (review) checks implementation against constitutional principles (not just the requirements)
- Constitution violations found during review are flagged as high-severity issues

#### Versioning

Semantic versioning ‚Äî MAJOR for principle removals, MINOR for additions, PATCH for clarifications. Changes to the constitution SHOULD be intentional and documented, not done as a side effect of a change.

### Relationship Between Configuration Files

- `config.yaml` holds **project settings** (identity, source/test paths, true-impact excludes, plan-acceptance categories, the `providers:` command table, the `agent.tiers` five-role-tier override, optional `stage_hooks` ‚Äî the real consumed surface; the former `review_tools` toggles moved to `code-review.md` ¬ß Review Tools and `agent.spawn_command` to `providers.claude.session_command`, both in tykw)
- `constitution.md` holds **principles and constraints** (MUST/SHOULD/MUST NOT rules)
- `context.md` holds **free-form project context** (tech stack, conventions, architecture)
- `code-quality.md` holds **coding standards** (principles, anti-patterns, test strategy)
- `code-review.md` holds **review policy** (severity definitions, scope, rework budget)
- All five are loaded by the "Always Load" context layer (see [context-loading](/_shared/context-loading.md)); `context.md`, `code-quality.md`, and `code-review.md` are optional

## Lifecycle Management

### Updating Config

Run `/fab-setup config` to see all editable sections (menu renumbered in c5tr ‚Äî the former `stage_directives` and `model_tiers` items are gone with their keys):

1. `project` ‚Äî name and description
2. `source_paths` ‚Äî implementation code directories
3. `checklist` ‚Äî extra plan-acceptance categories
4. `context.md` ‚Äî free-form project context
5. `code-quality.md` ‚Äî coding standards for apply/review
6. `code-review.md` ‚Äî review policy for validation sub-agent
7. Done

Skip the menu with `/fab-setup config <section>` (e.g., `/fab-setup config context`); `stage_directives` is rejected as an unknown section.

Updates use targeted string replacement on the specific section being edited. Comments and formatting in other sections are preserved. This is important because `config.yaml` relies on inline comments for self-documentation.

| Scenario | Command |
|----------|---------|
| See every available config option (the full schema) | `fab config reference` (canonical schema-discovery surface ‚Äî see above) |
| Add a new tech to the stack | Edit `fab/project/context.md` directly or `/fab-setup config context.md` |
| Add a new source directory | `/fab-setup config source_paths` |
| Add a custom checklist category | `/fab-setup config checklist` |
| Verify config after manual edit | `/fab-setup validate` |

### Amending Constitution

Run `/fab-setup constitution` when `constitution.md` exists to enter amendment mode:

1. Current constitution is displayed
2. Amendment menu offers: add principle, modify principle, remove principle, add/modify constraint, update governance
3. Multiple amendments can be made per session
4. Version is bumped automatically based on change severity

When `constitution.md` doesn't exist, `/fab-setup constitution` generates one from project context: `config.yaml`, README, codebase patterns, and conversation.

#### Semantic Versioning

Constitution versions follow `MAJOR.MINOR.PATCH`:

| Change type | Bump | Example |
|-------------|------|---------|
| Remove or fundamentally change a principle | MAJOR | `1.2.0 ‚Üí 2.0.0` |
| Add a new principle or constraint | MINOR | `1.2.0 ‚Üí 1.3.0` |
| Clarify wording without changing meaning | PATCH | `1.2.0 ‚Üí 1.2.1` |

When multiple amendments are made in one session, the highest-severity bump takes precedence (MAJOR > MINOR > PATCH).

#### Structural Rules

The constitution maintains a consistent structure:

- Level-1 heading: `# {Project Name} Constitution`
- `## Core Principles` with Roman numeral headings (`### I.`, `### II.`, etc.)
- `## Additional Constraints`
- `## Governance` with version, ratified date, and last amended date

When principles are removed, remaining principles are re-numbered sequentially.

Amendment summaries are included in the command output. The constitution file itself does not contain a changelog ‚Äî git history serves as the authoritative record, and the version number provides semantic signal.

### Validation

Run `/fab-setup validate` to check structural correctness of both files:

- `config.yaml`: 8 checks (YAML syntax, required fields, stage dependencies, circular deps)
- `constitution.md`: 6 checks (structure, headings, governance section, version format)

Every failure includes an actionable fix suggestion. Useful after manual edits or as a pre-commit check. `/fab-setup config` also validates automatically after each edit, offering to revert invalid changes.

See [setup](/distribution/setup.md) for the complete command suite.

## Design Decisions

### Config for Facts, Constitution for Principles
**Decision**: Separate factual configuration (`config.yaml`) from principles and constraints (`constitution.md`).
**Why**: Config changes frequently (new dependencies, naming tweaks). Constitution changes rarely and deliberately. Separation makes the governance boundary clear.
**Rejected**: Single config file mixing facts and principles ‚Äî blurs what's configurable vs. what's immutable.
*Source*: doc/fab-spec/ARCHITECTURE.md

### Constitution Loaded Implicitly, Not Gated
**Decision**: Skills load `constitution.md` as context and are expected to respect principles implicitly. Constitutional violations are caught during `/fab-continue` (review), not gated at plan time.
**Why**: Lightweight enforcement. A formal gate at every stage adds friction without proportional benefit ‚Äî most violations are caught naturally by the agent.
**Rejected**: Explicit "Constitution Check" gate in the plan template (SpecKit's approach) ‚Äî too heavyweight for Fab's lightweight workflow.
*Source*: doc/fab-spec/TEMPLATES.md

### Stage Graph in Schema, Not Config
**Decision**: The stage pipeline is defined by the workflow schema authority ‚Äî since c5tr the Go state machine (`src/go/fab/internal/status` + `statusfile`; the former declarative `$(fab kit-path)/schemas/workflow.yaml` was itself retired with zero consumers ‚Äî see [pipeline/schemas.md](/pipeline/schemas.md)) ‚Äî not in `config.yaml`. The `stages:` section was removed from config in v0.8.0 because no skill ever consumed it ‚Äî all skills derive stage ordering from the state machine's CLI surface or their own hardcoded logic.
**Why**: The stages section was dead config. Removing it reduces config surface area without losing functionality.
**Superseded**: Earlier design placed stages in config for inspectability. In practice, this was never used.
*Source*: 260218-bb93-restructure-config-yaml; *Updated by*: 260612-c5tr-scaffold-config-truth-srad-coherence (the workflow.yaml the decision pointed at was retired; the stage graph lives in the Go state machine)

### Author-vs-Critic Split (code-quality.md vs code-review.md)
**Decision**: Separate coding standards for the writing agent (`code-quality.md`) from review policy for the validating sub-agent (`code-review.md`).
**Why**: The apply agent and review sub-agent operate in different cognitive modes. The author needs principles and anti-patterns to guide implementation. The critic needs severity definitions, scope boundaries, and rework budgets to guide validation. Combining both in one file conflates two distinct concerns.
**Rejected**: Single `code-quality.md` with a review section ‚Äî blurs the author/critic boundary and makes it unclear which sections are consumed by which agent.
*Introduced by*: 260218-xkkc-add-code-review-5cs-quality

### Companion Markdown Files Over YAML Sections
**Decision**: Extract `context:` and `code_quality:` from `config.yaml` into standalone markdown files (`fab/project/context.md`, `fab/project/code-quality.md`).
**Why**: Free-form content (tech stack descriptions, coding principles) is awkward in YAML ‚Äî multiline strings, quoting, indentation issues. Markdown is the natural format and allows richer structure (headings, lists, examples). Also reduces config.yaml to pure settings.
**Rejected**: Keeping everything in config.yaml ‚Äî bloats a settings file with prose content.
*Source*: 260218-bb93-restructure-config-yaml

### Single config.yaml Parser in the fab Module (`internal/config`)
**Decision**: The fab module reads `fab/project/config.yaml` exclusively through `internal/config` ‚Äî every consumed key (`stage_hooks`, `true_impact_exclude`, `test_paths`, `branch_prefix`, `fab_version`, `agent.spawn_command`, `project.linear_workspace`) is modeled on the single `Config` struct and read via a nil-safe accessor (`GetStageHook`, `GetBranchPrefix`, `GetFabVersion`, `GetSpawnCommand`, `GetLinearWorkspace`). `Load(fabRoot)` is a thin join over `LoadPath(path)`; explicit-path callers (`fab spawn-command --repo <path>`) use `LoadPath` directly. A missing file yields an empty config without error; default semantics live with each consumer (spawn's default command, empty branch prefix, empty Linear workspace, preflight's silent staleness skip), not in the loader.
**Why**: The file was previously parsed at five independent fab-module sites with one-off structs (the shared `Config`, `internal/spawn`'s local parse, `batch_switch.go`'s `getBranchPrefix`, `internal/prmeta`'s `readLinearWorkspace` ‚Äî which parsed the same file twice within one derivation ‚Äî and `internal/preflight`'s anonymous staleness struct), each a silent-drift surface. yaml.v3 ignores unknown keys, so widening the shared struct is free for existing configs.
**Known coupled-failure caveat (deliberate)**: a single Unmarshal couples failure modes ‚Äî a YAML type error on *any* modeled key fails the whole parse and sends *every* accessor to its documented fallback. This is safe because each consumer defines a fallback (verified by the malformed-yaml coupled-fallback test in `internal/config`), but a malformed config now degrades all consumers together instead of per-key ‚Äî a recorded semantic change accepted with the consolidation.
**Rejected**: Keeping per-caller structs (the status-quo drift source the consolidation removes); sharing the parser with the fab-kit module (its `readFabVersion` stays ‚Äî Go internal-package visibility forbids cross-module imports).
> **Updated by tykw**: the consolidation principle stands, but the accessor set moved ‚Äî `GetSpawnCommand` was **removed** with `agent.spawn_command`'s relocation to `providers.claude.session_command`; the fab module now reads the command via `GetProvider(name)` + `agent.ResolveProvider`, and tiers via `GetAgentTier`. `fab spawn-command --repo <path>` (an explicit-path `LoadPath` caller) was retired for `fab agent --repo <path>`, which reads the target repo's config the same way.
*Introduced by*: 260612-ye8r-cli-single-sourcing-doc-conformance; *Updated by*: 260702-tykw-agent-providers-role-tiers (accessor set: `GetSpawnCommand` ‚Üí `GetProvider`; `fab spawn-command --repo` ‚Üí `fab agent --repo`)

### `agent.tiers` Overrides Budget, Not Taxonomy (Fixed Mapping vs Overridable Profile)
**Decision**: `agent.tiers` is the **only** per-stage-model override surface, and it overrides *what each tier means* (provider + model + effort ‚Äî since tykw; model + effort before), never *which stages belong to a tier*. Since tykw the mapping is five role tiers: default={intake (advisory), fab batch, fab agent}, operator={fab operator}, doing={apply, review-pr, hydrate}, review={review}, fast={ship}. The mapping and the default profile per tier are fab-owned, fixed, and live in the Go `internal/agent` package ‚Äî there is no `stage_tiers` config and no per-stage escape hatch.
**Why**: The taxonomy is fab's considered judgment from a dimensional analysis of each stage's cognitive mode; users legitimately disagree about *budget* (what a tier costs), not *taxonomy* (which stages cluster). Abstract tiers also survive model churn ‚Äî when a new top model (Fable) lands, fab bumps the one default table and every non-overriding project upgrades for free, where pinned model IDs in config would rot. Per-field merge (with default-tier inheritance since tykw) means an omitted field inherits rather than blanking it. The split keeps the override surface small (5 role tiers, 3 dials each) instead of six stages √ó three dials.
**Rejected**: `stage_tiers` reassignment / per-stage `{provider, model, effort}` pins (invites undoing reasoning the user hasn't done; explodes the knob count). Pinned model IDs in config (opts every project out of the upgrade curve). A user (`~/.fab-kit`) config layer (dropped in design ‚Äî only fab-kit defaults + project override).
**Related**: `review` (the critic ‚Üí `review` tier) and `review-pr` (responsive ‚Üí `doing`) are deliberately in **different** tiers despite the shared word ‚Äî discovering bugs from a diff vs. fixing already-articulated feedback are different cognitive modes. `apply` stays on `doing`, not a cheaper tier: apply produces the diff review critiques, so a cheaper apply drives more (capped-at-3) rework rounds. Before tykw the `review` stage rode a `thinking` tier shared with intake; tykw split `review` into its own tier (author/critic separation) and dissolved `thinking` (intake never dispatches ‚Äî see the tykw decision below).
*Introduced by*: 260613-l3ja-per-stage-model-tiers; *Updated by*: 260702-tykw-agent-providers-role-tiers (five role tiers; `review` split from `thinking`; provider added to the tier profile)

### No Validation ‚Äî Provider-Neutral Verbatim Pass-Through
**Decision**: `fab resolve-agent` performs NO validation ‚Äî it maps stage‚Üítier‚Üí`{model, effort}` and echoes both strings verbatim, whatever they are. fab has no provider-specific knowledge in the resolution path; the resolved profile is consumed unchanged by the harness adapter.
**Why**: Provider neutrality (Constitution Principle I). Validating against Claude's effort enum (`low/medium/high/xhigh/max`, Opus-only `xhigh`, Haiku-rejects-all) would hard-code Claude into the resolver and bolt the door on other agents. Keeping it open is what lets a project switch the underlying agent by overriding `agent.tiers` with another provider's vocabulary. The safety net moves from fab to the runtime/harness: a misconfigured pair (e.g. Claude Sonnet + `xhigh`, which Sonnet 400s) surfaces as a dispatch-time error, not a fab correction. v1 is **architecture-neutral + documented, NOT shipped/tested against a non-Claude harness** ‚Äî no per-provider default tables, no provider-detection; the acceptance proof is "a non-Claude project can override the tiers and nothing in fab rejects it." The fab-kit shipped defaults use Claude model IDs/effort (documented as "fab-kit's Claude defaults", fully replaceable).
**Rejected**: Effort-enum enforcement / a degrade-gracefully drop of an incompatible effort (an earlier `stage-models.md` iteration ‚Äî explicitly removed in this change). Shipped/tested multi-provider support (a far larger change, out of scope for v1).
*Introduced by*: 260613-l3ja-per-stage-model-tiers

### `spawn_command` Placeholders ‚Äî Template Mode with Empty-Value Token-Drop (6tmi)
**Decision**: `spawn.WithProfile` ‚Äî the one place fab composes a spawn command line ‚Äî becomes provider-forgiving via literal `{model}`/`{effort}` placeholders. When either is present the command is a **template** (substitute every occurrence, all-or-nothing so the append is disabled entirely); when neither is present it keeps today's Claude-shaped append byte-for-byte (back-compat). An **empty** substituted value triggers a token-drop rule: tokenize on whitespace, drop the placeholder's token plus a preceding `-`-flag token ‚Äî deterministically covering the four flag shapes `-m {model}` / `--model {model}` / `--model={model}` / `-c key={effort}`. The all-non-empty path uses plain `strings.ReplaceAll` on the raw string, so author whitespace is preserved and tokenization runs only on the empty-value drop path. A named `spawn.StripPlaceholders(cmd)` (thin wrapper over `WithProfile(cmd, "", "")`) states the leak-prevention intent at every raw-consumer site (`fab spawn-command`, `fab batch new`, `fab batch switch`), so a templated command interpolated into a shell without a resolved profile degrades to a clean invocation rather than leaking literal braces (see [runtime/operator.md](/runtime/operator.md) ‚Üí "Operator Launch‚Ä¶" and [distribution/kit-architecture.md](/distribution/kit-architecture.md)).
**Why**: fab's whole resolution path (`fab resolve-agent`, `agent.tiers`) is deliberately provider-neutral / verbatim, but the pre-6tmi `WithProfile` blindly appended Claude CLI grammar (` --model ‚Ä¶ --effort ‚Ä¶`), so pointing `spawn_command` at a non-Claude CLI (codex wants `-m <model>` and `-c model_reasoning_effort=<level>`, has no `--effort`) produced a broken `fab operator` launch. Placeholders relocate provider grammar into the user's config ‚Äî the one composition site now honors the same no-validation philosophy as the resolver. The token-drop rule is the simplest deterministic empty-value handling that needs no templating engine; whitespace preservation on the non-empty path is strictly safer than an unconditional single-space rejoin.
**Rejected**: Per-half independence (would append Claude flags to a non-Claude command that only templated one half). Raw empty substitution (leaves a dangling `-m` / `model_reasoning_effort=`). A full templating language (heavy, YAGNI). Printing templated `spawn-command`/`batch` output verbatim + documenting the brace-leak hazard (a foot-gun; `StripPlaceholders` degrades it cleanly instead). Threading template awareness into `operator.go` (needless coupling ‚Äî `WithProfile` stays the single seam, its signature unchanged). **Non-goals (at the time)**: per-tier `spawn_command` and a cross-harness stage-dispatch adapter (a follow-up change, spec-first) ‚Äî **that follow-up is 24ec (below), now shipped**: it widened `TierProfile`/`Profile` with the opt-in `spawn_command` and made `fab resolve-agent` emit the `spawn=` line by reusing this decision's `spawn.WithProfile` substitution. No provider validation (placeholders only relocate *where* the strings land).
> **Updated by tykw**: `spawn.WithProfile` (the single composition seam) is unchanged and reused verbatim, but the raw-consumer `StripPlaceholders` print path was **deleted** ‚Äî the three consumers that stripped an empty profile (`fab spawn-command`, `fab batch new`, `fab batch switch`) no longer do so: `fab spawn-command` was retired for `fab agent [--print]`, and `fab batch` now composes the **default-tier** provider `session_command` + profile (workers spawn WITH a profile, so there is no empty-profile print path to leak-guard). Placeholders now live on the provider command (`providers.<name>.session_command`/`dispatch_command`), not on `agent.spawn_command` or a per-tier `spawn_command`.
> **Updated by gvxd**: the template mode this decision built is now used by the **built-in default itself** ‚Äî `agent.DefaultSessionCommand` / `spawn.DefaultSpawnCommand` (the built-in `claude` `session_command` and the no-`session_command` fallback) became the `{model}`/`{effort}` template `claude --dangerously-skip-permissions -n "$(basename "$(pwd)")" --model {model} --effort {effort}`, with the placeholders **placed last** so template-mode substitution yields output byte-identical to what the append path produced for the former plain form (verified empirically against `fab agent default/operator --print`; zero behavior change). Append mode is untouched and stays load-bearing for existing plain-form user configs carried forward by the 2.12.1‚Üí2.13.0 migration (some pin `--model`/`--effort` and rely on append-last/last-wins), so **no migration ships** ‚Äî the form drift is cosmetic since both resolve identically. *Why templated at all* (the change was structural, not behavioral): (1) **explicitness** ‚Äî a reader of the scaffold now *sees* where the tier profile lands instead of it being appended invisibly by `WithProfile`; (2) **internal consistency** ‚Äî the live claude `session_command` was the file's *only* non-templated command, while its own commented `dispatch_command` and the codex/gemini starter blocks were already templated; (3) **no live drift** ‚Äî `fab config reference` renders the `session_command` from `agent.DefaultSessionCommand` (`configref.go`), so a scaffold-only change would have made `fab init` output contradict `fab config reference`; templating the constant keeps scaffold ‚Üî reference in sync by construction.

### Per-Tier `spawn_command` ‚Äî Cross-Harness Stage-Dispatch Opt-In, No Cross-Fallback (24ec)
**Decision**: Widen the tier profile from `{model, effort}` to `{model, effort, spawn_command}` (`TierProfile` in `internal/config`, mirrored on the resolution-side `Profile` in `internal/agent`, with `Resolve`'s per-field merge extended by one `if override.SpawnCommand != ""` branch). A tier `spawn_command` is the **per-stage CLI-dispatch opt-in**: PRESENT ‚Üí that tier's stages are dispatched by running the command (cross-harness, e.g. codex); ABSENT ‚Üí native Agent-tool dispatch (default). fab-kit's built-in `defaultTiers` carry **no** `spawn_command` ‚Äî the field is populated exclusively from user config. `fab resolve-agent <stage>` emits a third `spawn=<command>` line only when the resolved tier carries one, with `{model}`/`{effort}` substituted by reusing 6tmi's `spawn.WithProfile` (never a reimplementation); the `spawn=` line always embeds the FULL model ID even under `--alias` (CLI dispatch never aliases). This is part 3b of the cross-harness stage-dispatch series ‚Äî it only *emits* the command; the 3c `fab dispatch` family *runs* it.
**Why**: The tier is already the per-stage unit and `fab resolve-agent <stage>` already resolves it, so widening that existing surface is the minimal, on-model extension ‚Äî and PR #456 (6tmi) already built the `{model}`/`{effort}` template machinery this reuses. The **no-cross-fallback** rule is the load-bearing decision: `agent.spawn_command` (session boundary) and a tier `spawn_command` (stage boundary) are independent surfaces, so the absence of a resolved tier `spawn_command` ‚Äî never a lookup of `agent.spawn_command` ‚Äî is the "native dispatch" signal. A cross-fallback would silently flip every project with a configured `agent.spawn_command` (common; this repo sets it) into CLI dispatch for all stages on upgrade. Mental model (user-stated): `agent.spawn_command` hires an employee; a tier `spawn_command` outsources one task.
**Rejected**: **Cross-fallback from a tier to `agent.spawn_command`** (the silent-behavior-flip above ‚Äî the single most important thing this change does NOT do). A new top-level `stage_dispatch` map (duplicates the stage‚Üítier resolution fab already owns). Reusing `agent.spawn_command` per-stage (conflates the session boundary with the stage boundary). Struct embedding of `TierProfile` into `Profile` (new pattern for no benefit; couples the two package structs ‚Äî the field-by-field merge stays). Any validation/quoting of the emitted command (verbatim post-substitution pass-through, Constitution I; 3c owns execution and any quoting it needs).
> **Superseded by tykw**: the per-tier `spawn_command` field was **removed** ‚Äî its cross-harness stage-dispatch semantics relocated to `providers.<name>.dispatch_command` (a tier names a provider; the provider carries the command). The load-bearing **no-cross-fallback** decision is preserved intact: absence of a resolved provider `dispatch_command` is the native-dispatch signal, never a lookup of `session_command` or any session command. `fab resolve-agent`'s optional third line was renamed `spawn=` ‚Üí `dispatch=` to match. The mental model carries over (`session_command` hires an employee; `dispatch_command` outsources one task).
*Introduced by*: 260702-24ec-tier-spawn-command; *Superseded by*: 260702-tykw-agent-providers-role-tiers (tier `spawn_command` ‚Üí `providers.<name>.dispatch_command`)

### Providers Extracted from Tiers; Five Role Tiers; `review_tools` ‚Üí `code-review.md` (tykw)
**Decision**: Agent config v3 restructures three coordinated surfaces. (1) **`providers:` extracted** ‚Äî a top-level table keyed by opaque provider names, each carrying `session_command` (interactive session, the relocated `agent.spawn_command`) and/or `dispatch_command` (one headless stage task, the relocated per-tier `spawn_command`); the two are NOT merged (session vs dispatch are different binary invocations); fab-kit ships `claude` as the built-in default provider (session command, no dispatch ‚Üí native); absent `dispatch_command` = native Agent-tool dispatch with NO cross-fallback to `session_command`. (2) **Five role tiers** ‚Äî `agent.tiers` keys become `default`/`operator`/`doing`/`review`/`fast` (concrete referents replacing the `thinking`/`doing`/`fast` cognitive modes); tier values are `{provider, model, effort}` (the command moved to `providers:`); `thinking` is dissolved (intake never dispatches, so with `review` split out it governs nothing); per-field inheritance now cascades a tier ‚Üí the project's `default` tier ‚Üí fab-kit's built-in; `provider:` is written explicitly on every tier line as documented style. (3) **`review_tools` retired** into `fab/project/code-review.md` ¬ß Review Tools (the outward Codex‚ÜíClaude cascade + the Copilot toggle become prose, resolving the configref/`_review.md` contradiction). Plus a new `fab agent [tier] [--print] [--repo <path>]` command (retiring `fab spawn-command`), `fab resolve-agent`'s `spawn=`‚Üí`dispatch=` rename + tier-name acceptance + `provider=` line, and migration `2.12.1-to-2.13.0`.
**Why**: The pre-v3 agent config conflated provider mechanics with role/budget policy and the names actively confused ‚Äî two different fields both named `spawn_command` (session vs stage) needed a 7-line config comment to disambiguate, and the `thinking`/`doing`/`fast` vocabulary named cognitive modes whose referents were hidden (`thinking` "meant" review because intake never dispatches). Extraction attacks the confusion at its source rather than adding comments; role tiers give every tier a concrete referent a user can point at ("the operator", "the reviewer"). Inheriting `{provider, model, effort}` is safe *precisely because commands left the tier* ‚Äî the dangerous cross-semantics command inheritance the l3ja/24ec merge avoided can no longer happen. `review_tools` was a dead, self-contradictory config block; its toggles are review policy and belong in the prose file both consumers already load.
**Rejected**: Merging `session_command`+`dispatch_command` into one `command` (the grammars differ ‚Äî one template can't express both). Folding `agent.spawn_command` in as a `default`-tier command (implies the fallback semantics the 3a‚Äì3d series deliberately rejected). Keeping the `thinking` tier (nothing left for it to govern once review split out). Inferring a provider from a model string (needs a provider registry the no-validation/provider-neutrality contract refuses). A `fab spawn-command` deprecation alias (its only CLI consumer, the operator skill, ships and is updated in the same kit).
*Introduced by*: 260702-tykw-agent-providers-role-tiers
