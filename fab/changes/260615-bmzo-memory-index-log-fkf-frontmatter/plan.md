### memoryindex: C-lite `log.md` generation

#### R1: Per-folder `log.md` generated for every domain and sub-domain
`fab memory-index` SHALL emit a `log.md` for each domain folder **and** each sub-domain folder (the
same folders that receive a generated `index.md`), recording that folder's change history. A pure
`RenderLog(LogData) string` function SHALL produce the markdown; all I/O (git, `.status.yaml`,
registry enumeration) SHALL live in `Gather` (or a sibling gatherer), preserving the established
pure-`Render*` / I/O-`Gather` split.

- **GIVEN** a `docs/memory/{domain}/` folder with committed topic files
- **WHEN** `fab memory-index` runs
- **THEN** `docs/memory/{domain}/log.md` is written
- **AND** every sub-domain `docs/memory/{domain}/{sub}/` with committed topic files also gets a `log.md`

#### R2: `log.md` output format conforms to `fkf.md` ┬з6.2 byte-for-byte
The rendered `log.md` SHALL be:
```markdown
# Log тАФ {Title}
<!-- Generated by `fab memory-index` from git history + per-change summaries. Do not hand-edit. -->

## {YYYY-MM-DD}
- {**Verb** }[{base}](/{bundle-rel}.md) тАФ {summary-or-slug} ({change-id})
```
Entries SHALL be **date-grouped, newest-date-first**, each date a `## YYYY-MM-DD` heading. Within a
date, entries SHALL be ordered deterministically (newest commit first; ties broken by a stable key тАФ
file base then change-id) so the output is byte-stable. Each entry line SHALL carry: an optional
leading bold verb, a bundle-relative link `[{base}](/{domain}[/{sub}]/{base}.md)` (beginning with
`/`, resolved from `docs/memory/`, per ┬з7), an em-dash separator, the change's one-line summary (or
the slug fallback, R6), and the `(change-id)` token (omitted per R7 when unattributable).

- **GIVEN** a folder whose files were touched by commits on dates D2 (newer) and D1 (older)
- **WHEN** `RenderLog` runs
- **THEN** the `## D2` heading and its entries precede `## D1`
- **AND** the header is `# Log тАФ {Title}` + the generated-comment line + a blank line

#### R3: A `log.md`-free folder is handled; the generated comment marks it single-writer
A folder with **no** attributable git history for its files SHALL still degrade gracefully (an empty
or header-only `log.md` is acceptable тАФ see Design Decision 4 for the chosen "skip vs. emit header"
behavior). `log.md` SHALL carry the `<!-- Generated by ... Do not hand-edit. -->` comment, marking it
a single-writer generated artifact (same discipline as `index.md`).

- **GIVEN** a folder whose files are all uncommitted (no git dates)
- **WHEN** `fab memory-index` runs
- **THEN** the run does not error and the behavior matches Design Decision 4

#### R4: Reuse the single batched `git log` pass тАФ no per-file spawns
`loadGitDates` SHALL be extended (or a sibling projection added over the **same** `git log` output) to
capture, per commit touching `docs/memory`, the tuple `(date, subject/body, [{name-status, path}])` тАФ
**every** commit per path, not just the newest. The implementation SHALL NOT reintroduce per-file
`git log` spawns (pw3k F34 collapsed those). The existing newest-date-per-path map used by the index
SHALL remain available and unchanged in behavior.

- **GIVEN** a file touched by three commits
- **WHEN** the git pass runs
- **THEN** all three commit tuples are captured in one `git log` invocation
- **AND** the index's "Last Updated" still reflects only the newest date

#### R5: Verb derivation from per-commit name-status
For each (commit, file) tuple the leading verb SHALL be derived from git's per-commit name-status:
`A` (added) тЖТ `**Creation**`, `D` (deleted) тЖТ `**Deprecation**`, `M`/`R{score}`/`C{score}`
(modified/renamed/copied) тЖТ `**Update**`. `change_type` MAY be a secondary hint. The verb is
**optional**: when the status is unknown/ambiguous the generator SHALL omit the verb (or emit
`**Update**`), never erroring.

- **GIVEN** a commit that adds `foo.md` and modifies `bar.md`
- **WHEN** `RenderLog` renders the entries
- **THEN** `foo.md`'s entry leads with `**Creation**` and `bar.md`'s with `**Update**`

#### R6: Summary source + slug fallback
For each entry the descriptive line SHALL be the changed change's `.status.yaml` `summary:` field.
When the summary is empty/absent, the generator SHALL project the **change slug** (the folder name's
slug portion) in place of the descriptive line, per `fkf.md` ┬з6.3.

- **GIVEN** an attributable change with `summary: "did X"`
- **WHEN** the entry renders
- **THEN** the line reads `тАж тАФ did X (change-id)`
- **GIVEN** an attributable change with no `summary`
- **THEN** the line reads `тАж тАФ {slug} (change-id)`

#### R7: Commit тЖТ change-id join via the registry, with graceful degradation
The generator SHALL build the canonical `(change-id, folder)` registry by enumerating
`fab/changes/*` and `fab/changes/archive/**`. Each memory-touching commit SHALL be correlated to a
registered change by recovering a `{YYMMDD}-{XXXX}-{slug}` (or registered `{XXXX}`) token from the
commit message тАФ the merge-commit branch token (`Merge pull request #N from owner/{folder}`) is the
primary signal in this repo's history; see Design Decision 2. When a commit yields a registered
change ID, the entry SHALL carry `(change-id)` and use that change's `summary`/slug. When a commit
cannot be attributed to any registered change (direct edit on `main`, pre-FKF history, squash-merge
that dropped the branch token), the entry SHALL **degrade gracefully**: omit the `(change-id)` token
(or use the short SHA) and use `тАФ` (or the commit subject) for the descriptive line. Degradation
SHALL never error.

- **GIVEN** a merge commit `Merge pull request #265 from wvrdz/260317-yrgo-...` touching a file
- **WHEN** the join runs
- **THEN** the entry carries `(yrgo)` and `260317-yrgo-...`'s summary/slug
- **GIVEN** a squash commit `feat: тАж (#420)` with no recoverable folder token
- **THEN** the entry omits `(change-id)` and uses the fallback descriptive line, no error

### memoryindex: FKF frontmatter

#### R8: `type: memory` round-trip mechanism (no bulk-stamp, no topic-file authoring)
The generator SHALL preserve `type: memory` frontmatter when present on a file it round-trips, and
the memory-file **template** SHALL carry `type: memory`. The generator SHALL NOT inject/author
`type:` into topic files it does not own (bulk-stamping is Change 4; authoring on new files is Change
3). The generated `index.md` frontmatter is the generator's own; `type:` is carried the same way
`description:` is round-tripped today.

- **GIVEN** the memory-file template
- **WHEN** a new memory file is scaffolded from it
- **THEN** it carries `type: memory`
- **AND** `fab memory-index` does not add or remove `type:` on existing topic files

#### R9: `fkf_version: "0.1"` on the root index only
`RenderRoot` SHALL prepend a `---\nfkf_version: "0.1"\n---\n` frontmatter block to the root
`docs/memory/index.md` output. No other `index.md` tier SHALL carry `fkf_version`. The output SHALL
remain byte-stable.

- **GIVEN** any memory tree
- **WHEN** `fab memory-index` runs
- **THEN** `docs/memory/index.md` begins with the `fkf_version: "0.1"` frontmatter block
- **AND** no domain/sub-domain `index.md` carries `fkf_version`

### memoryindex: loss classification (`--check`/`--json`)

#### R10: `log.md` becomes a `--check` drift target; drift is benign (tier 1) by default
`fab memory-index --check` SHALL classify `log.md` rendered-vs-existing drift in addition to
`index.md`. `log.md` (and FKF-frontmatter) drift SHALL be classified **benign (TierBenignDrift,
exit 1)** by default. The `--json` shape SHALL be unchanged unless a new tier-2 category is added
(R11). A born-FKF tree SHALL be provably never tier 2.

- **GIVEN** a stale `log.md` whose content differs from the regenerated form
- **WHEN** `fab memory-index --check` runs
- **THEN** it reports drift at tier 1 (exit 1), not tier 0 and not tier 2

#### R11: No new destructive-loss (tier-2) category for `log.md`/frontmatter (OQ4 decision)
Per intake OQ4 / assumption #9, this change SHALL treat all `log.md`/frontmatter drift as benign and
SHALL NOT introduce a new tier-2 `LossCategory`. (The existing description/tombstone/grouping
categories remain index-only.) The `--json` `losses[].category` enum is therefore unchanged.

- **GIVEN** a hand-edited `log.md` that the C-lite projection cannot reproduce
- **WHEN** `--check` runs
- **THEN** it is classified tier 1 (benign drift), not tier 2

### Design Decisions

1. **Reuse the batched `git log` pass with a richer projection** тАФ extend `loadGitDates` to a single
   `git log -c core.quotepath=off --date=short --format=<delimited H/ad/s/b> --name-status -- docs/memory`
   pass, parsing it into BOTH the existing newest-date-per-path map (index, unchanged) AND an ordered
   per-path commit list (log). *Why*: one subprocess, honors pw3k F34. *Rejected*: a second `git log`
   invocation (extra spawn) and per-file spawns (the regression F34 fixed).

2. **Commit тЖТ change-id join is registry-gated message-token recovery, not a live branch-graph walk**
   тАФ *Why*: this repo's real history is **squash-merged** (PR subjects `feat: тАж (#NNN)` carry no
   change ID and no second parent) for recent changes, while older changes are true merge commits
   whose subject embeds the branch (`Merge pull request #N from owner/{YYMMDD-XXXX-slug}`). A live
   branch/merge-graph walk (intake assumption #12's literal framing) is **not realizable from
   `git log` alone** in a fresh clone/CI: transient branch refs are gone after squash-merge. So the
   join recovers a `{YYMMDD-XXXX-slug}` (or registered `{XXXX}`) token from the commit message and
   gates it against the enumerated `fab/changes` registry; this works from `git log` alone for the
   merge-commit regime and degrades to the intake's documented graceful fallback (omit id, slug/`тАФ`)
   for the squash regime. The registry gate keeps it authoritative (the change owns its identity).
   *Rejected*: walking `git branch --contains`/`for-each-ref` (depends on transient local refs absent
   in CI); parsing raw commit-message prose for arbitrary IDs (not registry-gated тЖТ false positives).

3. **Verb from `--name-status`** тАФ `A`тЖТCreation, `D`тЖТDeprecation, `M`/`R`/`C`тЖТUpdate; optional, omit
   on ambiguity. *Why*: ┬з6.2 makes the verb optional and the name-status is the per-commit-per-file
   signal already in the batched pass.

4. **Empty `log.md` behavior** тАФ when a folder has zero attributable commits, **skip writing
   `log.md`** rather than emit a header-only file. *Why*: an empty log carries no information and a
   header-only file would still need byte-stable regen; skipping keeps the target set equal to
   "folders with history" and avoids churning empty files. A folder with тЙе1 commit always renders.
   (Recorded as assumption A-T below тАФ Tentative; the alternative header-only emission is equally
   spec-conforming and trivially switched.)

5. **`RenderLog` is pure over `LogData`** тАФ `LogData{Title, Domain (bundle-rel base), Entries
   []LogEntry}`, `LogEntry{Date, Verb, FileBase, BundleRelPath, Summary, ChangeID}`. The gatherer
   builds `LogData` from git + registry + `.status.yaml`; `RenderLog` does no I/O. Mirrors
   `RenderRoot`/`RenderDomain`.
