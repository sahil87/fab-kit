# Intake: Mechanical PR Meta Block via `fab pr-meta`

**Change**: 260604-rj31-mechanical-pr-meta
**Created**: 2026-06-04
**Status**: Draft

## Origin

> I want to make the top part of a PR generated by fab-kit more mechanically produced - to reduce drift. This can just for the top "Meta" section.

Conversational `/fab-new` invocation. The user supplied a real example of the drift — a screenshot of a PR generated by `/git-pr` (`260604-qemg-enrich-command-reference-parser`). Inspecting it against the `git-pr.md` template surfaced concrete deviations:

1. **Excludes not backtick-wrapped** — the template specifies the single-line Impact form renders excluded paths as inline code (`` `fab/`, `docs/`, `sites/_playground/` ``), but the rendered PR shows them as plain text (`fab/, docs/, sites/_playground/`). The agent dropped the backticks.
2. **Hand-joined config values** — the agent re-reads `true_impact_exclude` from config and joins it by hand every run (here three entries), a per-invocation formatting surface.
3. **Re-derived arithmetic** — the agent computes the impl/tests/total split and the single-vs-three-row decision in prose each run, even though `fab impact` already emits the structured numbers.

Key decisions reached in conversation (recorded as Certain/Confident assumptions below):

- **Self-contained command** — `fab pr-meta` reads `.status.yaml`, parses `plan.md` checkboxes, runs the impact math, and resolves git/`gh` context (branch, owner/repo, merge-base) itself. The skill passes only the resolved change reference and PR type. (User chose "Self-contained" over "skill passes inputs as flags".)
- **Meta block only** — the command renders the `## Meta` block (table + Pipeline + Issues + Impact); `## Summary` and `## Changes` stay agent-generated because they require prose synthesis from `intake.md`. (User chose "Meta block only" over "whole PR body".)

## Why

**The problem.** The entire `## Meta` block of a fab-generated PR — a 5-column table, a `**Pipeline**:` line, an optional `**Issues**:` line, and a multi-form `**Impact**:` line — is assembled by the agent from natural-language instructions in `git-pr.md` (Step 3c). The instructions span ~110 lines of prose describing exact cell population, link construction, checkmark placement, backtick-wrapping, Unicode-minus usage, three-row-vs-single-line gating, and per-component clamping. An LLM re-interpreting that prose on every `/git-pr` run produces **drift**: the example PR already shows dropped backticks on the exclude list. Every formatting rule in those 110 lines is a rule the agent can silently violate.

**The consequence if unfixed.** Each PR's Meta block is non-deterministic — the same change shipped twice could render differently. Reviewers lose the ability to trust the Meta block as a stable, scannable artifact. The drift compounds as the template grows: the more formatting rules, the more surfaces to deviate. The arithmetic re-derivation (impl = total − tests, per-component `max(0, …)` clamping) is also error-prone in prose form and wastes tokens recomputing what `fab impact` already produces structurally.

**Why this approach over alternatives.** fab-kit already has the right pattern: `fab impact` (`src/go/fab/cmd/fab/impact.go`) mechanically owns the diff arithmetic and emits structured YAML; the skill consumes it. We extend that pattern one level up — a new `fab pr-meta` subcommand renders the *entire Meta block as final markdown*, so `git-pr.md` Step 3c collapses from ~110 lines of formatting prose to "call `fab pr-meta`, paste the result verbatim." Determinism moves into Go, where it is tested (`impact_test.go`, `score_test.go` are precedents) and constant across runs. This honors Constitution Principle III (Idempotent Operations — same inputs, same output) and keeps human/prose judgment (Summary, Changes) with the agent where it belongs. Rejected: pushing the *whole* PR body into Go would force Summary/Changes into mechanical text extraction, degrading prose quality (Constitution favors agent prose for narrative sections). Rejected: a flags-only formatter would leave the skill re-deriving git/gh inputs each run — partial drift remains.

## What Changes

### New `fab pr-meta` subcommand (Go)

A new cobra subcommand under `src/go/fab/cmd/fab/` (e.g., `pr_meta.go`), registered in `main.go` alongside `impactCmd()`, `scoreCmd()`, etc. It renders the complete `## Meta` block to stdout as final markdown.

**Signature** (mirrors `fab impact` / `fab score` conventions — positional change ref + flags):

```
fab pr-meta <change> --type <type> [--issues "DEV-1 DEV-2"]
```

- `<change>` — 4-char ID, folder substring, or full folder name (resolved via the existing `resolve` package, same as every other subcommand).
- `--type <type>` — the resolved PR type (`feat|fix|refactor|docs|test|ci|chore`). Required. The skill resolves type via its existing resolution chain (Step 0b) and passes it in; type resolution stays in the skill because it depends on the diff and user argument, which the binary shouldn't re-derive.
- `--issues "<space-joined IDs>"` — optional. The skill already gathers these via `fab status get-issues`; passing them in avoids the binary re-reading issue state. Empty/absent → no Issues line.

**Self-contained data sourcing** (the command reads everything else itself):

- **`.status.yaml`** — read via the existing `statusfile` package for `id`, `confidence.score`, `plan.acceptance_count`, `plan.acceptance_completed`, `progress.*` (pipeline + review state), `stage_metrics.review.iterations`.
- **`plan.md`** — parse `## Tasks` checkboxes (`- [x]` vs `- [ ]`) for the `{done}/{total} tasks` count. (Legacy `tasks.md` fallback for pre-1.9.0 changes — see Non-Goals if we choose to drop it.)
- **`fab/project/config.yaml`** — read `true_impact_exclude`, `test_paths`, and `project.linear_workspace`.
- **Impact math** — reuse the existing `internal/impact` package (`impact.ComputeForRepo`) directly rather than shelling to `fab impact`. Compute merge-base against `origin/main`/`origin/master` internally.
- **Git/gh context** — branch (`git branch --show-current`), owner/repo (`gh repo view --json nameWithOwner`) for blob URLs. Resolve merge-base internally.

**Output** — the exact `## Meta` block markdown, byte-for-byte what `git-pr.md` Step 3c specifies today, including:

- The 5-column table (`ID | Type | Confidence | Plan | Review`) with all cell-population rules (`—` fallbacks, ` ✓` completion suffix on Plan, `✓/✗ {N} cycle{s}` on Review).
- `**Pipeline**:` line — six stages in fixed order, ` ✓` suffix per `done` stage, hyperlinked `intake`/`apply` labels to blob URLs when artifacts exist.
- `**Issues**:` line (only when `--issues` non-empty) — Linear-linked when `linear_workspace` set, bare IDs otherwise, positioned below Pipeline and above Impact.
- `**Impact**:` line — three-row form when a `tests` pair exists, single-line form otherwise; **backtick-wrapped excludes built from actual config values**; Unicode minus `−` (U+2212); per-component `max(0, …)` clamping for the impl residual; omitted entirely when total is `+0/−0`, merge-base missing, or impact computation fails.

**Behavior when not in a fab context** — if the change can't be resolved or `.status.yaml` is absent, the command exits non-zero (the skill then omits the Meta block, exactly as today when `{has_fab}` is false). The skill treats a non-zero exit / empty stdout as "no Meta block."

### `git-pr.md` skill simplification

Step 3c's body-generation prose (the ~110 lines covering Meta table population, Pipeline line, Issues line, Impact computation/rendering, blob URL construction) is **replaced** with:

1. Resolve PR type (Step 0b — unchanged).
2. Gather issues (Step 1 — unchanged).
3. Call `META=$(fab pr-meta <change> --type <type> --issues "<issues>" 2>/dev/null)`.
4. If exit 0 and non-empty: the `## Meta` block is `$META` verbatim. Else: omit the Meta block.
5. Generate `## Summary` and `## Changes` from `intake.md` (unchanged — agent prose).

The progress token printed stays `✓ body — meta + summary + changes` (with tokens dropped when sections are omitted, as today).

### Tests

New `pr_meta_test.go` following the `impact_test.go` / `score_test.go` table-driven pattern: golden-output tests for the Meta block across the matrix of states (has/no plan, has/no tests pair, has/no issues, review done/failed/pending, empty `true_impact_exclude`, missing merge-base). Per Constitution Principle VII, tests verify conformance to the rendering spec.

## Impact

- **New code**: `src/go/fab/cmd/fab/pr_meta.go`, `src/go/fab/cmd/fab/pr_meta_test.go`. Possibly a new `internal/prmeta` package if rendering logic warrants extraction (mirrors `internal/impact`, `internal/score`).
- **Modified Go**: `src/go/fab/cmd/fab/main.go` (register the subcommand).
- **Reused Go**: `internal/impact` (impact math), `internal/statusfile` (status read), `internal/resolve` (change resolution), config reader.
- **Modified skills/docs**: `src/kit/skills/git-pr.md`, `src/kit/skills/_cli-fab.md`, `docs/specs/skills/SPEC-git-pr.md`.
- **No behavior change to**: PR title derivation (Step 3c step 2), Summary/Changes generation, ship-stage bookkeeping (Steps 4a–4c), the `fab impact` subcommand itself (still public; `git-pr.md` no longer calls it directly but it remains for other consumers).
- **Dependency note**: `fab pr-meta` invokes `git` and `gh` internally. `gh` is already a declared single-binary dependency (Constitution I). Failure to reach `gh` (no network, not authed) must degrade gracefully — blob URLs fall back to plain-text stage labels, never a hard error that breaks `/git-pr`.
