**Created**: 2026-06-01
**Affected memory**: none (implementation-only correction to skill source text; no spec-level behavior change)

## Non-Goals

- Changing the `fab` binary or any Go code — the binary signatures are already correct; this aligns skill text *to* them.
- Editing deployed copies under `.claude/skills/` — those are gitignored and regenerated by `fab sync` (constitution §V).
- Rewriting `SPEC-fab-new.md` / `SPEC-fab-draft.md` — inspection shows both already document the correct `<change>` form; only verification is in scope.
- Broadening the sweep to other skills — a full `grep` of `src/kit/skills/*.md` confirmed no other files carry the path form.

## Skills: fab-new and fab-draft status invocations

### Requirement: Status subcommands SHALL receive a change reference, not a path

In `src/kit/skills/fab-new.md` and `src/kit/skills/fab-draft.md`, every invocation of
`fab status add-issue`, `fab status set-change-type`, and `fab status advance` SHALL pass a
change reference (the captured change folder name `{name}`) as the first positional argument
— never a `.status.yaml` file path. This MUST match the installed binary's signatures:
`fab status set-change-type <change> <type>`, `fab status advance <change> <stage> [driver]`,
and `fab status add-issue <change> <id>`, as documented in `_cli-fab.md`.

#### Scenario: set-change-type uses the change reference

- **GIVEN** `fab-new.md` Step 6 instructs writing the inferred change type
- **WHEN** the corrected skill text is read
- **THEN** the command form is `fab status set-change-type {name} <type>`
- **AND** no `fab/changes/{name}/.status.yaml` path appears as the first argument

#### Scenario: advance uses the change reference

- **GIVEN** `fab-new.md` Step 9 (and `fab-draft.md` equivalent) advances intake to ready
- **WHEN** the corrected skill text is read

#### Scenario: add-issue uses the change reference

- **WHEN** the corrected skill text is read
- **THEN** the command form is `fab status add-issue {name} DEV-988`

#### Scenario: invocation executes against the installed binary without error

- **WHEN** an agent follows the corrected skill text and runs `fab status set-change-type {name} fix`
- **THEN** the command succeeds (exit 0) and writes `change_type: fix` to the change's `.status.yaml`

## Design Decisions

1. **Use `{name}` (folder name) as the change reference, not `{id}`**
   - *Why*: `{name}` is already captured from `fab change new` stdout at every call site in
     both skills; it is a valid `<change>` reference (full folder name) per `_cli-fab.md`, and
     keeps all six edits uniform with no new variable capture.
   - *Rejected*: `{id}` (4-char) — also valid, but not consistently in scope at each call site
     and would require deriving/capturing the ID, adding needless variance versus `{name}`.
