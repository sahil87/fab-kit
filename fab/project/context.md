# Project Context

## Tech Stack

- **Languages**: Go (the `fab`/`fab-kit`/`fab-go` binaries), Markdown (skills, specs, memory), Bash (helper scripts)
- **Content**: Specification-driven change management tooling, plus research analyses of SpecKit and OpenSpec and cross-cutting findings
- **Workflow**: The Fab kit drives changes through the six-stage pipeline (intake → apply → review → hydrate → ship → review-pr)

## Repository Structure

Monorepo with these top-level directories:

- `src/` — canonical sources: `src/kit/` (skills, templates, migrations — the kit content), `src/go/` (the `fab`/`fab-kit`/`fab-go` Go binaries)
- `docs/` — `docs/specs/` (pre-implementation design intent) and `docs/memory/` (post-implementation source of truth)
- `fab/` — project config (`fab/project/`), changes (`fab/changes/`), and the local backlog (`fab/backlog.md`)
- `scripts/` — Bash helper scripts
- `.claude/skills/` — deployed skill copies generated by `fab sync` (gitignored; never edited directly)

## Distribution

Kit content (skills, templates, migrations) is distributed to the system cache at `~/.fab-kit/versions/<version>/kit/`. User projects contain only `fab/` (project config, changes) and `.claude/skills/` (deployed copies). **Kit content is NOT copied into user projects.**

## Skills

Canonical skill sources live at `src/kit/skills/*.md` in the dev repo. The files in `.claude/skills/` are deployed copies generated by `fab sync` from the system cache.

> When modifying a skill, always update the source at `src/kit/skills/` — never edit only the deployed copy under `.claude/skills/`, or the change will be overwritten on the next sync.

## Migrations

When a change requires restructuring existing user data (config files, `.status.yaml`, archive layout, etc.), it MUST ship as a migration file in `src/kit/migrations/` — not as a new subcommand or ad-hoc script. Migrations are markdown instruction files applied by `/fab-setup migrations`.

See `docs/memory/distribution/migrations.md` for the format and versioning model.
