# Mechanical config.yaml upgrades — `fab config upgrade`

> Backlog detail doc — written 2026-07-08 after a `/fab-discuss` session on making
> config.yaml upgrades across fab-kit versions mechanical. All open decisions below are
> **resolved** (user-confirmed in that session). The work is split into three changes,
> intended to be picked up by other agents — each is independently shippable, in
> dependency order. The triggering bug (`setFabVersion` mashing config.yaml on every
> `fab upgrade-repo`) already shipped its stopgap fix separately: change `260708-yogn`,
> PR #473 — a targeted line splice. This effort deletes that function entirely.

## Goal

`fab/project/config.yaml` becomes a **sparse overrides file**, and `fab config upgrade`
reconciles it mechanically against a canonical, binary-owned reference on every kit
upgrade. Field categories:

- **A) user-overridden** → live YAML in the file
- **B) not overridden** → absent (inherited from defaults)
- **C) not overridden but worth advertising** → generated comment scaffold inside a
  managed fence

The design inverts comment ownership: comments become **regenerable generated output**
instead of precious hand-maintained user state — which retires the
masher-wipes-comments bug class (observed 2026-07-03 and 2026-07-08) at the root, and
replaces the fragile comment-backfill migration pattern (e.g. `2.13.1-to-2.13.2`)
going forward.

**Override cascade**: project (`fab/project/config.yaml`) > system
(`~/.fab-kit/config.yaml`) > built-in defaults (a Go table in the `fab` binary).
Per-field deep merge: maps merge per-key (existing `agent.tiers` precedent), **lists
replace** (never concatenate), scalars replace.

## The three changes

### Change 1 — reference table + `fab config reference` restructure (additive)

`fab config reference` **already exists** (change `6nke` — fab-go
`src/go/fab/internal/configref/`, ~257 lines) but as a text-template renderer
(`refData` struct → template): no per-field metadata, no scope, no `--json`. This
change restructures it into a **per-field metadata table** — the single source:

- Per field: `default`, `description` (drives the generated comment), `scope`
  (project / system / both), `advertise` (the C flag), `renamed_from` (mechanical
  rename carry-forward, so future field renames don't each need a hand-written
  migration).
- `fab config reference` renders the table as commented YAML (output should stay
  equivalent to today's); add `--json` for tooling.
- The spec `docs/specs/config.md` lands here — this is where the schema decisions
  are recorded.
- Obligations: `_cli-fab.md` update + tests (constitution: CLI changes), specs index.

### Change 2 — cascade resolution + visibility commands

- Three-layer per-field merge in fab's config loader (see cascade above).
- Scope enforcement: a project-scoped field in the system file is **ignored with a
  warning** (fail-open — config must never brick).
- `fab config show --origin` — effective config with per-field provenance (the
  `git config --show-origin` precedent). Non-negotiable companion: three layers
  without provenance is archaeology, and it surfaces typo'd overrides that today
  silently no-op.
- `fab config init --system` — writes `~/.fab-kit/config.yaml` scaffold containing
  ONLY `scope: system/both` fields, all commented. This is the user's answer to
  "what can I safely override at system level", generated from the same table so it
  can't drift.

### Change 3 — `fab config upgrade` + migration (the risky one; user-data restructure)

- The upgrader: live fields kept **verbatim**; unknown fields parked in a
  `# removed in X.Y.Z, your value was:` comment block (never silently deleted);
  renames carried via `renamed_from`; the managed fence regenerated byte-stable and
  idempotent (same discipline as `fab memory-index`).
- Move `fab_version` out of config.yaml to `fab/.fab-version` (decision 1 below);
  delete `setFabVersion`; ship the migration. After this, `fab config upgrade` is
  the **only** writer of config.yaml, ever.
- `fab upgrade-repo` auto-runs the upgrader (decision 4 below).
- Historical comment-backfill migrations are left alone; the pattern is retired
  going forward.
- Obligations: migration file (constitution: user-data restructure), `_cli-fab.md`,
  tests; `ResolveConfig`/`readFabVersion` (fab-kit) and the shim's kit resolution
  (`configref`/`kitpath`) move to `.fab-version` with a config.yaml fallback for one
  compat window.

## Resolved decisions (all user-confirmed 2026-07-08)

1. **`fab_version` → `fab/.fab-version`** — one-line plain text, committed, sibling to
   `.kit-migration-version`. Kept separate from it: deployed-kit version vs
   migration baseline diverge exactly when migrations are pending, which is when both
   values are needed distinctly. Readers get a config.yaml fallback for one compat
   window; the migration moves the value and deletes the key.
2. **Presence = intent** — any live field is an override even if its value equals the
   default. `fab config upgrade` never auto-removes live fields; B-hygiene is
   advisory only ("these fields equal current defaults — remove?" as a report line).
   Rationale: a value-diff classifier can't distinguish "deliberately pinned" from
   "never touched", and auto-dropping silently changes behavior when the default
   later moves.
3. **Fence contract** — see the example below. The `>>>`/`<<<` marker lines are
   byte-exact splice anchors; upgrade rewrites ONLY between them. Everything outside
   the fence is the user's, including their own comments on A fields.
4. **Auto-run** — `fab upgrade-repo` (fab-kit binary) shells out to the installed
   `fab config upgrade` after sync. Both binaries ship in one brew package, so
   binary/kit skew only occurs on explicit-version upgrades (acceptable; the fence
   stamp shows which kit rendered it). If the installed `fab` predates the
   subcommand → printed reminder fallback (fail-open; an upgrade must never break on
   the config step).
5. **System config path = `~/.fab-kit/config.yaml`** (co-located with the existing
   version cache; XDG `~/.config/fab-kit/` rejected).
6. **Scope taxonomy** — `agent.tiers` / `providers` = both; `project.*`,
   `source_paths`, `test_paths`, `true_impact_exclude`, `checklist` = project.
   Rationale: the system layer is restricted to preference-class fields; semantics-
   class fields stay repo-reproducible for teammates/CI.

## The fence — worked example

What config.yaml looks like after `fab config upgrade`:

```yaml
# fab/project/config.yaml — project overrides.
# Presence = override: any live field here wins over ~/.fab-kit/config.yaml
# and built-in defaults, even if its value equals the default.

project:
    name: fab-kit
    description: FAB Kit — Tooling for Specification-Driven Development

source_paths:
    - src/

# pin review to fable — sonnet missed doc-claim regressions   <- user comment, preserved
agent:
    tiers:
        review:
            model: claude-fable-5
            effort: xhigh

# >>> fab reference (kit 2.15.0) >>> -----------------------------------------
# Overridable fields you have NOT overridden, with current defaults.
# REGENERATED by `fab config upgrade` on every upgrade — edits inside this
# fence are overwritten. To override a field: move it ABOVE the fence
# and uncomment it.
#
# agent:
#     tiers:
#         doing:                    # apply-stage worker       [project|system]
#             provider: claude
#             model: claude-opus-4-8
#             effort: xhigh
#         fast:                     # cheap mechanical stages  [project|system]
#             provider: claude
#             model: claude-sonnet-5
#             effort: low
#
# providers:                        #                          [project|system]
#     codex:
#         session_command: codex --model {model} --effort {effort}
#
# checklist:                        #                          [project]
#     extra_categories: []
#
# true_impact_exclude: []           #                          [project]
# <<< end fab reference <<< ---------------------------------------------------

# removed in 2.16.0 (parked by fab config upgrade — delete when done):
#   checklist.legacy_mode: true
```

Generation rules the fence carries:

- Every scaffold block is **fully commented including parent keys** — a live `agent:`
  key over comment-only children is exactly what the old masher collapsed to
  `agent: null`.
- The scaffold **omits fields already overridden** above the fence (the user's
  `review` tier is not re-advertised) — it shows what you *could* override but
  haven't.
- The kit-version stamp in the BEGIN line makes staleness visible and enables a
  later `--check` drift mode (memory-index precedent).
- Removed-field parkings are appended **below** the fence exactly once — user
  territory, never regenerated away. No fence found in a legacy file → append one at
  the bottom.

## Context: why this design

- **Triggering bug**: `setFabVersion` (fab-kit `internal/init.go`) unmarshalled the
  whole config into `map[string]interface{}` and re-marshalled — destroying every
  comment, alphabetizing keys, 4-space-normalizing, and nulling comments-only keys —
  on every `fab init`/`fab upgrade-repo`. Wiped the migration-backfilled providers
  template permanently (migration never re-applies once `.kit-migration-version` is
  past it). Stopgap line-splice fix shipped as `260708-yogn` / PR #473; this effort
  deletes the function.
- Under this design no fab command ever writes config.yaml except the explicit,
  comment-aware `fab config upgrade` — single-writer by construction.
