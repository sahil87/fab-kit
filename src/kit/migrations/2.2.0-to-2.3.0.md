# Migration: 2.2.0 to 2.3.0

## Summary

Surface the new optional `agent.tiers` per-stage-model override to existing users.
A **tier** is a named `{model, effort}` agent profile; fab-kit owns a FIXED,
non-overridable stage→tier mapping and a default profile per tier, and resolves
the profile per pipeline stage at sub-agent dispatch time via
`fab resolve-agent <stage>`. The only thing a project overrides is **what each
tier means** (model + effort), under `agent.tiers` in `fab/project/config.yaml`.

This migration adds a **fully-commented** `agent.tiers` reference block under the
existing `agent:` block, documenting the fixed mapping, fab-kit's built-in
default profiles, and the override shape — all commented out. By default fab-kit
uses its built-in defaults; the user opts in by uncommenting/editing. The block
is documentation only — it changes no behavior until a user uncomments a tier.

This is config-only and additive: there is **no `.status.yaml` schema change** and
existing configs without `agent.tiers` are fully supported (fab-kit defaults
apply). See `docs/specs/stage-models.md` for the full design.

## Pre-check

1. Check if `fab/project/config.yaml` exists — if not, skip this migration
   entirely. Print: `Skipped: fab/project/config.yaml not present.`
2. Check if `fab/project/config.yaml` already contains an `agent:` block. If it
   does **not**, this migration is still applicable (it adds the `agent:` block
   with the commented `tiers` reference) — but most initialized projects have an
   `agent:` block with `spawn_command`.
3. Check if the config already contains a `tiers:` key under `agent:` **or** the
   `agent.tiers` reference comment marker `# agent.tiers — per-stage model`
   (the sentinel this migration writes). If either is present, skip — the block
   is already documented. Print: `Skipped: agent.tiers reference already present.`

## Changes

### 1. Add the fully-commented `agent.tiers` reference block

If `fab/project/config.yaml` exists and does not already contain an `agent.tiers`
block or the reference-comment sentinel:

1. Locate the `agent:` block. If an `agent:` block exists, insert the commented
   reference block immediately after the last existing field under `agent:`
   (typically after `spawn_command:`). If no `agent:` block exists, append a new
   `agent:` block at the top level with only the commented reference block under
   it (no `spawn_command` line — the binary falls back to its default spawn
   command when absent).
2. Insert exactly this block (all lines commented; preserve the leading `agent:`
   key's existing indentation — the comment lines sit at the same nesting depth
   as `spawn_command`):

   ```yaml
     # agent.tiers — per-stage model override (optional). A "tier" is a named
     # {model, effort} profile. fab-kit owns the FIXED, non-overridable stage→tier
     # mapping below; you override only WHAT EACH TIER MEANS (model + effort).
     # Omit any tier (or the whole tiers: block) to use fab-kit's built-in default.
     # Resolved per stage by `fab resolve-agent <stage>` at sub-agent dispatch time.
     # See docs/specs/stage-models.md.
     #
     # FIXED stage→tier mapping (fab-owned, NOT overridable — shown for reference):
     #   thinking: intake, review            (generative judgment)
     #   doing:    apply, review-pr, hydrate  (execution that must not err)
     #   fast:     ship                       (speed on near-mechanical work)
     #
     # fab-kit's built-in default profiles (today):
     #   thinking: { model: claude-opus-4-8,   effort: xhigh }
     #   doing:    { model: claude-opus-4-8,   effort: high  }
     #   fast:     { model: claude-sonnet-4-6, effort: low   }
     #
     # Override shape (uncomment + edit any tier; an omitted field inherits the
     # default for that tier; an empty model means "inherit the session model";
     # values are passed through verbatim — fab does no provider validation):
     # tiers:
     #   doing: { model: claude-sonnet-4-6, effort: medium }   # example: run doing cheaper
   ```

3. Print a note:
   ```
   Added commented-out agent.tiers reference block to fab/project/config.yaml.
   fab-kit uses its built-in tier defaults by default. To customize a tier's
   model/effort, uncomment the `tiers:` example under `agent:` and edit it.
   See docs/specs/stage-models.md.
   ```

## Verification

1. `fab/project/config.yaml` contains the commented `agent.tiers` reference block
   under the `agent:` block, including the `# agent.tiers — per-stage model`
   sentinel comment line.
2. The `tiers:` block remains **commented out** — `yq '.agent.tiers' fab/project/config.yaml`
   returns `null` (the block is documentation only; fab-kit's built-in defaults
   apply).
3. All other keys and their contents are unchanged — including `project`,
   `source_paths`, `checklist`, `fab_version`, `agent.spawn_command`, and (when
   present) `true_impact_exclude`, `test_paths`, `stage_hooks`, `branch_prefix`.
4. Re-running this migration is a complete no-op (the Pre-check sentinel trips
   and prints `Skipped: agent.tiers reference already present.`).
