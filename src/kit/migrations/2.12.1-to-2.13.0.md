# Migration: 2.12.1 to 2.13.0

## Summary

Restructure `fab/project/config.yaml` for **agent config v3 ‚Äî providers & role
tiers**. Three coordinated schema changes, all config-only:

1. **Providers extracted.** `agent.spawn_command` moves to a new top-level
   `providers.claude.session_command` (verbatim value move). Per-tier
   `spawn_command` values move to `providers.<name>.dispatch_command`, with the
   tier pointing at that provider by name. The command grammar now lives in a
   named `providers:` table instead of inline on `agent.spawn_command` / each tier.
2. **Five role tiers.** The `thinking`/`doing`/`fast` tier vocabulary becomes
   `default`/`operator`/`doing`/`review`/`fast`. Tier values become
   `{provider, model, effort}` (the command moved to `providers:`). A `thinking`
   override maps to `review` (its only dispatched stage); `doing`/`fast` overrides
   carry over field-by-field.
3. **`review_tools` retired.** The `review_tools` config block is removed. Its two
   live semantics (the outward-reviewer Codex‚ÜíClaude cascade and the
   `/git-pr-review` Copilot toggle) move to a `fab/project/code-review.md`
   `## Review Tools` prose section ‚Äî read by the agents that already load
   `code-review.md`. An absent section/key = all enabled (today's absent-key
   default, unchanged).

The load-bearing behavior is preserved: absence of a `dispatch_command` still
signals **native Agent-tool dispatch** (there is NO fallback to `session_command`),
and unset tier fields inherit from the project's `default` tier. See
`docs/specs/stage-models.md` for the full design and
`fab config reference` for the canonical schema.

This migration edits an existing config in place. It is idempotent
(sentinel-guarded) and additive-then-subtractive: it writes the new shape and
removes the retired keys only once. There is **no `.status.yaml` schema change**.

## Pre-check

1. Check if `fab/project/config.yaml` exists ‚Äî if not, skip this migration
   entirely. Print: `Skipped: fab/project/config.yaml not present.`
2. Check if the config already contains a top-level `providers:` key (the sentinel
   this migration writes). If present, skip ‚Äî the config is already on the v3
   shape. Print: `Skipped: providers: block already present (agent config v3).`

## Changes

Apply the following to `fab/project/config.yaml`. Preserve all unrelated keys
(`project`, `source_paths`, `test_paths`, `true_impact_exclude`, `checklist`,
`stage_hooks`, `branch_prefix`, `fab_version`) and their values verbatim.

### 1. Relocate `agent.spawn_command` ‚Üí `providers.claude.session_command`

1. If `agent.spawn_command` is present, create a top-level `providers:` block with
   a `claude` entry whose `session_command` is the **verbatim** value of the old
   `agent.spawn_command`, then delete `agent.spawn_command`.

   ```yaml
   providers:
     claude:
       session_command: '<the old agent.spawn_command value, verbatim>'
   ```

2. **Non-claude spawn command ‚Äî halt and ask.** If the old `agent.spawn_command`
   is a **templated** command (contains `{model}` or `{effort}`) or is otherwise
   not a plain `claude ‚Ä¶` invocation (e.g. a `codex ‚Ä¶` command), it cannot be
   auto-attributed to the `claude` provider. Relocate it under a placeholder
   provider name and STOP with a message asking the user to name the provider:

   ```yaml
   providers:
     UNNAMED_PROVIDER:   # ‚Üê rename to your provider (e.g. codex) and point the tiers at it
       session_command: '<the old value, verbatim>'
   ```

   Print:
   ```
   Your agent.spawn_command was not a plain `claude` command, so it was moved
   under providers.UNNAMED_PROVIDER.session_command. Rename UNNAMED_PROVIDER to
   your provider (e.g. codex), set its dispatch_command if you dispatch stages via
   it, and point the relevant agent.tiers at it (provider: <name>). See:
   fab config reference.
   ```

3. If `agent.spawn_command` was **absent**, add the fab-kit default claude
   provider so the config is explicit:

   ```yaml
   providers:
     claude:
       session_command: 'claude --dangerously-skip-permissions -n "$(basename "$(pwd)")"'
   ```

### 2. Map `agent.tiers` to the five-tier shape

For each existing entry under `agent.tiers`:

1. **`doing` / `fast`** ‚Äî carry over field-by-field into the new `doing` / `fast`
   tiers. Add `provider: claude` if the tier sets a `model` or `effort` (documented
   style ‚Äî provider written explicitly), unless a `spawn_command` move in step 3
   dictates a different provider.
2. **`thinking`** ‚Äî map to the new **`review`** tier (its only dispatched stage),
   field-by-field.
3. Any per-tier **`spawn_command`** value moves to
   `providers.<name>.dispatch_command` (step 3 below); the tier keeps only
   `{provider, model, effort}`.
4. If `agent.tiers` was absent, leave it absent ‚Äî fab-kit's built-in defaults apply
   (per-field inheritance). Do NOT synthesize a five-tier block for a project that
   never overrode a tier.

### 3. Move per-tier `spawn_command` ‚Üí `providers.<name>.dispatch_command`

For each tier that carried a `spawn_command`:

1. Choose a provider name for it (derive from the command, e.g. `codex` for a
   `codex ‚Ä¶` command; otherwise `UNNAMED_PROVIDER` and ask the user to rename, as
   in step 1.2).
2. Add `dispatch_command: '<the tier spawn_command value, verbatim>'` under that
   provider in `providers:` (alongside any `session_command`).
3. Set `provider: <name>` on the tier and remove its `spawn_command`.

### 4. Remove `review_tools`

1. If a `review_tools` block is present:
   - If **every** key is `true` (or the block is empty), it is a no-op ‚Äî **delete
     it silently** (no `code-review.md` change needed; absent = enabled).
   - If **any** key is explicitly `false`, delete the block AND seed a
     `fab/project/code-review.md` `## Review Tools` section recording the disabled
     tools. Create `code-review.md` if absent. Insert (only the false keys need an
     entry; the others default to enabled):

     ```markdown
     ## Review Tools

     <!-- Retired review_tools toggles. Absent = enabled; list a tool as false to
          disable. codex/claude = the review-stage outward Codex‚ÜíClaude cascade;
          copilot = the /git-pr-review Copilot request. -->

     - <tool>: false
     ```
2. Print a note when a `code-review.md` seed was written:
   ```
   Moved disabled review_tools entries to fab/project/code-review.md ¬ß Review Tools.
   ```

## Verification

1. `fab/project/config.yaml` has a top-level `providers:` block with at least a
   `claude` (or user-named) provider carrying a `session_command`; `yq '.providers'`
   is non-null.
2. `agent.spawn_command` is gone: `yq '.agent.spawn_command' fab/project/config.yaml`
   is `null`.
3. `review_tools` is gone: `yq '.review_tools' fab/project/config.yaml` is `null`.
4. Any `agent.tiers` keys are within the five role tiers
   (`default`/`operator`/`doing`/`review`/`fast`); no `thinking` tier remains; no
   tier carries a `spawn_command` (`yq '.agent.tiers.*.spawn_command'` is empty).
5. Every per-tier or whole-session command relocated preserves its original value
   verbatim (no re-quoting or reformatting of the command string).
6. When a `review_tools` key was explicitly `false`, `fab/project/code-review.md`
   has a `## Review Tools` section listing the disabled tool(s); an all-true block
   left `code-review.md` unchanged.
7. `fab config reference` still parses (the reference reflects the v3 schema) and
   `fab resolve-agent apply` resolves without error.
8. Re-running this migration is a complete no-op (the Pre-check `providers:`
   sentinel trips and prints `Skipped: providers: block already present ‚Ä¶`).
