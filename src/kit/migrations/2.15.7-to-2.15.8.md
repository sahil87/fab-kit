# Migration: 2.15.7 to 2.15.8

## Summary

Sweep **every worktree's** `.claude/settings.local.json` for stale `fab hook`
entries, cleaning the copies the prior settings migrations
(`2.10.1-to-2.11.0.md`, `2.13.6-to-2.14.0.md` §1) could never reach.

The `fab hook` command family was **removed outright** in 2.14.0 (agent-state
divestment, `ioku`) with **no deprecation shim**. Any hook entry still
registered in a `.claude/settings.local.json` now **errors every time it fires**:
Claude Code invokes `fab hook <x>`, cobra prints an *unknown command "hook" for
"fab"* message on stderr, and the user sees a non-blocking
`PostToolUse:{Write,Edit}` (or `SessionStart`/`Stop`/`UserPromptSubmit`)
hook-error warning on every single file write / session event. Harmless but
relentless noise.

**Why the two prior migrations don't cover it.** Both `2.10.1-to-2.11.0.md` (the
two `fab hook artifact-write` PostToolUse entries) and `2.13.6-to-2.14.0.md` §1
(the three session-scoped entries) remove these entries, but each edits **only
the checkout in which `/fab-setup migrations` runs**. Migration applicability is
gated on `fab/.kit-migration-version` — a **committed, repo-wide** file — so once
a migration runs in one checkout and the version bump merges, **no sibling
checkout ever re-runs it**. Meanwhile `.claude/settings.local.json` is
**gitignored, per-checkout** state: pre-2.14.0, `fab sync`'s hook-sync step
minted the hook entries into every checkout it ran in (and `wt create` → `wt
init` runs `fab sync` in every new worktree). Result: every worktree synced
before 2.14.0 carries its own stale copy that no version-gated migration will
ever touch again.

**Why the main checkout is the live poison.** Claude Code resolves project
settings **through worktrees to the main repository root**
(`.claude/settings.local.json` is read at the root of the git repository,
resolved through worktrees to the main checkout, so one file covers sessions
started in any subdirectory or worktree). A stale **main checkout** therefore
poisons **every** worktree session — including worktrees created *after* 2.14.0
whose own settings files are clean. (Pre-v2.1.211 Claude Code read the starting
directory's file directly, so the per-worktree copies matter too; hooks are
re-read live by a file watcher, so cleanup takes effect without a session
restart.) This is why the sweep MUST cover the main checkout — which is the first
entry reported by `git worktree list --porcelain`.

This restructures user-owned data (`.claude/settings.local.json`), so it ships as
a migration rather than an ad-hoc script (constitution + code-quality rule),
following the `settings.local.json`-editing precedent in `2.10.1-to-2.11.0.md`
§1 / `0.46.0-to-1.1.0.md` §1 and the **worktree-sweep** precedent in
`2.13.6-to-2.14.0.md` §2 (which enumerated worktrees via `git worktree list
--porcelain` to delete `.fab-runtime.yaml` everywhere) — this change applies the
same worktree discipline to the settings edits its own §1 (and 2.11.0's) missed.
Every step is sentinel-guarded and idempotent — re-running is a complete no-op.

There is **no `.status.yaml` schema change**, **no binary capability pre-check**
(pure prompt / JSON-edit logic — no new flag or command is needed), **no `fab/`
data change**, and **no commit** (unlike `2.15.1-to-2.15.2.md`, which committed
`fab/.fab-version`): the edited `.claude/settings.local.json` files are
gitignored, so nothing here lands in git.

`src/kit/VERSION` is bumped `2.15.7` → `2.15.8` (patch — a pure fix with no
schema or binary change; migration name FROM=released TO=next).

## Pre-check

The **target set** — a hook action is stale when its `command` either:

- **starts with the prefix `fab hook `** (a prefix match, **NOT** an enumeration
  of the four known subcommands — the entire command family is gone, so *any*
  `fab hook <x>` errors), **or**
- matches the **legacy script-shim forms** carried by the prior migrations' target
  sets: `bash "$CLAUDE_PROJECT_DIR"/fab/.kit/hooks/on-<script>.sh` **or**
  `bash fab/.kit/hooks/on-<script>.sh` (the `on-*.sh` scripts no longer exist, so
  these also fail if fired).

1. **Not a git repository** — handle only the current directory (apply the Changes
   step to `<current-dir>/.claude/settings.local.json`), and **skip the sibling
   sweep** (mirror `2.13.6-to-2.14.0.md` §2).
2. **Enumerate all worktrees** via `git worktree list --porcelain`. The **main
   checkout is the first entry**, so the sweep inherently covers it — and it MUST,
   since it is the live poison (settings resolve through worktrees to the main
   root, per the Summary).
3. **Sentinel (idempotency).** If **no** `<worktree>/.claude/settings.local.json`
   contains a target entry (under any event), skip this migration entirely. Print:
   `Skipped: no stale fab hook entries found in any worktree.`
   (Re-running is a complete no-op.)

## Changes

For **each worktree path** reported by `git worktree list --porcelain` (or, in a
non-git directory, the current directory only) whose `.claude/settings.local.json`
exists and carries at least one target entry:

1. Read the file and parse it as JSON. In **every event array** under `hooks`
   (any event — `PostToolUse`, `SessionStart`, `Stop`, `UserPromptSubmit`, etc.),
   drop every `{"type": "command", "command": "<cmd>"}` action whose `<cmd>` is in
   the target set above (prefix `fab hook ` **or** either legacy `on-*.sh` shim
   form). An entry that mixes a target action with an unrelated custom command in
   the same `hooks[]` array SHALL have **only the target action removed**,
   preserving the custom command (mirror the `2.10.1-to-2.11.0.md` §1 /
   `0.46.0-to-1.1.0.md` §1 preserve-non-fab-hooks discipline). Remove the entry
   entirely when its `hooks[]` becomes empty. If an event array becomes empty,
   leave it as an empty array (or omit the key) — **do not delete the `hooks`
   object**.
2. Preserve every non-hook top-level key (`permissions`, `model`, …) **verbatim**.
3. Use an **atomic write** (write to a temp file in the same directory, then
   `rename` to `.claude/settings.local.json`) to prevent partial writes if the
   user has the file open in an editor.
4. Print one line per cleaned worktree:
   `Removed stale fab hook entries from <worktree-path>/.claude/settings.local.json.`

Worktrees whose `.claude/settings.local.json` is absent or carries no target
entry are skipped silently — this makes the per-worktree step idempotent and
re-run-safe.

Example of the stale shape being removed (as found in the wild — the full pre-2.14.0
entry set: the two `artifact-write` PostToolUse rows and the three session events):

```json
{
  "hooks": {
    "PostToolUse": [
      { "matcher": "Write", "hooks": [ { "type": "command", "command": "fab hook artifact-write" } ] },
      { "matcher": "Edit",  "hooks": [ { "type": "command", "command": "fab hook artifact-write" } ] }
    ],
    "SessionStart":     [ { "matcher": "", "hooks": [ { "type": "command", "command": "fab hook session-start" } ] } ],
    "Stop":             [ { "matcher": "", "hooks": [ { "type": "command", "command": "fab hook stop" } ] } ],
    "UserPromptSubmit": [ { "matcher": "", "hooks": [ { "type": "command", "command": "fab hook user-prompt" } ] } ]
  }
}
```

After the sweep, an entry whose `hooks[]` held only a target action is gone; an
emptied event array is left empty (or its key omitted); the `hooks` object and
all non-hook top-level keys remain.

## Verification

1. No `.claude/settings.local.json` in **any worktree** reported by
   `git worktree list --porcelain` (including the main checkout, the first entry)
   contains a command starting with `fab hook ` or a legacy
   `bash …/fab/.kit/hooks/on-*.sh` script-shim command, under **any** event. In a
   non-git directory, the same holds for the current directory's file.
2. Any non-fab hook entries and non-hook top-level keys (`model`, `permissions`,
   …) are preserved verbatim; the `hooks` object is not deleted.
3. Re-running this migration is a complete no-op: the Pre-check sentinel trips
   (`Skipped: no stale fab hook entries found in any worktree.`) and nothing is
   changed.
