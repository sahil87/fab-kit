# Migration: 2.11.0 to 2.12.0

## Summary

Surface the new optional **per-tier `spawn_command`** to existing users. A tier
(`agent.tiers.<tier>`) already carries a `{model, effort}` profile resolved per
pipeline stage by `fab resolve-agent <stage>`; this release widens the tier
profile with an opt-in `spawn_command` — the **cross-harness stage-dispatch**
knob. When a resolved tier carries a `spawn_command`, `fab resolve-agent` emits a
third `spawn=<command>` line (with `{model}`/`{effort}` substituted) and that
tier's stages are dispatched by RUNNING the command (e.g. `codex exec …`) instead
of a native Agent-tool sub-agent; when absent, native Agent-tool dispatch (the
default) is unchanged.

The load-bearing semantic: this is **INDEPENDENT of `agent.spawn_command`** (which
opens whole agent *sessions* for `fab operator` / `fab batch` / `fab spawn-command`)
— there is **NO fallback** from a tier to `agent.spawn_command`. The absence of a
resolved tier `spawn_command` is itself the "native dispatch" signal. (Mental
model: `agent.spawn_command` hires an employee; a tier `spawn_command` outsources
one task.)

This migration adds a **short, fully-commented** reference note under the existing
`agent:` block announcing the field and pointing at `fab config reference` for the
canonical documentation. It is config-only and additive: there is **no
`.status.yaml` schema change**, and existing configs without a tier `spawn_command`
are fully supported (native dispatch applies). It needs no binary capability
pre-check for the comment itself, though the field it documents requires the
widened binary — which is the version-gating point of shipping it in this slot.
Follows the `2.2.0-to-2.3.0.md` precedent (comment-sentinel idempotency, insert
under `agent:`). See `docs/specs/stage-models.md` for the full design.

## Pre-check

1. Check if `fab/project/config.yaml` exists — if not, skip this migration
   entirely. Print: `Skipped: fab/project/config.yaml not present.`
2. Check if the config already contains the tier-`spawn_command` reference-comment
   sentinel `# agent.tiers.<tier>.spawn_command` (the marker this migration
   writes). If present, skip — the block is already documented. Print:
   `Skipped: agent.tiers spawn_command reference already present.`

## Changes

### 1. Add the commented per-tier `spawn_command` reference note (sentinel-guarded)

If `fab/project/config.yaml` exists and does not already contain the
tier-`spawn_command` reference-comment sentinel:

1. Locate the `agent:` block. If an `agent:` block exists, insert the commented
   note immediately after the last existing line under `agent:` (typically after
   the `spawn_command:` line or the existing commented `agent.tiers` reference
   block). If no `agent:` block exists, append a new `agent:` block at the top
   level with only the commented note under it (no `spawn_command` line — the
   binary falls back to its default spawn command when absent).
2. Insert exactly this block (all lines commented; preserve the leading `agent:`
   key's existing indentation — the comment lines sit at the same nesting depth as
   `spawn_command`):

   ```yaml
     # agent.tiers.<tier>.spawn_command — per-stage CLI dispatch (optional, opt-in).
     # PRESENT on a tier → that tier's stages are dispatched by RUNNING this command
     # (cross-harness, e.g. codex); ABSENT → native Agent-tool dispatch (default).
     # INDEPENDENT of agent.spawn_command above (which opens whole agent SESSIONS):
     # there is NO fallback from a tier to agent.spawn_command. {model}/{effort}
     # placeholders are substituted at resolve time (the {model} always the FULL
     # model ID, never an alias). See: fab config reference.
   ```

3. Print a note:
   ```
   Added the commented per-tier spawn_command reference note to
   fab/project/config.yaml. It documents the optional cross-harness stage-dispatch
   opt-in; fab-kit uses native Agent-tool dispatch by default. To customize, add a
   spawn_command under a tier in agent.tiers. See: fab config reference.
   ```

## Verification

1. `fab/project/config.yaml` contains the commented per-tier `spawn_command`
   reference note under the `agent:` block, including the
   `# agent.tiers.<tier>.spawn_command` sentinel comment line.
2. The note remains **commented out** — it changes no behavior. `yq '.agent.tiers'
   fab/project/config.yaml` is unchanged by this migration (still `null` unless the
   user had already configured tiers).
3. All other keys and their contents are unchanged — including `project`,
   `source_paths`, `checklist`, `fab_version`, `agent.spawn_command`, any existing
   commented `agent.tiers` reference block, and (when present) `true_impact_exclude`,
   `test_paths`, `stage_hooks`, `branch_prefix`.
4. Re-running this migration is a complete no-op (the Pre-check sentinel trips and
   prints `Skipped: agent.tiers spawn_command reference already present.`).
