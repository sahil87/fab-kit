# Migration: 2.4.2 to 2.5.0

## Summary

Add the new optional `.status.yaml` `summary:` field to in-flight changes. The
`summary:` field is a per-change one-line log summary — the C-lite source line
the FKF `log.md` generator (a later FKF change) joins with git history to emit a
descriptive, conflict-free memory changelog (see `docs/specs/fkf.md` §6.3).
Because each change touches only *its own* `.status.yaml`, the summary has zero
merge-conflict surface.

This is a `.status.yaml` **schema change** (a new optional string field modeled on
`change_type_source`: `yaml:"summary,omitempty"`), so per the project's
data-migration rule it ships with this migration. The migration adds `summary: ""`
to in-flight `fab/changes/*/.status.yaml`, inserted immediately before
`last_updated` (the position the binary's writer uses). It is **idempotent** — a
file that already carries a `summary:` key is skipped.

Graceful absence: the field is `omitempty`, so a change with no `summary` is fully
valid and the generator falls back to the change slug. Seeding `summary: ""` into
in-flight files documents the field for humans and matches the backlog's explicit
"adding the field to in-flight changes" scope.

This migration does NOT wire any stage to auto-populate `summary` (that is a later
FKF change) and makes no `fab memory-index` / `log.md` changes.

## Pre-check

1. Confirm `fab/changes/` exists at the project root — if not, this project has no
   in-flight changes and the migration is a complete no-op. Print:
   `Skipped: fab/changes/ not present.`

## Changes

### 1. Walk every in-flight change folder

For each `fab/changes/*/.status.yaml` (the single-level glob already excludes
`fab/changes/archive/**` — archived changes are historical and SHALL NOT be
touched; state this explicitly):

1. **Idempotency check**: if the `.status.yaml` already contains a top-level
   `summary:` key, skip it — re-running is a no-op. Print:
   `Skipped: {folder} (summary already present).`

2. **Otherwise**: insert a `summary: ""` line into the top-level mapping,
   positioned immediately **before** the `last_updated:` key (mirroring the
   binary's `insertKey` position, between `prs:`/`true_impact:` and
   `last_updated:`). If `last_updated:` is somehow absent, append `summary: ""` at
   the end of the top-level mapping. Preserve all other keys, values, comments,
   and indentation. Write the file atomically (temp file + rename).

   Print: `Added summary: "" to {folder}.`

## Changes — Part 2: migrate the `docs/memory/` tree onto FKF (the cutover)

The 2.5.0 release lands the full FKF tooling (`docs/specs/fkf.md`): the
`.status.yaml` `summary:` field (Part 1), `fab memory-index` emitting per-folder
`log.md` + stamping FKF frontmatter, the doc skills authoring FKF, and the
`log.seed.md` seed-merge. This part moves the **existing** `docs/memory/` tree
onto that format. It is a one-time, **idempotent** agent-driven conversion (FKF
§10 enumerates the same six parts); a project with no `docs/memory/` directory
skips it entirely. The conversion runs **per topic file** under
`docs/memory/{domain}/*.md` (excluding the reserved `index.md` / `log.md` /
`log.seed.md`), in this order, so it is safe to re-run (a fully-converted tree is
a no-op):

1. **Seed the per-folder `log.seed.md`** (FKF §6 seed-merge — *do this first,
   before stripping*). For each file's trailing `## Changelog` table, convert every
   data row `| {YYMMDD-XXXX-slug} | {YYYY-MM-DD} | {summary} |` into one §6.2 seed
   entry in that **folder's** `docs/memory/{domain}/log.seed.md`:
   `- **Update** [{base}](/{domain}/{base}.md) — {summary} ({XXXX})` under a
   `## {YYYY-MM-DD}` heading (the row's own Date column — pre-FKF authored dates,
   independent of git), date-grouped newest-first. The change-id in parens is the
   **bare 4-char id** (the row's `Change`-column prefix `{YYMMDD-XXXX-...}` → `XXXX`)
   — matching what `fab memory-index` emits for attributed commits. The summary cell
   is preserved **verbatim** (DECISION b — faithful pre-FKF history), with two
   mechanical fixups: unescape table-escaped pipes (`\|` → `|`, since the entry is a
   list item, not a table cell) and apply the link rewrite of step 2 to any
   memory↔memory link inside it. The seed is a **curated input** `fab memory-index`
   reads and merges beneath the git projection — never the generated output, so the
   single-writer discipline holds. **Idempotent**: regenerate the seed from the
   changelog source; once changelogs are stripped (step 4) a re-run finds no rows
   and leaves the existing seed untouched.
2. **Convert memory↔memory links to bundle-relative** (FKF §7). Same-domain
   `]({file}.md)` / `](./{file}.md)` → `](/{this-domain}/{file}.md)` (only when
   `{file}.md` is a real topic file in that domain — illustrative example links are
   left alone); cross-domain `](../{domain}/{file}.md)` → `](/{domain}/{file}.md)`
   **only** when `{domain}` is a real memory domain. Preserve `#anchors`. Links OUT
   of the bundle (`../specs/…`, `../../README.md`, `../../../src/…`, external URLs)
   stay repo-relative/absolute-URL — unchanged. Already-bundle-relative `](/…)` links
   are left as-is (idempotent).
3. **Stamp `type: memory` frontmatter** (FKF §3.1) on every topic file missing it,
   alongside the existing `description:`. Files that already carry `type: memory`
   are untouched (no double-stamp). Reserved files are exempt.
4. **Strip the `## Changelog` section** (heading through EOF — it is the trailing
   section) from every topic file. A file with no `## Changelog` is a no-op.
5. **Regenerate** with `fab memory-index` (the 2.5.0 binary — the seed-merge is NOT
   in 2.4.2): it rewrites every `index.md`, regenerates each `log.md` merging the
   `log.seed.md` seed beneath the git projection, and stamps `fkf_version` on the
   root index. Confirm `fab memory-index --check` exits **0 or 1** (never 2 — a
   born-FKF tree is provably never destructive-loss), and that **zero** memory↔memory
   `](/…)` links dangle (every target resolves to an extant file under `docs/memory/`).

### Verification — Part 2

1. No topic file retains a trailing `## Changelog` section heading (inline
   backtick-wrapped `` `## Changelog` `` prose references are fine).
2. Every topic file carries `type: memory`.
3. Every `docs/memory/{domain}/log.seed.md` holds the preserved pre-FKF entries; the

4. `fab memory-index --check` exits 0/1; a dangling-link sweep finds no broken

5. Re-running the conversion is a complete no-op (no double-strip, no doubled
