# Migration: 2.13.6 to 2.14.0

## Summary

Two cleanups for the agent-state divestment: (1) remove the three
session-scoped hook entries — on `SessionStart`, `Stop`, and `UserPromptSubmit`
— from an existing project's `.claude/settings.local.json`, in **both** forms:
the current inline `fab hook {session-start,stop,user-prompt}` commands **and**
the older `bash …/fab/.kit/hooks/on-{session-start,stop,user-prompt}.sh` script
shims; and (2) delete the now-unread `.fab-runtime.yaml` and its
`.fab-runtime.yaml.lock` sibling from every worktree of the current repository.

fab no longer PRODUCES agent active/idle lifecycle state. Those three hooks
wrote `.fab-runtime.yaml` `_agents` entries that `fab pane map` / `fab pane
send` / `fab pane capture` read to gate keystroke sends and populate the Agent
column. That whole producer subsystem — the hook writes, the throttled GC
sweep, the grandparent PID walker (`internal/proc`), the runtime file
(`internal/runtime` / `.fab-runtime.yaml`), and the `_agents` matching in
`internal/pane` — was deleted. fab is now a pure CONSUMER of a shared tmux
pane-option convention: the pane commands read `@rk_agent_state`
(`"<state>:<epoch_seconds>"`, `state ∈ active | waiting | idle`), written by
run-kit's `rk agent-setup` global agent-harness hooks. Keeping the removed
hooks registered would fire writes nothing reads (dead weight outside tmux) and
tracks only Claude Code (blind to codex/copilot/gemini) — exactly what the
divestment removes.

Concretely this migration drops, on each of the three events (`SessionStart`,
`Stop`, `UserPromptSubmit`), any entry whose command is the inline
`fab hook {session-start,stop,user-prompt}` form **or** the legacy
`bash "$CLAUDE_PROJECT_DIR"/fab/.kit/hooks/on-{session-start,stop,user-prompt}.sh`
/ `bash fab/.kit/hooks/on-{session-start,stop,user-prompt}.sh` script-shim form.
Both forms are removed because nothing rewrites or re-mints them any more: the
`fab hook` command family (including the `sync` step that used to register and
rewrite these entries) was removed outright in 2.14.0, so a project still
carrying either the inline commands or the legacy script shims would otherwise
keep them forever — and, worse, they would now *error* if fired (the
subcommand no longer exists). The `artifact-write` PostToolUse rows are already
gone (removed by `2.10.1-to-2.11.0`); this migration leaves any non-fab hook and
every non-hook top-level key (`model`, `permissions`, …) untouched.

With the producer subsystem gone, the on-disk `.fab-runtime.yaml` file (and its
`.fab-runtime.yaml.lock` flock sibling) is dead: no code writes it anymore and
no code reads it. Unlike the `1.4.0-to-1.5.0` clean-slate delete (where the file
re-populated on the next hook event), these files will now stay gone — nothing
re-creates them. This migration deletes both, across every worktree of the
current repository, following the worktree-sweep precedent in
`1.4.0-to-1.5.0.md` (which enumerated worktrees via `git worktree list
--porcelain` and deleted `.fab-runtime.yaml` from each).

This restructures user-owned data (`.claude/settings.local.json` and the
worktree runtime files), so it ships as a migration rather than an ad-hoc
script (constitution + code-quality rule), following the
`settings.local.json`-editing precedent in `2.10.1-to-2.11.0.md` (which removed
the `artifact-write` entries the same way) and `0.46.0-to-1.1.0.md` §1, and the
runtime-file worktree-sweep precedent in `1.4.0-to-1.5.0.md`. There is **no
`.status.yaml` schema change**. Every step is sentinel-guarded and idempotent —
re-running is a complete no-op.

The `fab hook` command family was **removed outright** in 2.14.0 — there is no
deprecation shim period. Any remaining registered `fab hook
stop|user-prompt|session-start` entry that a hand-edited (or un-migrated)
settings file still carries will now **error if fired**: the subcommand no
longer exists, so Claude Code invoking `fab hook <x>` gets a cobra *unknown
command* message on stderr and a non-zero exit. That is the honest consequence
of removing the commands without a shim, and it is exactly what this migration
prevents by deleting the entries. `fab sync` no longer registers or rewrites any
hook entry (the whole `fab hook sync` step was removed), so a fresh project
never grows these entries in the first place; this migration backfills the
removal of both the inline `fab hook …` form and the legacy `on-*.sh` script
shims for projects already on fab-kit.

## Pre-check

1. Check if `.claude/settings.local.json` exists — if not, skip this migration
   entirely. Print: `Skipped: .claude/settings.local.json not present.`
2. Parse the file's `hooks` object. For each event in `SessionStart`, `Stop`,
   `UserPromptSubmit`, the target commands are the inline form
   (`fab hook session-start` / `fab hook stop` / `fab hook user-prompt`
   respectively) **and** the two legacy script-shim forms for that event's
   `on-*.sh` script — `bash "$CLAUDE_PROJECT_DIR"/fab/.kit/hooks/on-<script>.sh`
   and `bash fab/.kit/hooks/on-<script>.sh` (where `<script>` is
   `session-start`, `stop`, or `user-prompt`). If no entry under any of the three
   events has a hook whose `command` matches any of that event's target commands,
   skip §1 — the entries are already gone. Print:
   `Skipped: session-scoped hook entries already removed.`

The settings pre-check gates only §1. §2 (runtime-file deletion) is
independently guarded by file presence — it runs regardless of the settings
outcome and is a no-op when the runtime files are already absent.

## Changes

### 1. Remove the three session-scoped hook entries (sentinel-guarded)

If `.claude/settings.local.json` exists and its `hooks` object contains at least
one of the target entries above:

1. Read the file and parse it as JSON.
2. For each of the three events — `SessionStart` (script `on-session-start.sh`),
   `Stop` (script `on-stop.sh`), `UserPromptSubmit` (script `on-user-prompt.sh`)
   — the event's **target command set** is: the inline
   `fab hook <session-start|stop|user-prompt>` form **and** both legacy
   script-shim forms, `bash "$CLAUDE_PROJECT_DIR"/fab/.kit/hooks/<script>` and
   `bash fab/.kit/hooks/<script>`. Drop from `hooks[event]` every
   `{"type": "command", "command": "<cmd>"}` action whose `<cmd>` is in that
   event's target command set. An entry that mixes a target command with an
   unrelated custom command in the same `hooks[]` array SHALL have only the
   target action removed, preserving the custom command (mirror the
   `2.10.1-to-2.11.0.md` §1 / `0.46.0-to-1.1.0.md` §1 preserve-non-fab-hooks
   discipline). Remove the entry entirely when its `hooks[]` becomes empty. If an
   event array becomes empty, leave it as an empty array (or omit the key) — do
   not delete the `hooks` object.
3. Leave every other event (any `PostToolUse`, `PreToolUse`, etc.) and any
   non-hook top-level key (`model`, `permissions`, …) unchanged.
4. Use an atomic write (write to a temp file in the same directory, then
   `rename` to `.claude/settings.local.json`) to prevent partial writes if the
   user has the file open in an editor.
5. Print a note:
   ```
   Removed the session-scoped hook entries (SessionStart/Stop/UserPromptSubmit,
   both the inline fab hook commands and any legacy on-*.sh script shims) from
   .claude/settings.local.json. fab no longer produces agent active/idle state —
   the fab pane commands now read the @rk_agent_state tmux pane-option convention
   written by run-kit's rk agent-setup.
   ```

### 2. Delete the now-unread runtime files across all worktrees (sentinel-guarded)

The `.fab-runtime.yaml` state file and its `.fab-runtime.yaml.lock` flock
sibling are no longer written or read by any code. Delete both from every
worktree of the current repository, following the worktree-sweep precedent in
`1.4.0-to-1.5.0.md` §1–§2. Deletion is guarded by file presence — a missing
file is skipped silently, so this step is idempotent and re-run-safe.

1. **Current worktree**: at the repo root of the current worktree, if
   `.fab-runtime.yaml` exists, delete it. If `.fab-runtime.yaml.lock` exists,
   delete it too. Print `Removed: .fab-runtime.yaml at <repo-root> (agent-state
   production divested; nothing re-creates it).` when at least one file was
   deleted; otherwise print nothing for this worktree.
2. **Sibling worktrees**: if the current directory is a git repository, run
   `git worktree list --porcelain` to enumerate every worktree. For each
   worktree path other than the current one, delete `<worktree-path>/.fab-runtime.yaml`
   and `<worktree-path>/.fab-runtime.yaml.lock` if they exist, printing
   `Removed: .fab-runtime.yaml at <worktree-path>.` for each worktree where at
   least one file was deleted. Skip silently where neither exists. If the current
   directory is not a git repository, handle only the current worktree (step 1)
   and skip the sibling sweep.

## Verification

1. `.claude/settings.local.json` `hooks` no longer contains any entry whose
   command is `fab hook session-start`, `fab hook stop`, or
   `fab hook user-prompt`, nor any legacy
   `bash …/fab/.kit/hooks/on-{session-start,stop,user-prompt}.sh` script-shim
   command (under any event).
2. Any non-fab hook entries and non-hook top-level keys (`model`, `permissions`,
   …) are preserved verbatim; any `PostToolUse` entries are untouched.
3. `fab sync` run afterward does not touch `.claude/settings.local.json` hooks
   at all — the `fab hook` command family (including the sync step) was removed
   in 2.14.0, so nothing re-adds the three session entries or rewrites the
   removed legacy scripts back in. (`fab hook sync` itself no longer exists;
   invoking it, or any `fab hook <x>`, is a cobra unknown-command error.)
4. `.fab-runtime.yaml` and `.fab-runtime.yaml.lock` do NOT exist at the current
   worktree's repo root, nor at any worktree reported by
   `git worktree list --porcelain`. Nothing re-creates them (the producer
   subsystem is gone).
5. Re-running this migration is a complete no-op: §1's Pre-check sentinel trips
   (`Skipped: session-scoped hook entries already removed.`) and §2 finds the
   runtime files already absent and deletes nothing.
