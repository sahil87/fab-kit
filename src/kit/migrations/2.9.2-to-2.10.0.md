# Migration: 2.9.2 to 2.10.0

## Summary

Surface the new `fab config reference` command to existing projects by adding a
one-line pointer comment to the top of `fab/project/config.yaml`:

```yaml
# Full reference of all available options: fab config reference
```

`fab config reference` prints a fully-commented reference `config.yaml`
documenting every available option — both binary-consumed keys and
skill-consumed keys — generated from the binary's own constants (so its default
values cannot drift). New projects get this pointer line from the scaffold; this
migration backfills it for projects already on fab-kit, so an existing config
also names the discovery command.

This is config-only and additive, in the same shape as `1.9.1-to-1.9.2`
(`true_impact_exclude`), `2.2.0-to-2.3.0` (`agent.tiers`), and `2.7.1-to-2.8.0`
(`test_paths`): there is **no `.status.yaml` schema change** and **no Go binary
change** — the migration only adds a comment line. It needs **no binary
capability pre-check** (unlike the `2.5.5-to-2.6.0` / `2.6.6-to-2.7.0`
re-baselines): the pointer is a plain comment and `fab config reference` is a
new command that requires no project-file change to work. It is sentinel-guarded
and idempotent — re-running is a complete no-op.

## Pre-check

1. Check if `fab/project/config.yaml` exists — if not, skip this migration
   entirely. Print: `Skipped: fab/project/config.yaml not present.`
2. Check if `fab/project/config.yaml` already contains the pointer sentinel line
   `# Full reference of all available options: fab config reference`. If it is
   present anywhere in the file, skip — the pointer is already documented.
   Print: `Skipped: config reference pointer already present.`

## Changes

### 1. Add the `fab config reference` pointer line (sentinel-guarded)

If `fab/project/config.yaml` exists and does not already contain the pointer
sentinel line:

1. Prepend exactly this line as the file's header (the first line of the file),
   followed by the existing content unchanged:

   ```yaml
   # Full reference of all available options: fab config reference
   ```

   This matches the scaffold placement at
   `$(fab kit-path)/scaffold/fab/project/config.yaml`, so migrated projects and
   newly-scaffolded projects converge on the same header.

2. All existing keys, values, comments, and formatting — `project`,
   `source_paths`, `test_paths`, `true_impact_exclude`, `checklist`,
   `review_tools`, `agent` (incl. `spawn_command` and any `tiers`),
   `stage_hooks`, `branch_prefix`, `fab_version` — SHALL be preserved verbatim
   below the new header line.

3. Use an atomic write (write to a temp file in the same directory, then
   `rename` to `fab/project/config.yaml`) to prevent partial writes if the user
   has the file open in an editor.

4. Print a note:
   ```
   Added the `fab config reference` pointer line to the top of
   fab/project/config.yaml. Run `fab config reference` to print the fully
   commented reference of all available config options.
   ```

## Verification

1. `fab/project/config.yaml` begins with the pointer line
   `# Full reference of all available options: fab config reference`.
2. All other keys and their contents are unchanged — `project`, `source_paths`,
   `test_paths`, `true_impact_exclude`, `checklist`, `review_tools`, `agent`,
   `fab_version`, and (when present) `stage_hooks`, `branch_prefix`. `yq` over
   any pre-existing key returns the same value it had before the migration.
3. `fab config reference` runs and prints a commented reference config.yaml
   (exit 0) — the command the pointer names is available.
4. Re-running this migration is a complete no-op (the Pre-check sentinel trips
   and prints `Skipped: config reference pointer already present.`).
