# Migration: 1.9.1 to 1.9.2

## Summary

Add the optional `true_impact_exclude` field to `fab/project/config.yaml` so
the `/git-pr` skill can render a "true impact" line-count block in the PR
body that excludes noise directories (defaults: `fab/`, `docs/`).

This migration is project-wide (one-shot, not per-change) and idempotent.
Existing values are preserved; the field is added only when missing.

## Pre-check

1. Confirm `fab/project/config.yaml` exists at the project root. If not, this
   project is not initialized and the migration is a complete no-op.
   Print: `Skipped: fab/project/config.yaml not present.`

## Changes

### 1. Idempotency check

Read `fab/project/config.yaml`. If the file already contains a top-level
`true_impact_exclude:` key (with any value — including `[]`, `null`, or a
user-customized list such as `[vendor/]`), this migration is a no-op for
this project. Print:

`Skipped: true_impact_exclude already present.`

The user's existing value SHALL NOT be overwritten, even when it differs
from the default `[fab/, docs/]`. Re-running the migration after a prior
apply MUST detect the field and skip without error.

### 2. Append `true_impact_exclude` as a top-level entry

Otherwise, append the following line to `fab/project/config.yaml` as a new
top-level entry (YAML flow style, single line):

```yaml
true_impact_exclude: [fab/, docs/]
```

Placement guidance: insert the entry adjacent to other top-level scalar/
sequence keys (e.g., immediately after `source_paths:` if present). All
other top-level sections — `project:`, `source_paths:`, `checklist:`,
`stage_directives:`, `agent:`, comments, and formatting — SHALL be
preserved verbatim.

Use an atomic write (write to a temp file in the same directory, then
`rename` to `fab/project/config.yaml`) to prevent partial writes if the
user has the file open in an editor.

Print: `Added: true_impact_exclude: [fab/, docs/] to fab/project/config.yaml.`

## Verification

1. `fab/project/config.yaml` contains a top-level `true_impact_exclude:` key.
2. The value is `[fab/, docs/]` for fresh applies; pre-existing values
   (e.g., `[vendor/]`) are preserved unchanged when the field was already
   present.
3. All other top-level keys (`project`, `source_paths`, `checklist`,
   `stage_directives`, `agent`) and their contents are unchanged.
4. Re-running this migration on the same project is a complete no-op:
   the idempotency check in step 1 hits the
   `true_impact_exclude already present` branch.
5. When `fab/project/config.yaml` is absent, the migration prints the
   pre-check skip message and exits without error.
