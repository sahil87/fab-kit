# Migration: 2.13.1 to 2.13.2

## Summary

Backfill the v2.13.1 **providers config template** (#467) into existing projects'
`fab/project/config.yaml`. #467 pre-filled three providers in the **scaffold**
template — claude live, codex/gemini as commented starter blocks, plus an
expanded explanatory header and claude's commented `dispatch_command` line — but
scaffold files are **copy-if-absent** (`fab sync`), so existing projects never
picked these up, and no migration targeted `2.13.0→2.13.1`. As a result the
installed base permanently diverges from the shipped template: users on migrated
configs never discover multi-provider support or claude CLI dispatch.

This migration surfaces that template to existing configs by adding, all
**comment-only**:

1. the providers **explanatory header** (including the per-provider-notes
   paragraph),
2. claude's commented **`dispatch_command`** line, and
3. the commented **codex/gemini starter blocks**.

No live key is added, removed, or modified — a user who wants codex/gemini
uncomments and adapts. This is config-only and additive, the same comment-backfill
shape as `2.9.2-to-2.10.0` (config-reference pointer line) and `2.11.0-to-2.12.0`
(commented per-tier `spawn_command` note): Summary / Pre-check / Changes /
Verification, atomic write. It is **sentinel-guarded** (comment-sentinel
idempotency, per `2.2.0-to-2.3.0` and `2.11.0-to-2.12.0`) and value-preserving —
re-running is a complete no-op.

There is **no `.status.yaml` schema change** and **no Go binary change**. It needs
**no binary capability pre-check** (unlike the `2.5.5-to-2.6.0` / `2.6.6-to-2.7.0`
re-baselines): the added content is entirely comments — no new binary behavior is
required for the comments to be valid (the codex/gemini/dispatch grammar they
document already works on the 2.13.x binary). `src/kit/VERSION` is bumped
`2.13.1` → `2.13.2` (patch — comment-only backfill with no binary change;
patch-target precedent: `1.9.1-to-1.9.2`). FROM is the real current VERSION
(`2.13.1`) per the `2.9.2-to-2.10.0` chaining precedent; projects at local
`2.13.0` reach it via a gap-skip to `2.13.1` then apply.

## Pre-check

1. Check if `fab/project/config.yaml` exists — if not, skip this migration
   entirely. Print: `Skipped: fab/project/config.yaml not present.`
2. Check for a top-level `providers:` key. If **absent**, STOP with guidance —
   the config has not run the `2.12.1-to-2.13.0` migration (which introduces the
   `providers:` block). For projects migrating from ≤ 2.12.1 the chained
   `/fab-setup migrations` flow runs `2.12.1-to-2.13.0` first (FROM-ascending),
   so this is normally hit only by a direct-file invocation or a hand-set
   `fab/.kit-migration-version`. Print:
   ```
   Stopped: no top-level `providers:` block in fab/project/config.yaml. Run the
   2.12.1-to-2.13.0 migration first (it introduces the providers block), then
   re-run this migration.
   ```
3. **Sentinel.** Skip when the config already carries a `codex` or `gemini`
   provider — live (`codex:` / `gemini:` mapping keys under `providers:`) or as
   the commented starter marker (`# codex:` / `# gemini:` — the marker this
   migration writes). For the commented-marker case print:
   `Skipped: codex/gemini provider template already present.`
   For the live-key case (the user already configured that provider — the
   template was never backfilled, and none is needed) print:
   `Skipped: codex/gemini provider already configured — leaving config untouched.`
   (Comment-sentinel precedent: `2.2.0-to-2.3.0`, `2.11.0-to-2.12.0`.)

## Changes

If `fab/project/config.yaml` exists, has a top-level `providers:` block, and does
not already carry a codex/gemini provider (per Pre-check), apply the following.
All edits are **comment-only** — no live key is added, removed, or modified.
Preserve all unrelated keys (`project`, `source_paths`, `test_paths`,
`true_impact_exclude`, `checklist`, `agent`, `stage_hooks`, `branch_prefix`,
`fab_version`) and their values verbatim. Use an atomic write (write to a temp
file in the same directory, then `rename` to `fab/project/config.yaml`) to prevent
partial writes if the user has the file open in an editor.

### 1. Providers explanatory header (refresh or insert)

Detect the #467 per-provider-notes paragraph by its distinctive line:

```
# Per-provider notes (kept out of the blocks below so uncommenting a whole block
```

If that line is **already present** anywhere above `providers:`, the header is
current — leave it untouched and proceed to step 2.

If it is **absent**:

- **If a pre-#467 providers header is present** (configs scaffolded on the 2.13.0
  template — distinctive old line
  `# dispatch; ABSENT → native Agent-tool dispatch). The two are NOT merged.`),
  replace that header comment paragraph — the contiguous run of `#`-prefixed lines
  **starting at the old header's first line
  `# providers — named agent invocation grammars`** and ending immediately above
  `providers:` — with the v2.13.1 wording below. Do not remove `#` lines above
  that starting anchor: a user's own comments sitting contiguously above the old
  header are unrelated comments, preserved verbatim per step 5.
- **If no providers header exists at all** (configs migrated by
  `2.12.1-to-2.13.0`, which writes only a bare `providers:` block), insert the
  full v2.13.1 header immediately above the `providers:` line.

The v2.13.1 header (verbatim from the scaffold —
`$(fab kit-path)/scaffold/fab/project/config.yaml`; top-level comment, no
indentation):

```yaml
# providers — named agent invocation grammars. Each provider MAY carry a
# session_command (opens an interactive session: fab operator / fab batch /
# fab agent) and/or a dispatch_command (runs one headless stage task via fab
# dispatch, which pipes the stage prompt to the command's STDIN; ABSENT →
# native Agent-tool dispatch). The two are NOT merged. {model}/{effort}
# placeholders are substituted from the resolved tier profile; a plain Claude
# command has --model/--effort appended instead. Provider names are opaque,
# user-chosen strings. See: fab config reference.
#
# claude is the built-in default (session_command shown live). codex and gemini
# are a commented starter TEMPLATE — uncomment and adapt to add that provider.
# Anything whose uncommenting changes default behavior ships commented: claude's
# dispatch_command (flips claude native→headless CLI dispatch) and the opt-in
# codex/gemini blocks.
#
# Per-provider notes (kept out of the blocks below so uncommenting a whole block
# yields valid YAML — strip the leading '# ' from every line of a block):
#   claude.dispatch_command — claude -p reads the prompt from stdin; uncommenting
#     runs claude's stages as headless CLI processes instead of native sub-agents.
#   codex — codex exec reads the prompt from stdin. Substitute a current model ID
#     for {model} (e.g. gpt-5.3-codex); {model}/{effort} come from the tier.
#   gemini — no {effort} (the gemini CLI has no reasoning-effort flag) and no -p:
#     gemini's -p takes prompt TEXT (appended after stdin), whereas fab dispatch
#     pipes the prompt to stdin, which gemini reads as the prompt in non-TTY mode.
```

### 2. Claude commented `dispatch_command` line

When a `claude:` provider exists under `providers:` and carries no
`dispatch_command` (neither live nor commented), append the commented line
directly after its `session_command` line (or as the first line under `claude:`
in the edge case where no `session_command` exists), at the provider's own field
indent (see step 4 for indent adaptation). When the old scaffold note
`# no dispatch_command → claude's stages dispatch natively via the Agent tool`
is present after the `session_command`, **replace** that note with this line
instead of adding a second line.

When **no `claude:` provider exists** (the user renamed it, or a non-claude
config such as `UNNAMED_PROVIDER` from the `2.12.1-to-2.13.0` halt-and-ask path),
**skip this piece** — a claude-specific comment cannot attach to a non-claude
provider — and still append the codex/gemini blocks in step 3.

The line (verbatim from the scaffold; shown at the scaffold's 2-space indent —
the `dispatch_command` comment sits one level under `claude:`, i.e. at the
`session_command` field indent):

```yaml
    # dispatch_command: 'claude -p --dangerously-skip-permissions --model {model} --effort {effort}'
```

### 3. Codex/gemini starter blocks

Append the commented codex/gemini starter blocks after the last entry of the
`providers:` mapping (before the next top-level key, or at end of file when
`providers:` is the last top-level key). Verbatim from the scaffold,
at the scaffold's 2-space indent (the `# codex:` / `# gemini:` markers sit at the
mapping-key indent, one level under `providers:`):

```yaml
  # codex:
  #   session_command: 'codex -m {model} -c model_reasoning_effort={effort}'
  #   dispatch_command: 'codex exec -m {model} -c model_reasoning_effort={effort}'
  # gemini:
  #   session_command: 'gemini -m {model}'
  #   dispatch_command: 'gemini -m {model}'   # no {effort} flag; no -p (fab dispatch pipes the prompt to stdin)
```

### 4. Indent adaptation

The scaffold above is **2-space** indented. Configs written by the go-yaml
marshaller (e.g. fab-kit's own) are **4-space** indented. Detect the file's
mapping indent from the existing `providers:` block children (the indentation of
the first provider key under `providers:`, and of its first field — `claude:` and
`session_command:` in the common case; when the mapping has no children to
measure, fall back to the scaffold's 2-space form) and emit **all** commented
lines — the claude `dispatch_command` (step 2) and the
codex/gemini blocks (step 3) — so that stripping the leading `# ` from every line
of a block yields valid YAML at the file's own indent.

Worked example — a **4-space** config (proven by fab-kit's own hand-patched
config). The `# codex:` marker sits at the 4-space mapping-key indent; its
children carry `# ` then the field content at the file's 8-space field indent, so
that stripping `# ` per line yields valid 4-space YAML:

```yaml
    # codex:
    #     session_command: 'codex -m {model} -c model_reasoning_effort={effort}'
    #     dispatch_command: 'codex exec -m {model} -c model_reasoning_effort={effort}'
    # gemini:
    #     session_command: 'gemini -m {model}'
    #     dispatch_command: 'gemini -m {model}'   # no {effort} flag; no -p (fab dispatch pipes the prompt to stdin)
```

and the claude `dispatch_command` at 4-space (after `session_command`):

```yaml
        # dispatch_command: 'claude -p --dangerously-skip-permissions --model {model} --effort {effort}'
```

(Uncommenting: strip `# ` — the comment marker plus one space — from each line of
the block. In the 4-space form, `    # codex:` → `    codex:` and
`    #     session_command: …` → `        session_command: …`.)

### 5. Value preservation

All live keys, values, and unrelated comments are preserved verbatim. No live key
is added, removed, or modified — the migration only inserts comment lines. The
`yq -o=json '.providers'` output (JSON output strips comments) is **semantically
identical** before and after, as is `yq -o=json '.agent'` and every other
top-level key.

Print a note after applying (adjust to the pieces actually added — e.g. omit the
claude `dispatch_command` mention on the no-`claude:` path, and the header mention
when it was already current):

```
Backfilled the providers config template into fab/project/config.yaml (comment-only):
the explanatory header, claude's commented dispatch_command, and the codex/gemini
starter blocks. To wire codex or gemini, uncomment the relevant block (strip the
leading '# ' from every line) and adapt it. See: fab config reference.
```

## Verification

1. YAML still parses: `yq '.' fab/project/config.yaml` succeeds (exit 0).
2. Live semantics unchanged: `yq -o=json '.providers'` and `yq -o=json '.agent'`
   (JSON output strips comments — a plain `yq` diff would trip on the very
   comments this migration adds) return values identical to the pre-migration
   config — no live key was added, removed, or modified.
3. The comment content is present: the per-provider-notes detection line
   `# Per-provider notes (kept out of the blocks below so uncommenting a whole block`
   appears above `providers:`; the commented claude `dispatch_command` line is
   present after claude's `session_command` (only when a `claude:` provider exists
   **and carried no `dispatch_command` — live or commented — before the migration**,
   i.e. exactly when step 2 applied; a claude provider with a live
   `dispatch_command`, e.g. relocated there by `2.12.1-to-2.13.0`, correctly has
   no commented line and passes this step); and the `# codex:` and `# gemini:`
   starter markers are present.
4. A starter block mechanically uncomments to valid YAML: in a **temp copy**,
   strip the leading `# ` from every line of the `# codex:` block, then
   `yq '.providers.codex' <temp>` returns the codex mapping and `yq '.' <temp>`
   parses — confirming the emitted indent matches the file's own.
5. Re-running this migration is a complete no-op: the Pre-check sentinel trips
   (a `codex`/`gemini` key or `# codex:`/`# gemini:` marker is now present) and
   prints `Skipped: codex/gemini provider template already present.`
