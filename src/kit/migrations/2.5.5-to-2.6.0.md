# Migration: 2.5.5 to 2.6.0

## Summary

Re-baseline existing projects onto **freeze-on-write** `log.md` generation. As of
2.6.0, `fab memory-index` no longer regenerates each `docs/memory/{domain}/log.md`
as a pure function of live git history. Instead, the existing `log.md` is
**authoritative and write-once**: regeneration reads it back and **appends only**
new entries keyed on `(file-base, change-id)`; unattributable commits (migrations,
direct-`main` edits, squash-merges that dropped the branch token) that are not
already recorded are **frozen, not re-projected**. This makes `log.md` deterministic
across git history rewrites (squash-merge + branch-delete) — the failure mode where
merely touching `docs/memory/` dragged dozens of unrelated `log.md` files into churn
(and kept `fab memory-index --check` permanently red). See `docs/specs/fkf.md` §6.

Existing projects carry `log.md` files generated under the **old** (pure-projection)
model. After a squash-merge those files are already stale relative to live git, so
the first freeze-on-write run would treat the stale-but-committed lines as the frozen
truth and preserve them. The fix is a **one-time re-baseline**: run
`fab memory-index --rebuild` once to produce a clean frozen baseline from current
git, then commit it. From that commit on, every `fab memory-index` run is
append-only stable.

**This migration IS the fix for existing repos** — there is no separate manual step.

This migration ships **no `.status.yaml` schema change** and **no `fab/` data
change**; it only regenerates `docs/memory/` `log.md` files (and the indexes, which
`--rebuild` also rewrites) and commits the result. A project with no `docs/memory/`
directory skips it entirely. It is **idempotent**: re-running `--rebuild` + commit on
an already-clean tree is a no-op diff.

## Pre-check

> **Upgrade ordering (MUST be respected).** The new **binary** lands first
> (`brew upgrade fab-kit`), *then* `/fab-setup migrations` applies this migration.
> Applying it with an older binary would fail on the unknown `--rebuild` flag, so the
> pre-check below verifies the running binary understands `--rebuild` and **aborts
> before rewriting anything** if it does not.

1. **Binary capability probe.** Confirm the running `fab` binary understands the
   `--rebuild` flag (added in 2.6.0). Run `fab memory-index --help` and check its
   output for `--rebuild`. If the flag is **absent**, this is an old binary —
   **abort** with no partial rewrite. Print:
   `Aborted: this migration needs fab ≥ 2.6.0 (the --rebuild flag). Upgrade the binary first: brew upgrade fab-kit.`
   Do NOT run `fab memory-index --rebuild` (it would error on the unknown flag and
   could leave a half-written tree).

2. **Memory presence.** Confirm `docs/memory/` exists at the project root — if not,
   this project has no memory tree and the migration is a complete no-op. Print:
   `Skipped: docs/memory/ not present.`

## Changes

### 1. Produce the clean frozen baseline

Run the destructive re-projection once:

```sh
fab memory-index --rebuild
```

This discards the accumulated (stale) `log.md` state and re-projects every `log.md`
from current git — the old pure-projection behavior, made explicit and opt-in. It
also rewrites the `index.md` files and stamps the root `fkf_version`. The result is
the **last** churn the repo sees from this issue; every run afterward is append-only
stable (a plain `fab memory-index` preserves the frozen lines).

### 2. Commit the baseline

Stage and commit the regenerated `docs/memory/` tree as the freeze-on-write starting
point:

```sh
git add docs/memory/
git commit -m "chore: re-baseline log.md onto freeze-on-write (2.6.0)"
```

The re-baseline commit is itself a **one-time, intentional churn** (it rewrites the
currently-stale `log.md` files into the clean frozen form). That commit is the *last*
churn; from there freeze-on-write keeps every subsequent `fab memory-index` run
append-only stable.

### 3. Bump the kit VERSION

`src/kit/VERSION` (the kit copy) reads `2.6.0`.

## Verification

1. The running binary understands `--rebuild`: `fab memory-index --help` lists the
   `--rebuild` flag. (On an old binary the pre-check aborts here with the
   upgrade-first message and rewrites nothing.)
2. After `fab memory-index --rebuild` + commit, `fab memory-index --check` exits
   **0 or 1** (never 2 — a freshly re-projected tree is provably never
   destructive-loss).
3. A subsequent plain `fab memory-index` run is **append-only stable**: it preserves
   every frozen `log.md` line (no rewrite, no re-dating, no re-wording) and only
   appends entries for genuinely new `(file-base, change-id)` pairs. A NEW
   unattributable commit after the baseline is NOT projected into the log.
4. Re-running this migration on the same project is a complete no-op: re-running
   `fab memory-index --rebuild` + commit on an already-clean tree produces a no-op
   diff (nothing to commit), and the `--rebuild` pre-check still passes.
