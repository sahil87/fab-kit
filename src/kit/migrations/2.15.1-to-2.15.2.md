# Migration: 2.15.1 to 2.15.2

## Summary

Un-ignore `fab/.fab-version` and commit it, so worktrees, clones, and CI have a
version source.

As of 2.15.0 the config-upgrade effort's Change 3 relocated the project-pinned
engine version out of `config.yaml` into the plain-text sibling `fab/.fab-version`
(one line, bare semver + newline, **committed**). But the `.fab-*` line added
2026-03-11 for the root runtime files (`.fab-status.yaml`, `.fab-backend`,
`.fab-runtime.yaml`) is an **unanchored** gitignore pattern — with no slash it
matches at any directory depth, so it also silently swallows `fab/.fab-version`.
The file exists on disk but can never be committed; every fresh checkout (new
worktree, new clone, CI) then has **no** version source and `fab sync` / `wt init`
fail-loud (`no fab version found in fab/.fab-version or config.yaml`). The design
said "committed"; the ignore class silently defeated it.

The fix is a negation line `!fab/.fab-version` in `.gitignore` (immediately after
`.fab-*`), plus the commit the binary cannot make. As of 2.15.2 the shipped
scaffold fragment (`src/kit/scaffold/fragment-.gitignore`) carries the negation,
and the fragment is applied by `lineEnsureMerge` on **every** `fab sync` (not only
at init) — so a repo's `.gitignore` **self-heals** the negation on its next
`fab upgrade-repo`/`fab sync`. That reduces this migration's job to **verification
plus the commit**: confirm the negation took effect (or add it if the repo's sync
predates the fixed fragment) and land `fab/.fab-version` in git.

This is a **user-data restructure** (it edits the user's `.gitignore` and commits a
file), so it ships as a migration per the constitution, not an ad-hoc script. It is
**sentinel-guarded** (idempotent) and never touches any `.gitignore` line other
than adding the single negation.

`src/kit/VERSION` is bumped `2.15.1` → `2.15.2` (patch — a pure fix, no schema
change; migration name FROM=released TO=next).

## Pre-check

1. **Not a git repository, or `fab/.fab-version` absent** — skip this migration
   entirely (a non-git checkout cannot commit anyway; a repo without
   `fab/.fab-version` predates version stamping and reaches this migration only
   after `2.14.0-to-2.15.0` has run). Print:
   `Skipped: not a git repository or fab/.fab-version not present.`
2. **Sentinel (idempotency).** Skip when the fix is already complete:
   `fab/.fab-version` is **not ignored** (`git check-ignore -q fab/.fab-version`
   exits non-zero) **AND** it is **committed**
   (`git ls-files --error-unmatch fab/.fab-version` exits 0). Print:
   `Skipped: fab/.fab-version already un-ignored and committed.`
   (Re-running is a complete no-op.) This sentinel also covers the
   **sync-first order**: `fab upgrade-repo`/`fab sync` merges the fixed fragment,
   which self-heals the `!fab/.fab-version` negation before `/fab-setup migrations`
   runs — so the check-ignore half of the sentinel already holds and the migration
   only needs to land the commit (or is a full no-op if a prior run already did).

## Changes

Apply only when the Pre-check did not trip (git repo, `fab/.fab-version` present,
and the file is still ignored and/or uncommitted).

### 1. Ensure the negation is present in `.gitignore`

If `git check-ignore -q fab/.fab-version` reports the file **still ignored** (the
repo's sync predates the fixed fragment, or there is no `.gitignore` yet), add the
negation:

- If `.gitignore` contains a `.fab-*` line, insert `!fab/.fab-version` on the line
  immediately after it.
- Otherwise (no `.fab-*` line, or no `.gitignore`), append `!fab/.fab-version` at
  the end of `.gitignore` (creating the file if needed).

Do **not** touch or reorder any other line. If the negation is already present
(fragment self-heal already ran), make no change here — unless the file is
nonetheless still ignored (a hand-crafted `.gitignore` where a later `.fab-*`
line follows the negation: gitignore last-match-wins), in which case move the
`!fab/.fab-version` line below the last `.fab-*` line so the negation wins.

### 2. Commit `fab/.fab-version`

Stage and commit the version file and the (possibly modified) `.gitignore`:

```sh
git add .gitignore fab/.fab-version
git commit -m "chore: commit fab/.fab-version (un-ignore for worktrees/clones/CI)" -- .gitignore fab/.fab-version
```

The commit is **pathspec-scoped** to exactly these two files: if the user's index
happens to carry unrelated staged changes when `/fab-setup migrations` runs, they
MUST NOT be swept into the migration commit.

If `git add`/`commit` finds nothing to commit (already committed by a prior run or
the fragment self-heal plus an earlier commit), that is fine — the Verification
sentinel is the source of truth.

## Verification

1. `fab/.fab-version` is no longer ignored:
   `git check-ignore -q fab/.fab-version` exits **non-zero**.
2. `fab/.fab-version` is committed:
   `git ls-files --error-unmatch fab/.fab-version` exits **0**.
3. A fresh worktree/clone now resolves a version:
   `cat fab/.fab-version` prints the pinned semver (e.g. `2.15.1`), so
   `fab sync` / `wt init` no longer fail with "no fab version found".
4. Re-running this migration is a complete no-op — the Pre-check sentinel trips
   (not ignored AND committed) and prints
   `Skipped: fab/.fab-version already un-ignored and committed.`
