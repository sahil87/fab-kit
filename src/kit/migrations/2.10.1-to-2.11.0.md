# Migration: 2.10.1 to 2.11.0

## Summary

Remove the `artifact-write` PostToolUse hook entries from an existing project's
`.claude/settings.local.json`. Artifact-derived `.status.yaml` state
(`change_type`, `confidence`, plan counts) is no longer owned by a Claude Code
hook — it is now recomputed on demand by the new `fab status refresh` command
and self-healed at the transition seams (`fab status advance`/`finish`, `fab
preflight`). Keeping the hook would leave two writers for the same fields (drift
risk) and would fire only in the Claude harness, so it was removed;
`fab hook sync` no longer registers it for new projects, and this migration
backfills the removal for projects already on fab-kit.

Concretely this migration drops the two PostToolUse entries whose command is
`fab hook artifact-write` (one `Write` matcher, one `Edit` matcher) from
`.claude/settings.local.json`. The three session-scoped hook entries
(`SessionStart` → `fab hook session-start`, `Stop` → `fab hook stop`,
`UserPromptSubmit` → `fab hook user-prompt`) are **left untouched** — they are
push-by-nature runtime telemetry that owns no correctness-critical state.

This restructures user-owned data (`.claude/settings.local.json`), so it ships
as a migration rather than an ad-hoc script (constitution + code-quality rule),
following the `settings.local.json`-editing precedent in `0.46.0-to-1.1.0.md`
§1. There is **no `.status.yaml` schema change** — `change_type_source` already
exists; only *who* writes the derived fields changed. It is sentinel-guarded and
idempotent — re-running is a complete no-op.

Any remaining registered `fab hook artifact-write` entry that a hand-edited
settings file still carries is harmless if invoked: the subcommand is retained
for one release as a silent no-op shim (it exits 0 and emits nothing on stdout).

## Pre-check

1. Check if `.claude/settings.local.json` exists — if not, skip this migration
   entirely. Print: `Skipped: .claude/settings.local.json not present.`
2. Parse the file's `hooks.PostToolUse` array. If no entry has a hook whose
   `command` is `fab hook artifact-write` (for either the `Write` or `Edit`
   matcher), skip — the entries are already gone. Print:
   `Skipped: artifact-write hook already removed.`

## Changes

### 1. Remove the `artifact-write` PostToolUse entries (sentinel-guarded)

If `.claude/settings.local.json` exists and its `hooks.PostToolUse` array
contains at least one entry whose `command` is `fab hook artifact-write`:

1. Read the file and parse it as JSON.
2. In `hooks.PostToolUse`, drop every entry whose `hooks[]` contains a
   `{"type": "command", "command": "fab hook artifact-write"}` action — this
   covers both the `Write`-matcher and `Edit`-matcher entries. An entry that
   mixes `fab hook artifact-write` with an unrelated custom command in the same
   `hooks[]` array SHALL have only the `fab hook artifact-write` action removed,
   preserving the custom command (mirror the `0.46.0-to-1.1.0.md` §1
   preserve-non-fab-hooks discipline). Remove the entry entirely when its
   `hooks[]` becomes empty. If `hooks.PostToolUse` becomes empty, leave it as an
   empty array (or omit the key) — do not delete the `hooks` object.
3. Leave the `SessionStart`, `Stop`, and `UserPromptSubmit` entries (and any
   non-hook top-level keys such as `model`/`permissions`) unchanged.
4. Use an atomic write (write to a temp file in the same directory, then
   `rename` to `.claude/settings.local.json`) to prevent partial writes if the
   user has the file open in an editor.
5. Print a note:
   ```
   Removed the artifact-write PostToolUse hook entries from
   .claude/settings.local.json. Artifact-derived .status.yaml state is now
   recomputed by `fab status refresh` (self-healed at fab status
   advance/finish and fab preflight).
   ```

## Verification

1. `.claude/settings.local.json` `hooks.PostToolUse` no longer contains any
   entry whose command is `fab hook artifact-write` (neither the `Write` nor the
   `Edit` matcher).
2. The three session-hook entries are unchanged — `SessionStart` →
   `fab hook session-start`, `Stop` → `fab hook stop`, `UserPromptSubmit` →
   `fab hook user-prompt` — and any non-hook top-level keys (`model`,
   `permissions`, …) are preserved verbatim.
3. `fab hook sync` run afterward is a no-op for hooks
   (`.claude/settings.local.json hooks: OK`) — it does not re-add the
   artifact-write entries.
4. Re-running this migration is a complete no-op (the Pre-check sentinel trips
   and prints `Skipped: artifact-write hook already removed.`).
