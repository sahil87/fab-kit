# Migration: 1.8.0 to 1.9.0

## Summary

Collapse the `tasks` pipeline stage into the `apply` stage. Replace the two
artifacts `tasks.md` + `checklist.md` with a single `plan.md` that exposes
heading-keyed `## Tasks` and `## Acceptance` sections. The pipeline goes from
8 stages to 7 (`intake → spec → apply → review → hydrate → ship → review-pr`).

This migration runs over every in-flight change folder under `fab/changes/`
(excluding `archive/`) and:

1. Produces `plan.md` by merging the legacy `tasks.md` and `checklist.md`
   bodies under unified `## Tasks` and `## Acceptance` headings.
2. Rewrites `.status.yaml` to drop `progress.tasks` and replace the
   `checklist:` block with a `plan:` block.
3. Annotates each legacy file with a `<!-- Migrated to plan.md ... -->`
   comment so users can verify the merge before deleting the originals.

## Pre-check

1. Confirm `fab/changes/` exists at the project root — if not, this project
   has no in-flight changes and the migration is a complete no-op.

## Changes

### 1. Walk every in-flight change folder

For each entry in `fab/changes/` that is a directory and is **not** named
`archive` (and is not nested under `archive/`):

Apply the per-change logic in steps 2–4 below. Archived folders
(`fab/changes/archive/**`) are historical and SHALL NOT be touched.

### 2. Per-change idempotency check

Inside each change folder:

1. **If `plan.md` already exists** → no-op for this change. Print:
   `Skipped: {folder} (plan.md already present).` Continue to the next folder.
2. **If neither `plan.md` nor `tasks.md` nor `checklist.md` exists** → no-op
   (the change is pre-planning). Print:
   `Skipped: {folder} (pre-planning — no legacy artifacts).` Continue.
3. **Otherwise** → proceed to step 3.

### 3. Produce `plan.md` from legacy artifacts

Build the new `plan.md` body by concatenating:

1. A standard frontmatter block:

   ```markdown
   # Plan: {CHANGE_NAME}

   **Change**: {folder-name}
   **Status**: In Progress
   **Intake**: `intake.md`
   **Spec**: `spec.md`
   ```

   Resolve `{CHANGE_NAME}` from the legacy `tasks.md` H1 (preferred) or
   `checklist.md` H1 (fallback) — strip the `Tasks: ` / `Quality Checklist: `
   prefix to recover the human-readable name. If neither H1 is present, use
   the folder name verbatim.

2. A `## Tasks` heading, followed by the body of `tasks.md`:
   - Strip the H1 line (`# Tasks: ...`) and the immediately-following
     metadata block (`**Change**: ...`, `**Spec**: ...`, `**Intake**: ...`,
     leading HTML-comment block if present, leading blank lines).
   - Preserve subheadings (`### Phase 1: Setup`, etc.), task IDs (`T001`,
     `T002`), `[P]` markers, and all body content verbatim.
   - If `tasks.md` is missing, write a single placeholder line and log a
     warning:
     ```markdown
     <!-- tasks content not migrated — original tasks.md was missing -->
     ```
     Print: `Warning: {folder}: tasks.md missing; placeholder written.`

3. A `## Acceptance` heading, followed by the body of `checklist.md`:
   - Strip the H1 line (`# Quality Checklist: ...`) and the metadata block
     (`**Change**: ...`, `**Generated**: ...`, `**Spec**: ...`, leading
     HTML-comment block, leading blank lines).
   - Preserve category subheadings, `CHK-NNN` IDs, and body content
     verbatim. **Do not rewrite IDs** — in-flight changes keep their
     existing `CHK-NNN` IDs; only newly generated plans use `A-NNN`.
   - If `checklist.md` is missing, write a placeholder + warning:
     ```markdown
     <!-- acceptance content not migrated — original checklist.md was missing -->
     ```
     Print: `Warning: {folder}: checklist.md missing; placeholder written.`

Write the merged result to `{folder}/plan.md` (atomic write — temp file then
rename).

### 4. Rewrite `.status.yaml`

Load `{folder}/.status.yaml` and rewrite it as follows:

1. **Drop `progress.tasks`** entirely from the `progress:` mapping.

2. **Migrate `progress.apply`** based on the prior `progress.tasks` value:
   - If `progress.tasks` was `done` or `skipped` → leave `progress.apply` at
     its current state (the change was past the former tasks gate).
   - If `progress.tasks` was `active` or `ready` → set `progress.apply` to
     that value (the change was mid-planning; with plan.md generated, it is
     now mid-apply at the same activity level).
   - If `progress.tasks` was `pending` (or absent) → leave `progress.apply`
     unchanged (already pending or further along).

3. **Replace the `checklist:` block with a `plan:` block**:
   ```yaml
   plan:
     generated: true
     task_count: <count of "- [ ]" + "- [x]" lines under the merged ## Tasks section>
     acceptance_count: <old checklist.total>
     acceptance_completed: <old checklist.completed>
   ```
   The old `checklist.path` field is dropped.

   If the change had no `checklist:` block (pre-planning state that somehow
   reached this branch), set `acceptance_count: 0` and
   `acceptance_completed: 0`.

4. Save the file (atomic temp + rename, refresh `last_updated`).

### 5. Annotate legacy files

For each legacy file that exists in this folder (`tasks.md`,
`checklist.md`):

1. Append a single line:
   ```
   <!-- Migrated to plan.md on {YYYY-MM-DD} — safe to delete. -->
   ```
   Use today's date in `YYYY-MM-DD` format.

2. **Do not delete** the legacy files. Users may want to verify the merge
   before cleanup; deletion is cheap to do later by hand.

For each successfully migrated change, print:
`Migrated: {folder} (tasks: {N}, acceptance: {M}/{K}).`

### 6. Prune `stage_directives.tasks` from `fab/project/config.yaml`

After the per-change loop, update the project config to remove the legacy
`tasks` stage directive. This is project-wide (one-shot, not per-change).

1. **Idempotency check**: Read `fab/project/config.yaml`. If the
   `stage_directives:` block does not exist, or if it does not contain a
   `tasks:` key, this step is a no-op. Print:
   `Skipped: stage_directives.tasks already absent.`

2. **Otherwise**: Remove the `tasks:` entry from the `stage_directives:`
   mapping. Preserve all other directives (`apply`, `hydrate`, `intake`,
   `review`, `spec`) verbatim, including their list contents and
   formatting where possible. Write the updated config atomically (temp
   file + rename).

   Print: `Pruned: stage_directives.tasks from fab/project/config.yaml.`

   If `fab/project/config.yaml` does not exist, this step is a complete
   no-op (project not initialized). Print:
   `Skipped: fab/project/config.yaml not present.`

## Verification

1. For every change folder under `fab/changes/` (excluding `archive/`):
   - `plan.md` exists with `## Tasks` and `## Acceptance` headings.
   - `.status.yaml` `progress:` mapping does NOT contain a `tasks` key.
   - `.status.yaml` contains a `plan:` block with `generated`,
     `task_count`, `acceptance_count`, `acceptance_completed` fields.
   - `.status.yaml` does NOT contain a `checklist:` block.

2. For every change folder that had legacy artifacts:
   - `tasks.md` and/or `checklist.md` (whichever existed) ends with the
     `<!-- Migrated to plan.md ... safe to delete. -->` comment line.

3. For every folder under `fab/changes/archive/**`:
   - Files are unchanged from their pre-migration state.

4. `fab/project/config.yaml` (when present) MUST NOT contain a `tasks:`
   key under `stage_directives:`.

5. Re-running this migration on the same project is a complete no-op:
   every change folder hits the `plan.md already present` branch in step 2,
   and `stage_directives.tasks` is already absent.
