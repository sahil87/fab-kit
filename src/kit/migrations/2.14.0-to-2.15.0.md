# Migration: 2.14.0 to 2.15.0

## Summary

Relocate the project-pinned engine version out of `fab/project/config.yaml` into a
new plain-text sibling file `fab/.fab-version`, and delete the `fab_version:` key
(with any stale comment line) from `config.yaml`.

As of 2.15.0 the config-upgrade effort's Change 3 landed: `fab config upgrade` is
the **only** writer of `config.yaml`, and the comment-clobbering `setFabVersion`
masher is deleted. For that single-writer invariant to hold, the one machine-managed
field the old masher owned — `fab_version` — leaves `config.yaml` entirely. It now
lives in `fab/.fab-version` (one line, bare semver + newline), a committed sibling
to `fab/.kit-migration-version`. Both the fab-kit router (pinned-version resolution)
and the fab-go preflight staleness check read `.fab-version` first, falling back to
the config.yaml `fab_version:` key for one compat window — this migration closes that
window by moving the value and removing the key.

This is a **user-data restructure** (a config-file field moves to a new file), so it
ships as a migration per the constitution, not an ad-hoc script. It is config-only
and value-preserving: the semver value is carried verbatim; no other key, value, or
comment in `config.yaml` is touched. It is **sentinel-guarded** (idempotent) and uses
an **atomic write** (temp file + `rename`) for both files.

`src/kit/VERSION` is bumped `2.14.0` → `2.15.0` (minor — a new `fab config upgrade`
subcommand plus this user-data restructure). The **managed reference fence** the same
change introduces needs **no migration step**: it materializes on the first
`fab config upgrade` run, which the next `fab upgrade-repo` auto-runs after sync.

Historical comment-backfill migrations (e.g. `2.13.1-to-2.13.2`) are left untouched;
that pattern is **retired going forward** — future field adds/renames/removals are
mechanical registry data (`advertise`, `renamed_from`, the parked-removals block)
reconciled by `fab config upgrade`, not hand-written migrations.

## Pre-check

1. Check if `fab/project/config.yaml` exists — if not, skip this migration entirely.
   Print: `Skipped: fab/project/config.yaml not present.`
2. **Sentinel (idempotency).** Skip when the relocation is already complete:
   `fab/.fab-version` exists **AND** `config.yaml` has no top-level `fab_version:`
   key. Print: `Skipped: fab_version already relocated to fab/.fab-version.`
   (Re-running is a complete no-op.) This sentinel also covers the
   **upgrade-repo-first order**: `fab upgrade-repo` stamps `fab/.fab-version` and
   auto-runs `fab config upgrade`, which *parks* the now-unknown `fab_version:`
   key below the fence (it removes the live key, leaving a `#   fab_version: …`
   comment) — so by the time `/fab-setup migrations` runs, both sentinel
   conditions already hold and this migration is a clean no-op. The parked comment
   is harmless user territory (delete when done); the value is already in
   `fab/.fab-version` from the stamp.

## Changes

Apply only when `fab/project/config.yaml` exists and the sentinel did not trip.
Preserve every other key (`project`, `source_paths`, `test_paths`,
`true_impact_exclude`, `checklist`, `providers`, `agent`, `stage_hooks`,
`branch_prefix`) and their values and comments verbatim.

### 1. Read the pinned version

Read the value of the top-level `fab_version:` key from `fab/project/config.yaml`
(a line whose key begins at column 0, not indented, not a `#` comment). Strip
surrounding quotes and whitespace to get the bare semver (e.g. `2.14.0`).

- If **no** top-level `fab_version:` key is present but `fab/.fab-version` is also
  absent, the repo predates version stamping in `config.yaml`. Fall back to the
  value in `fab/.kit-migration-version` if present; otherwise use the engine version
  (`$(cat "$(fab kit-path)/VERSION")`). Print a note that the value was inferred.

### 2. Write `fab/.fab-version`

Write the resolved semver plus a single trailing newline to `fab/.fab-version`
(sibling of `config.yaml`'s parent `fab/` directory — i.e. `fab/.fab-version`, next
to `fab/.kit-migration-version`). Use an atomic write: write to a temp file in the
same directory, then `rename` it into place. Do **not** merge it into
`.kit-migration-version` — the two files are deliberately distinct (deployed-kit
version vs. migration baseline).

If `fab/.fab-version` already exists with a **different** value, prefer the existing
`.fab-version` value (it is the authoritative post-relocation source) and proceed to
step 3 to remove the stale config key.

### 3. Delete the `fab_version:` key from config.yaml

Remove the top-level `fab_version:` line from `fab/project/config.yaml`, plus any
immediately-adjacent comment line that documented it (a `# fab_version — …`
machine-managed note directly above or below the key, if present). Leave a single
blank-line boundary — do not collapse unrelated blank lines. Do **not** touch any
indented or commented `fab_version` occurrence elsewhere (only the column-0 live
key). Use an atomic temp-file + `rename` write.

## Verification

1. `fab/.fab-version` exists and contains the moved semver plus a trailing newline:
   `cat fab/.fab-version` prints e.g. `2.14.0`.
2. `fab/project/config.yaml` no longer carries a top-level `fab_version:` key:
   `yq '.fab_version' fab/project/config.yaml` prints `null`.
3. YAML still parses: `yq '.' fab/project/config.yaml` succeeds (exit 0).
4. Live semantics of every other key are unchanged: `yq -o=json '.'` over the
   config (JSON output strips comments) differs from the pre-migration config only
   by the removed `fab_version` key.
5. The router still resolves the pinned version: `fab --version` reports the same
   version as before (now read from `fab/.fab-version`).
6. Re-running this migration is a complete no-op — the Pre-check sentinel trips
   (`.fab-version` present AND no `fab_version:` key) and prints
   `Skipped: fab_version already relocated to fab/.fab-version.`
