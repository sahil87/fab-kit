# Migration: 1.9.7 to 1.10.0

## Summary

Collapse the `spec` pipeline stage into the `apply` stage and frontload SRAD
confidence scoring to intake. The requirement discipline that lived in a
separate `spec.md` artifact is absorbed into `plan.md` as a `## Requirements`
section. The pipeline goes from 7 stages to 6
(`intake → apply → review → hydrate → ship → review-pr`).

This is the direct sibling of migration `1.8.0-to-1.9.0.md` (which merged
`tasks` into `apply`). It runs over every in-flight change folder under
`fab/changes/` (excluding `archive/`) and:

1. Rewrites `.status.yaml` to drop `progress.spec`, folding its activity level
   into `apply`.
2. Folds a leftover `spec.md` body into `plan.md`'s `## Requirements` section
   **only when both files exist** (state 3 below); other states are handled
   without creating stub files.
3. Relocates `stage_directives.spec` content in `fab/project/config.yaml` into
   `stage_directives.apply` (apply now owns requirement generation).
4. Leaves any `confidence.indicative` key on disk untouched (the 1.10.0 binary
   tolerates it on read and simply stops writing it).

`confidence.indicative` is NOT stripped by this migration: the binary
round-trips it harmlessly, and stripping it from every un-migrated/archived
file the binary happens to re-save is unnecessary churn.

## Pre-check

1. Confirm `fab/changes/` exists at the project root — if not, this project
   has no in-flight changes and the per-change loop is a complete no-op (still
   run step 6, the config prune, which is independently guarded).

## Changes

### 1. Walk every in-flight change folder

For each entry in `fab/changes/` that is a directory and is **not** named
`archive` (and is not nested under `archive/`):

Apply the per-change logic in steps 2–5 below. Archived folders
(`fab/changes/archive/**`) are historical and SHALL NOT be touched.

### 2. Per-change idempotency sentinel

Inside each change folder, before doing any spec.md→plan.md merge:

- **Skip the merge** (step 4) if `plan.md` already contains a `## Requirements`
  heading OR a `<!-- migrated from spec.md -->` marker. The `.status.yaml`
  progress rewrite (step 3) is itself idempotent (dropping an absent key and
  re-folding an already-folded state are both no-ops), so it always runs.

Print `Skipped merge: {folder} (## Requirements already present).` when the
sentinel trips.

### 3. Rewrite `.status.yaml`

Load `{folder}/.status.yaml` and rewrite it as follows:

1. **Drop `progress.spec`** entirely from the `progress:` mapping (if present).

2. **Fold the prior `progress.spec` value into `progress.apply`** (mirroring the
   tasks→apply logic):
   - If `progress.spec` was `done` or `skipped` → leave `progress.apply` at its
     current state (the change was past the former spec gate).
   - If `progress.spec` was `active` or `ready` → set `progress.apply` to that
     value (the change was mid-spec; with spec absorbed into apply entry, it is
     now mid-apply at the same activity level). Only do this if `apply` is
     currently `pending` — never downgrade an apply that has already advanced.
   - If `progress.spec` was `pending` (or absent) → leave `progress.apply`
     unchanged.

3. Leave the `confidence:` block as-is, including any `indicative:` key.

4. Save the file (atomic temp + rename, refresh `last_updated`).

Print: `Rewrote progress: {folder} (spec → apply).`

### 4. Four-state spec.md → plan.md handling

Determine the per-change state from which of `spec.md` / `plan.md` exist, and
act accordingly. (Step 2's sentinel has already been checked — if it tripped,
do nothing here.)

| State | `spec.md` | `plan.md` | Action |
|-------|-----------|-----------|--------|
| 1 | exists | absent | **Leave `spec.md` in place. Do NOT create a `plan.md` stub.** The change is mid-spec/pre-apply; the new Plan Generation Procedure detects the legacy `spec.md` and folds it into `## Requirements` on first apply. Creating a stub would trip the "plan.md exists → skip generation" resumability guard and deadlock plan generation. Print: `Left for on-apply ingestion: {folder} (spec.md, no plan.md).` |
| 2 | absent | exists | Progress rewrite only (step 3, already done). Print: `Progress-only: {folder} (plan.md, no spec.md).` |
| 3 | exists | exists | **Merge** (see below). |
| 4 | absent | absent | Pre-planning. Progress rewrite only. Print: `Pre-planning: {folder} (no spec.md, no plan.md).` |

**State 3 merge procedure:**

1. Extract the requirement body from `spec.md`: strip its H1 (`# Spec: ...`), the
   metadata block (`**Change**: ...`, `**Created**: ...`, `**Affected memory**: ...`),
   and any leading HTML-comment template scaffolding. Preserve the domain
   requirement sections, scenarios, and any `## Non-Goals` / `## Design Decisions`
   / `## Deprecated Requirements` subsections verbatim. The trailing
   `## Assumptions` table from `spec.md` is NOT moved into `## Requirements` (it
   is historical scoring data); leave it in `spec.md`.
2. Insert a `## Requirements` section into `plan.md` immediately after the
   frontmatter block (before `## Tasks`), annotated on its first line:
   ```markdown
   ## Requirements

   <!-- migrated from spec.md on {YYYY-MM-DD} -->
   ```
   followed by the extracted requirement body. Use today's date.
3. Write the updated `plan.md` (atomic temp + rename).
4. Append a single line to `spec.md`:
   ```
   <!-- Merged into plan.md ## Requirements on {YYYY-MM-DD} — safe to delete. -->
   ```
   **Do not delete** `spec.md`. Users may want to verify the merge before
   cleanup.

Print: `Merged: {folder} (spec.md → plan.md ## Requirements).`

### 5. (reserved)

No additional per-change step. (The precedent's legacy-file-annotation step is
folded into state 3 above.)

### 6. Relocate `stage_directives.spec` in `fab/project/config.yaml`

After the per-change loop, update the project config. This is project-wide
(one-shot, not per-change).

1. **Idempotency check**: Read `fab/project/config.yaml`. If it does not exist,
   this step is a complete no-op (project not initialized). Print:
   `Skipped: fab/project/config.yaml not present.` If the `stage_directives:`
   block does not exist, or does not contain a `spec:` key, this step is a
   no-op. Print: `Skipped: stage_directives.spec already absent.`

2. **Otherwise**: **Relocate** (do not drop) the `spec:` directive list into
   `stage_directives.apply`. Unlike the empty `tasks: []` the 1.9.0 migration
   pruned, `spec:` may carry real, possibly user-customized directives (e.g.,
   GIVEN/WHEN/THEN, [NEEDS CLARIFICATION] handling, impact-doc notes, methodology
   synthesis) — apply now owns requirement generation, so they belong under
   `apply`.
   - If `stage_directives.apply` is absent or empty (`[]`), set it to the
     `spec:` list verbatim.
   - If `stage_directives.apply` already has entries, append the `spec:` entries
     that are not already present (de-duplicate by exact string match), preserving
     order: existing apply entries first, then the new spec-derived entries.
   - Remove the `spec:` key from `stage_directives:`. Preserve all other
     directives (`hydrate`, `intake`, `review`, and any legacy `tasks`) verbatim.
   - Write the updated config atomically (temp file + rename).

   Print: `Relocated: stage_directives.spec → stage_directives.apply ({N} directives).`

## Verification

1. For every change folder under `fab/changes/` (excluding `archive/`):
   - `.status.yaml` `progress:` mapping does NOT contain a `spec` key.
   - `progress.apply` reflects the folded activity level per step 3.

2. For every change folder that had **both** `spec.md` and `plan.md`:
   - `plan.md` contains a `## Requirements` heading with the
     `<!-- migrated from spec.md ... -->` marker.
   - `spec.md` ends with the `<!-- Merged into plan.md ... safe to delete. -->`
     comment line.

3. For every change folder that had **`spec.md` but no `plan.md`** (state 1):
   - No `plan.md` stub was created.
   - `spec.md` is unchanged (left for on-apply ingestion).

4. `fab/project/config.yaml` (when present) MUST NOT contain a `spec:` key under
   `stage_directives:`; its former entries appear under `stage_directives.apply`.

5. For every folder under `fab/changes/archive/**`:
   - Files are unchanged from their pre-migration state.

6. Re-running this migration on the same project is a complete no-op: every
   merge hits the `## Requirements already present` sentinel, every progress
   rewrite re-folds an already-folded state (idempotent), and
   `stage_directives.spec` is already absent.
